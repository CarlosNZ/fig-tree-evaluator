## Cross-cutting conventions — **Draft**

### What a pass settles

For each operator, one section recording:

1. **Parameter table** — name, metadata type, required/optional + default, null policy (explicitly re-confirmed per parameter, per the Type area), and evaluation mode (eager / lazy / sequential / structural).
2. **Positional mapping** — the `positionalParams` declaration (see conventions below).
3. **Semantics & error cases** — including the empty-aggregate stance where relevant.
4. **`useCache` metadata default.**
5. **Contract-ledger entries** — any capability the operator needs that the declaration vocabulary doesn't yet cover.
6. **v2 disposition** — parameter renames, dropped property aliases, behaviour changes; feeds the converter and migration doc.

The flat reservation (Node grammar: no parameter may be named `operator`, `fragment`, `parameters`, `fallback`, `useCache`, `vars`, `//`) is checked mechanically on every pass and only mentioned where it actually bit.

### The disruption gradient: loud authoring, quiet runtime

Recorded from the batch-4 discussion (Carl, July 2026) as a stance every pass applies when a case sits between "error" and "degraded result": **loudness belongs at authoring time; runtime errors reach the wrong audience.** `validate()`, registration checks and the editor are maximally strict — every detectable seam is an error or warning there. At evaluation time, prefer the least disruptive result that keeps the problem *visible*: an end user hitting sloppy config should see slightly-off output that is self-explanatory to the admin they report it to (`My name is Carl {{last}}`), not a blocked workflow behind an inexplicable error. The degradation must be **self-signaling, never silent** — rendering an unbound token literally or a composite value as `<object>` is the pattern; v2's rendering unmatched tokens as `""` (the seam erased) is forbidden by this stance as firmly as the error is. Runtime failure remains correct where nothing meaningful can be produced — wrong-type operands to value operators, failed I/O, `match` with no branch and no `default` (though case #7 deserves a re-look under this lens — noted in the register). First applications: `buildString`'s unbound-token and composite-rendering rules (batch 4); rows drafted before the stance was articulated are re-weighed against it at the group review.

### No parameter-name aliases

**A parameter has exactly one name.** This resolves the remainder of the Alias policy area: operator names get one symbolic alias each (Operators), parameter names get none at all. v2 shipped property aliases on most operators (`valueIfTrue` / `ifTrue`, `branches` / `arms` / `cases`, …); they die for the same reasons the word-alias operator names died — one vocabulary for docs, editor and error messages, no drifting synonym sets — plus a new one: with unknown node keys now hard errors (No hoisting), every alias would be a second legal spelling the grammar must carry forever. The compensation is the same as for operators: the *one* name is chosen well, favouring short plain words.

### Naming conventions

- The plain-English rule (Operators rule 0) applies to parameter names.
- **The single main input of an operator is named `value`; the operand array of a variadic operator is named `values`.** Uniform across the whole set (`convert.value` was already agreed; `not.value`, `and.values`, `plus.values`, `min.values` follow). Parameters with a distinct semantic role get the role's name (`condition`, `then`, `branches`, `url`).
- v2's descriptive compounds are renamed to the short word (`valueIfTrue` → `then`).
- **Absence-default parameters are operator-specific, named for their firing condition, with the `…Default` suffix as the family marker** (recorded July 2026, batch-5 review — Carl; first instance `find.noMatchDefault`). This is the deliberate, argued exception to the short-word rule above: the bare word `default` doesn't say *which* absence fires it — and not firing on a found/stored `null` is the whole contract — so the firing condition is load-bearing and lives in the name. There is no special global `default` property; each is an ordinary parameter, riding the layered defaults chain like any other. The bare word survives only where it is itself the established precise term: `match.default` (the default arm of a dispatch — switch/case vocabulary, rule 0's established-term clause). Knock-on settled in batch 6: `get`'s drafted `default` became `missingDefault`. The division of labour stands: these parameters answer *absence*; `fallback` catches *failure* (see the register's headline `fallback` question, still open).

### Positional mapping: `positionalParams`

The declarative replacement for v2's imperative `parseChildren` functions (Operators § Shorthand grammar). **One form: an ordered list of parameter names, whose last entry may be a rest-binding (`'...name'`).** An array payload fills the leading positions in order (greedy, left-to-right — optional leading positions fill before the rest collects); the remainder binds to the rest parameter as an array. An earlier draft had a separate "whole-payload" form for variadic operators — it's the degenerate rest-only list.

- `if: ['condition', 'then', 'else']` — no rest. Trailing entries may be omitted iff the corresponding parameters are optional; surplus elements are a parse error. Positional spellings that skip a *middle* optional parameter don't exist — write the named form.
- `and: ['...values']` — every payload element is an operand.
- `length: ['...value']` — rest-only is also how array-*input* operators bind: `{ $length: [1, 2, 3] }` → `value: [1, 2, 3]`, where a plain `['value']` would misread it as three surplus arguments.
- `buildString` (batch 4): `['template', '...substitutions']` — first element, then everything else; this shape is why the rest form exists, and the References-area examples already use it.
- **A single non-array payload binds to the first listed position, verbatim**: `{ $not: x }` → `value: x`. When the first position *is* the rest parameter, verbatim means the payload is the whole array value, supplied dynamically: `{ $and: '$data.checks' }` → `values: '$data.checks'` (runtime-checked as an array). Decided by fiat, loud when tripped: `{ $and: '$data.isAdmin' }` is *not* a one-condition `and` — it fails the runtime array check unless `isAdmin` is an array; a single condition is written `{ $and: ['$data.isAdmin'] }`. The dynamic-supply reading wins because it matches every other dynamic-whole-value position (`match.branches`, fragment `parameters`).

```js
{ $if: ['$data.isMember', 'Welcome back', 'Please sign up'] }
                              // → condition / then / else, by position
{ $if: ['$data.isMember', 'Welcome back'] }
                              // trailing omission — legal, `else` is optional → null when falsy
{ $if: ['$data.isMember'] }   // parse error: `then` is required
{ $not: '$data.disabled' }    // single non-array payload → binds to the first position, `value`

{ $and: ['$data.a', '$data.b', '$data.c'] }   // rest-only → values
{ $and: '$data.checks' }                      // dynamic → values (runtime-checked array)
{ $length: [1, 2, 3] }                        // rest-only → value: [1, 2, 3] — not three surplus args
{ $buildString: ['%1 of %2', '$data.count', '$data.total'] }  // leading position + rest
```

One trap, loud by design: for a positional-*list* operator an array payload is always the argument list, never a single array-valued argument — `{ $not: [1, 2] }` is a surplus-argument parse error ("`not` takes 1 positional argument, got 2"), not "truthiness of `[1, 2]`"; an array-valued argument is written named (`{ $not: { value: [1, 2] } }`). It cannot silently misfire: the failure is at parse, with the mapping named, and the editor renders the mapping anyway.

### Runtime `default` vs editor seed — two different things

**The `default` field in a parameter declaration is the runtime default, exclusively**: presence implies optional (the agreed optionality rule), it participates in the layered chain (per-node → `operatorDefaults` → metadata default), and null-means-unset resolves to it. v2's `data.ts` metadata used the same field name for something else entirely — the value the [fig-tree-editor-react](https://github.com/CarlosNZ/fig-tree-editor-react) editor seeds a freshly-created node with, which is why v2 AND declares `values` as `required: true` *with* a `default` of `[true, true]` ([AND/data.ts:13](../src/operators/AND/data.ts#L13)) — not a contradiction, an editor seed in a runtime-looking field. v3 separates them:

- **Runtime default** → the `default` field. With editor seeds evicted, the `required` + `default` contradiction rule (Fragments) is unambiguous — a required parameter with an editor seed is now expressible without tripping it.
- **Editor seeds** (and any other UI concern) → the opaque **`metadata`** bag, alongside the display colours already settled for fragments (Fragments § Tooling metadata). The engine never reads it; `getOperators()` returns it verbatim; the editor defines its own keys within it.

**Per-parameter `metadata`**: parameter declarations gain an optional opaque `metadata` field, mirroring the definition-level bag — the rule becomes *`description` and `metadata` travel together at every declaration level*. (A definition-level map keyed by parameter name was considered and rejected: it splits a parameter's tooling data away from its declaration and invites key drift on renames.) Agreed — the matching row is added to the fragment parameter-declaration table (Fragments § Parameter declarations). Custom operators and plugin packages carry their own bags in-definition — the editor can't know them; the *core* operators carry none at all (§ Metadata on definitions the host didn't author, below).

**v2 disposition, global**: every `default` in v2's operator `data.ts` files is an editor seed, not a runtime default — **none migrates to the v3 `default` field automatically**. Each pass decides runtime defaults from scratch; the seed values leave the runtime entirely, moving to the editor-hints sub-path module (§ The editor-hints module, below).

### Metadata on definitions the host didn't author — settled tool-side

The gap the bags don't cover: display metadata for definitions the host doesn't own — the core operators above all ("in this system, `http` nodes are blue and seed with our API template").

**Considered and rejected: any engine-side attachment mechanism** — whether a `withMetadata()` decorator (a pure function merging bags into existing definitions before they enter the `operators` array) or, worse, an `operatorMetadata` option. The defeating argument is the round trip: the engine never reads metadata, so decorating a foreign definition injects values into the registry *solely for tooling to read back out* via `getOperators()` — the tool may as well accept the same map directly. Two supporting arguments: tool-side, the map is properly **typed** in the package that defines its keys (`backgroundColor: string`), where the opaque bag is untyped by construction; and the `operatorDefaults` visibility precedent does not transfer — runtime defaults change evaluation behaviour, so tooling that read only raw definitions would *lie* about behaviour without engine-side merging, while display metadata changes nothing and nothing lies.

**The rule: the engine's involvement with metadata is carry-and-return, on definitions only — and the line is authorship.** Display data for definitions you *author* (plugin operators, custom operators, fragments) goes in the in-definition bag, because it must travel with dynamically-registered definitions — a DB-stored fragment or a published plugin array has no tool-side channel of its own. Display data for definitions you *don't* author is the consuming tool's API (an editor component prop, a docs-generator option), merged tool-side. Expected tool-side layering, recorded as a pattern the editor package owns, not spec: tool built-ins ← definition bags via `getOperators()` ← host overrides passed to the tool. Consequently **core operator definitions ship with no display keys at all** — the engine carries zero UI knowledge, in spirit as well as mechanism.

### The editor-hints module

The v2 seed-and-colour data is genuinely useful and shouldn't die with the fields it was hiding in — but it doesn't belong in the runtime either. It moves to a **sub-path export**, working name `fig-tree-evaluator/editor-hints` (final name and packaging mechanics → Packaging area): a plain typed data module mapping canonical operator names to suggested display values — colours, per-parameter editor seeds — consumable by [fig-tree-editor-react](https://github.com/CarlosNZ/fig-tree-editor-react) or any other tooling as its fallback layer.

- **Zero runtime coupling**: never imported by the core entry point, never in the evaluation bundle; purely data.
- **Co-versioning is why it lives in this repo** rather than the editor's: the hints describe *this package's* operator set — per-parameter seeds name parameters this spec defines — so an operator or parameter change and its hint update land in the same PR, and drift is caught where it starts.
- The module's exported type doubles as the **documented key convention** for definition authors who want their plugin/custom operators to display well in the same tools.
- **v2 disposition**: the `data.ts` parameter seeds and the fragment/custom-function `textColor` / `backgroundColor` fields typed into the v2 runtime ([types.ts:100-110](../src/types.ts#L100-L110)) are the content this module absorbs.

### `useCache` metadata defaults

Pure operators default `false` (memoizing cheap computations is bookkeeping without benefit); I/O operators default `true` (v2 precedent — the whole point of the result cache). Each pass states its value; the blanket `useCache` option overrides per Options.

## Batch 1 — Logic & control — **Draft**

### `and` / `or`

| Parameter | Type                   | Required | Default | Null policy                                                  | Evaluation                                    |
| --------- | ---------------------- | -------- | ------- | ------------------------------------------------------------ | --------------------------------------------- |
| `values`  | array (elements `any`) | ✓        | —       | `value` — elements are truthiness positions; `null` is falsy | **parallel, early resolution + cancellation** |

- Each element is judged by shared FigTree truthiness; the result is an **actual boolean**, never an operand (Type area).
- **Parallel, with early resolution and cancellation.** All operands start concurrently (v2 continuity — [AND/operator.ts:7](../src/operators/AND/operator.ts#L7)); the first falsy (`and`) / truthy (`or`) value **resolves the node immediately and cancels the remaining in-flight operands** via a derived abort scope. Sequential short-circuit was considered and rejected: it buys laziness by serializing the everyday case — an `and` of independent checks needs *all* of them on the expected all-truthy path, so parallelism is the right default (deliberate contrast with `firstOf`, whose expected path needs only the first candidate). Cancellation is not new machinery: the whole-evaluation kill switch already requires abort threading through every node and into the HTTP/SQL clients (Node grammar, fallback rule 3); this applies the same mechanism at node scope (ledger #7). Pure subtrees stop at the next node boundary; in-flight requests abort via the threaded signal.
- **Deterministic error semantics — Kleene's strong logic.** An operand failure fails the node only if the result actually depends on that operand: a deciding value resolves the node even when another operand has already failed, the failure discarded along with the cancelled work — `and(error, false)` → `false`, `and(error, true)` → error, mirrored for `or`. Completion timing changes only speed and how much work gets cancelled, never the outcome. Cancellation is not failure (an aborted operand raises nothing — the node has already resolved), and a discarded failure is not an error of the evaluation: it does not surface under `mode: 'report'`; `trace` shows failed and cancelled operands.
- Side-effect note, recorded honestly: every operand *starts* — an `http` POST operand is always initiated, though it may be aborted mid-flight. Operands shouldn't rely on order or on being skipped; a mutation that must fire conditionally belongs behind `if`.
- **Empty input: vacuous identity — a recorded exception to the empty-aggregate policy.** `and: []` → `true` ("all of these are true" holds vacuously), `or: []` → `false` ("at least one is true" fails). A dynamically-supplied condition list legitimately has unknown cardinality — zero elements is an answer, not an anomaly — and the quantifier cousins must agree: *constraint recorded for batch 5*: `every: []` → `true`, `some: []` → `false` (JS parity). A literal `[]` gets a `validate()` warning (dead expression), not an error. The generating rule that keeps this from reopening the arithmetic question (recorded for batch 3): **vacuous identity is granted only where the identity element is unique and type-stable** — boolean conjunction/disjunction qualify; `plus`'s identity is mode-ambiguous (`0` / `""` / `{}` — v2's roulette) and `min`/`max` have none in the domain (`±Infinity` aren't values), so the arithmetic aggregates stay errors.
- Positional: `['...values']`. `useCache`: `false`.
- **v2 disposition**: `values` kept; parallel evaluation kept, early resolution + cancellation added; the v2 metadata defaults (`[true, true]` / `[true, false]`) were editor seeds, not runtime defaults — they move to the editor-hints module per the conventions (§ Runtime `default` vs editor seed); `values` stays required with no runtime default.

### `not`

| Parameter | Type  | Required | Default | Null policy                                                             | Evaluation |
| --------- | ----- | -------- | ------- | ----------------------------------------------------------------------- | ---------- |
| `value`   | `any` | ✓        | —       | `value` — truthiness position; `null` is falsy, so `not: null` → `true` | eager      |

- Boolean negation of FigTree truthiness. Everyday face is the single-value payload: `{ $not: '$data.user.disabled' }`; the null-is-falsy reading makes `{ $not: '$data.x' }` the idiomatic "is unset/empty" test. *The `not(null)` → `true` reading is under review — negating a propagated null turns absence into affirmation ([v3-cases-for-review.md](v3-cases-for-review.md) #9); a flip to `propagate` would break this idiom.*
- `condition` was considered as the name and rejected: `not` negates the truthiness of any *value*, and `value` matches the single-input convention.
- Positional: `['value']`. `useCache`: `false`. New in v3 (`!` meant NOT_EQUAL in v2).

### `if`

| Parameter   | Type  | Required | Default | Null policy                                                                             | Evaluation |
| ----------- | ----- | -------- | ------- | --------------------------------------------------------------------------------------- | ---------- |
| `condition` | `any` | ✓        | —       | `value` — truthiness position                                                           | eager      |
| `then`      | `any` | ✓        | —       | `value` (type `any` includes `null`, so an explicit `null` branch is an ordinary value) | **lazy**   |
| `else`      | `any` | optional | `null`  | `value`                                                                                 | **lazy**   |

- **Exactly one branch evaluates** — v2 continuity ([CONDITIONAL/operator.ts:21-23](../src/operators/CONDITIONAL/operator.ts#L21-L23)), now declared metadata (ledger #1) rather than hand-rolled.
- `else` is optional: omitted + falsy condition → `null`, via the ordinary optional-default machinery — no special case, and it composes with the standard null gradient downstream. `then` stays required: an if with no consequent has no honest meaning (`{ condition, else }` intent is written `not` + `then`).
- **`fallback` does not stand in for a missing `else`.** An unmet condition with no `else` is *success with `null`*, not failure — `fallback` catches failures only (the absence/failure boundary, References). A `fallback` on the `if` node still catches genuine failures in the condition or the taken branch. "Value when the condition isn't met" is precisely what `else` is for. Recorded explicitly because it will be a recurring user question. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #1.*
- Because both branches are `any`-typed, null-means-unset never triggers on them — `any` includes `null` (Type area), so `else: null` is a value, not an unset. Same observable result either way; recorded for precision.
- No `elseIf` parameter — chains are nested `if`s or a `match`; a chained face would change arity by mode, exactly what the mode rule (Operators) forbids.
- Positional: `['condition', 'then', 'else']` — `{ $if: [c, a, b] }` and `{ $if: [c, a] }` (else omitted) are both legal, matching the payload-table example already in the spec.
- `useCache`: `false`.
- **v2 disposition**: `condition` kept; `valueIfTrue` / `valueIfFalse` → `then` / `else` ([CONDITIONAL/data.ts:16](../src/operators/CONDITIONAL/data.ts#L16), [:24](../src/operators/CONDITIONAL/data.ts#L24)); property aliases `ifTrue` / `ifFalse` / `ifNot` die; v2 required both branches — `else` becomes optional.

### `match`

| Parameter  | Type                                                                                    | Required | Default | Null policy                                                                                                       | Evaluation                                                                                     |
| ---------- | --------------------------------------------------------------------------------------- | -------- | ------- | ----------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `value`    | `string` \| `number` \| `boolean` \| `null`                                             | ✓        | —       | `value` — `null` is a legal match value that matches no branch (Type area)                                        | eager                                                                                          |
| `branches` | object — a literal map (static keys, expression values) **or** a node computing the map | ✓        | —       | map mode: *(structural — not a value position)*; dynamic mode: a `null` result is a runtime type error (`reject`) | **map: lazy per branch** — only the matched branch evaluates; **dynamic: eager**, whole object |
| `default`  | `any`                                                                                   | optional | —       | `value`                                                                                                           | **lazy**                                                                                       |

- **Matching** is by the canonical string form (Type area, agreed): branch keys are JSON object keys, hence always strings; `value: 1` matches the key `"1"`, `true` matches `"true"`. A `null` match value matches no branch and falls to `default`.
- **No match and no `default` → runtime failure** (fallback-catchable), the same posture as v2's final throw ([MATCH/operator.ts:42](../src/operators/MATCH/operator.ts#L42)) — but the default branch is now a declared parameter. v2's magic `fallback` key *inside* the branches object ([MATCH/operator.ts:37](../src/operators/MATCH/operator.ts#L37)) dies: it collided with the reserved node key while meaning something different, and it made one branch name unusable as data.
- **Two modes — the fragments-`parameters` pattern exactly**, on the principle that every parameter is evaluable (a structural-only `branches` was considered and rejected as the arbitrary exception: fragments' `parameters` already established the two-mode grammar, and authors shouldn't carry a mental list of which object parameters evaluate). Mode is decided statically at parse by the standard node-classification grammar. A **plain-object** `branches` is the branch map: keys are static data, values are expressions, and only the matched branch evaluates (v2 continuity, now declared metadata — ledger #3). A **node or reference** `branches` (`'$data.statusLabels'`, a `$buildObject`, a fragment call) evaluates **eagerly, whole-object** ([MATCH/operator.ts:23-25](../src/operators/MATCH/operator.ts#L23-L25) made explicit); its result must be an object, its values are runtime data, and the matched value returns **verbatim** — never re-parsed or re-evaluated (References §4; v2 re-evaluated the extracted branch value, [MATCH/operator.ts:32-33](../src/operators/MATCH/operator.ts#L32-L33) — for data-sourced branches an injection path, dead). The everyday lookup-table case reads as it should: `{ $match: { value: '$data.status', branches: '$data.statusLabels', default: 'Unknown' } }`. The editor badges dynamic-branches calls, as it badges dynamic-arguments fragment calls.
- **Branch keys in the literal map are data — with two classification edges, recorded honestly.** Because the mode-deciding grammar must run against the `branches` value, a literal map can collide with it: (a) a map containing a branch key named `operator` / `fragment` classifies as a malformed or unknown node → **loud parse error**; (b) a map whose keys are *exactly one* recognized `$name` (plus at most reserved-key names) classifies as a shorthand node and silently becomes dynamic mode — possible only for single-branch matches, the same accepted hazard class as the sibling-key rule (multi-branch maps with a `$name` key fail loud as malformed nodes). The escape for colliding keys is to supply the map dynamically (`buildObject`, `vars`, or from data) — a *built* object's keys are runtime data, never classified. This amends Node grammar's "a branch legitimately named `operator` is safe" note: safe from v2's silently reading the node's own reserved properties, but loud-error rather than working as a literal map key (pointer added there).
- Positional: `['value', 'branches', 'default']`. `useCache`: `false`.
- **v2 disposition**: `matchExpression` → `value` (alias `matchValue` dies); `branches` kept as the one home for branches (aliases `arms` / `cases` die); root-hoisted branch keys die (Node grammar); the pairs-array branches form (`[key, val, key, val, …]` via [operatorUtils.ts:57-59](../src/operators/operatorUtils.ts#L57-L59)) dies — object form only; dynamic branch objects **kept, made explicit** as the second mode (with extracted values no longer re-evaluated); in-branches `fallback` key → the `default` parameter.

### `firstOf`

| Parameter | Type                   | Required | Default | Null policy                                            | Evaluation                         |
| --------- | ---------------------- | -------- | ------- | ------------------------------------------------------ | ---------------------------------- |
| `values`  | array (elements `any`) | ✓        | —       | `value` — skipping nulls *is* the operator's semantics | sequential, stop at first non-null |

- Evaluates candidates **in order**, returning the first non-`null` result; later candidates never evaluate. The sequencing is semantics, not optimization — a backup `http` candidate must not fire when the primary resolved. This is the deliberate mirror of `and`/`or`'s parallelism: their expected path needs every operand, so they parallelize and cancel; `firstOf`'s expects the first candidate to answer, so it never starts work it may not need. Note also the contrast with truthiness (Type area, agreed): `firstOf` skips *only* `null` — `""`, `0` and `false` are answers, not absences.
- **All candidates null → `null`.** Absence in, absence out — erroring here would defeat the layered-defaults gradient the operator exists to serve.
- **Empty input → `null`**, the degenerate all-null case, with a `validate()` warning on a literal `[]` (dead expression). This is a recorded exception to the empty-aggregate error policy, and a principled one: `and`/`or` must invent a boolean for empty input, while `firstOf`'s contract — "null when nothing is found" — is already total. Dynamic empty input behaves identically (`null`), which is exactly what a caller of an absence tool wants.
- **Failures propagate — `firstOf` skips nulls, not errors.** The absence/failure boundary (References) holds: a candidate that *fails* fails the node per the fallback rules. The composition recipe, recorded because it's the intended idiom: demote a risky candidate's failure to absence with its own fallback — `{ $firstOf: [{ $http: …, fallback: null }, '$data.cached', 'default'] }`.
- Positional: `['...values']` — `{ $firstOf: ['$data.nickname', '$data.name', 'Anonymous'] }`. `useCache`: `false`. New in v3.

### Batch 1 signature summary

```
and(values…)   or(values…)   not(value)
if(condition, then, else?)
match(value, branches, default?)
firstOf(values…)
```

## Batch 2 — Comparison — **Draft**

Shared shape, stated once: all six take their operands as `values`, return actual booleans (ordering may propagate `null`), and are pure (`useCache: false` throughout). A `values` that is *itself* `null` — a dynamic supply (`{ $greaterThan: '$data.pair' }`) whose path is missing — is a **runtime type error** in all six: the null policies below are element-wise, and the parameter itself is required with type `array`, which null isn't (ledger #8). That check sits in the pre-execution parameter-check layer, so `runtimeTypeCheck: false` degrades it to undefined behaviour, consistent with that option's agreed scope.

**Negation rule, recorded for the whole batch**: *negate a comparison by choosing its complement operator, never by `$not`-wrapping.* For the ordering four the two are not equivalent — `$not` over a propagated null turns absence into affirmation: `{ $not: { $greaterThan: ['$data.age', 18] } }` with `age` missing yields `true`, while the complement `{ $lessThanOrEqual: ['$data.age', 18] }` yields `null`, falsy downstream. Logged as [v3-cases-for-review.md](v3-cases-for-review.md) #9, where the `not(null)` → `true` reading is itself flagged for reconsideration. With `notEqual` in the set, every comparison has a direct complement.

### `equal`

| Parameter         | Type                                       | Required | Default | Null policy                                                                                                        | Evaluation      |
| ----------------- | ------------------------------------------ | -------- | ------- | ------------------------------------------------------------------------------------------------------------------ | --------------- |
| `values`          | array (elements `any` — opaques admitted)  | ✓        | —       | `value`, element-wise — null is comparable: `null` = `null` → `true`, null vs anything else → `false` (Type area) | eager, parallel |
| `caseInsensitive` | `boolean`                                  | optional | `false` | null-means-unset                                                                                                   | eager           |

- **All-equal**: every element deep-equals the first (equality is transitive, so pairing is irrelevant). Deep structural, key-order-insensitive — v2's dequal semantics kept. **Total**: always an actual boolean — a cross-type comparison is `false`, never an error (`1` ≠ `"1"`; Type area, agreed).
- **Variadic, and total over 0/1 elements: `equal: []` → `true`, `equal: [x]` → `true`.** The contract — "all elements are equal" — is a universally-quantified predicate that already holds vacuously over zero or one element; nothing is invented. This is a *different* justification from the fold-identity rule (batch 1), which stays untouched for arithmetic: predicates are total by contract, folds need an identity element. The dynamic case genuinely wants it ("all reviewers gave the same score" with one reviewer → `true`); a *literal* array of fewer than 2 elements gets a `validate()` warning (dead expression), matching `and`/`or`.
- **`caseInsensitive` is string-only and shallow**: it folds operands that are strings; it never reaches inside nested structures (deep folding would need a custom comparator and raises keys-vs-values murk — a documented limit, not an oversight). On non-string operands the flag is inert — observably equivalent to v2's fold-only-if-all-strings ([EQUAL/operator.ts:25-28](../src/operators/EQUAL/operator.ts#L25-L28)), since any string/non-string pair already yields `false` cross-type. This parameter is the flagship `operatorDefaults` example (Options).
- **Opaque values** (deferred from the Type area, settled here): `Date` compares by time value, `RegExp` by source + flags — dequal/lite's explicit branches, blessed as defined behaviour; they're the opaques that actually arrive in data objects, and the honest companion to the death of the accidental `Date > Date` coercion. Every other opaque compares by dequal's fallback (reference identity, or the own-enumerable-property walk for object-typed instances) — deterministic but *unspecified*, and never an error: equality stays total.
- **Considered and rejected: early resolution + Kleene** (a first mismatching pair decides `false`, discarding a failing operand, `and`/`or`-style). Technically the same shape — all-equal *is* `and` over pairwise equalities — but the payoff there was expensive independent checks, and comparison operands are almost always cheap data references. All operands evaluate concurrently; **any operand failure fails the node** — deterministic, nothing discarded.
- Positional: `['...values']` — `{ $equal: ['$data.role', 'admin'] }`; dynamic whole-array supply per the rest-position rule. `caseInsensitive` is named-only: the rest must be last, so there is no trailing option slot (the same shape question `join` faces in batch 4).
- `useCache`: `false`.
- **v2 disposition**: aliases `eq` / `equals` die; `nullEqualsUndefined` dies with `undefined` itself (Type area — nothing left to equate); the global `caseInsensitive` option → `operatorDefaults`; the editor-seed default moves to the hints module.

### `notEqual`

| Parameter         | Type                                       | Required | Default | Null policy                                                        | Evaluation      |
| ----------------- | ------------------------------------------ | -------- | ------- | ------------------------------------------------------------------ | --------------- |
| `values`          | array (elements `any` — opaques admitted)  | ✓        | —       | `value`, element-wise — powers the is-set idiom (below)            | eager, parallel |
| `caseInsensitive` | `boolean`                                  | optional | `false` | null-means-unset                                                   | eager           |

- **The exact boolean negation of `equal`, and nothing else** — one shared implementation (`not(allEqual)`), `caseInsensitive` passing straight through. Negation-by-construction has concrete v2 motivation: the hand-negated copy drifted into a real bug — [NOT_EQUAL/operator.ts:39-40](../src/operators/NOT_EQUAL/operator.ts#L39-L40) tests the impossible `value === null && value === undefined`, so with `nullEqualsUndefined: true` a null first operand made `notEqual(null, 5)` return `false`.
- **The >2 reading is "not all equal", *never* "all pairwise distinct"**: `notEqual: [1, 2, 1]` → `true`. The batch's one real misread hazard, carried entirely in the name; v2 semantics kept, and the operator description states it loudly.
- Vacuous cases follow by negation: `[]` → `false`, `[x]` → `false`; same `validate()` warning on literals under 2 elements.
- Null-as-comparable powers the blessed is-set idiom `{ $notEqual: ['$data.x', null] }` — the precise presence test (truthiness misfires on legitimate `0` / `""` / `false`).
- **Kept deliberately, as pure sugar** — the only comparison exactly expressible by composition: because `equal` is total, `not(equal(…))` is equivalent in every case including errors. Kept anyway for: the uniform negation rule above (dropping it would make equality the one comparison you must `$not`-wrap), the is-set idiom staying one node, sheer frequency of negative equality tests, and `!=` familiarity. Interlock recorded in case #9: if `not(null)` flips to `propagate`, `$not`-wrapping comparisons becomes safe and this rationale rests on frequency + idiom alone.
- Positional: `['...values']`. `useCache`: `false`.
- **v2 disposition**: `!` is reassigned to `not`; `ne` dies; otherwise as `equal`.

### `greaterThan` / `greaterThanOrEqual` / `lessThan` / `lessThanOrEqual`

One pass for the four: identical shape, one relation apiece, and a single shared ordering implementation parameterized by direction and inclusivity (the `equal`/`notEqual` shared-implementation argument, applied again).

| Parameter | Type                                                              | Required | Default | Null policy                                                                       | Evaluation      |
| --------- | ----------------------------------------------------------------- | -------- | ------- | --------------------------------------------------------------------------------- | --------------- |
| `values`  | array — **exactly 2**, homogeneous: both `number` or both `string` | ✓        | —       | **`propagate`**, element-wise — a null operand resolves the node to `null` (Type area) | eager, parallel |

- **Semantics**: strict `>` / inclusive `>=` / strict `<` / inclusive `<=`. Numbers compare numerically — the engine finite-number guard (batch 3) means there are no NaN/Infinity cases to define. Strings compare in plain codepoint order (Type area, agreed: deterministic, locale-independent); gotchas recorded for docs: `"Z" < "a"` (uppercase sorts first) and `"10" < "9"` (string order is not numeric). The equality half of the inclusive pair is primitive equality on the homogeneous pair — dequal is not involved.
- **Heterogeneous operands are a runtime type error** — booleans, arrays, objects and opaques are out entirely. The deliberate asymmetry with equality, restated: `equal` answers cross-type questions with `false`; ordering refuses them (Type area, agreed — v2's accidental `Date > Date` via `valueOf` coercion dies here).
- **Result is `boolean | null`**: the propagate policy is the composition gradient the Type area used to sell the default (`{ $greaterThan: ['$data.age', 18] }` with `age` missing → `null` → falsy in condition position), re-confirmed here. One edge recorded loudly because users will expect otherwise: **`>=` / `<=` on two nulls is `null`, not `true`** — propagation resolves the node before any comparison, and "null equals null" belongs to `equal`'s `value` policy exclusively ([v3-cases-for-review.md](v3-cases-for-review.md) #10).
- **Operand failure beats propagation**: both operands evaluate; a failing operand fails the node even when the other is already `null` (the same no-Kleene stance as `equal`).
- **Exactly 2**: a literal `values` of any other length fails `validate()`; a dynamic one fails at runtime ("`greaterThan` takes exactly 2 values, got 3") — the constraint is parameter metadata (ledger #9), not a positional-surplus error, so the dynamic whole-array supply gets the same check. v2 silently ignored everything past the second element ([GREATER_THAN/operator.ts:19](../src/operators/GREATER_THAN/operator.ts#L19)) — dies. Chained comparison (`a > b > c` as three elements) considered and rejected: chains are written with `and`.
- **Considered and rejected: null coerces to `0`** ("any positive number is greater than null"). It readmits the most notorious case of the JS relational coercion v3 killed (`null >= 0` → `true`), and zero sits mid-number-line, so it doesn't even deliver "null sorts below everything": `{ $greaterThan: ['$data.temperature', -5] }` with `temperature` missing → `true` (missing data is "warmer than −5"), and `{ $lessThan: ['$data.age', 18] }` with `age` missing → `true` ("an unknown age is definitely a minor") — the case-#9 absence-becomes-affirmative trap baked into the operator with nothing to wrap. Strings would need a `""` twin and `[null, null]` couldn't pick a mode (the `plus` identity ambiguity again); keying the policy to the sibling operand's runtime type would break the per-parameter-metadata model (fixed point 2). The author who wants zero-when-missing states it at the site, as a claim about that field: `{ $greaterThan: [{ $firstOf: ['$data.age', 0] }, 18] }`. SQL agrees: `NULL > 5` is unknown in `WHERE` even though `ORDER BY` defines null placement — where nulls *sort* is a sorting-parameter question, not comparison truth.
- No `caseInsensitive` on ordering — v2 didn't have it, and case-folding an *order* invites locale semantics; fold explicitly with `lower` (batch 4) if needed.
- **Convention amendment, recorded**: `values` here names a fixed 2-tuple, bending the naming convention's "the operand array of a *variadic* operator is named `values`". Amended to: **`values` is the operand array, variadic or fixed-arity** — the named-pair alternatives all read badly, and the array face `{ $greaterThan: ['$data.count', 0] }` is already the spec's own idiom.
- Positional: `['...values']`. `useCache`: `false` (all four).
- **v2 disposition**: `strict` dies, replaced by the operator split — settling issue #22's commented-out alias-extension idea ([GREATER_THAN/operator.ts:7](../src/operators/GREATER_THAN/operator.ts#L7)). Recorded as motivation: v2's metadata documented `strict` defaulting to `false` ([GREATER_THAN/data.ts:19](../src/operators/GREATER_THAN/data.ts#L19), [LESS_THAN/data.ts:19](../src/operators/LESS_THAN/data.ts#L19)) while both implementations defaulted it to `true` ([GREATER_THAN/operator.ts:11](../src/operators/GREATER_THAN/operator.ts#L11), [LESS_THAN/operator.ts:11](../src/operators/LESS_THAN/operator.ts#L11)) — documented and actual defaults contradicted each other, so the strict/inclusive boundary was never reliable; the split retires the trap. Converter: `{ operator: '>', strict: false }` → `greaterThanOrEqual`, `{ operator: '<', strict: false }` → `lessThanOrEqual` (mechanical, already in the migration table). Aliases `higher` / `larger` / `smaller` die; `lower` dies here and is reborn as the batch-4 string operator — no overlap within any single tree version, but the converter must map v2 `lower` → `lessThan` (migration-doc line).

## Batch 3 — Arithmetic & math — **Draft**

Batch-wide ground rules, inherited rather than invented, stated once: operands are **numbers**, with two deliberate exceptions (`plus`'s mode polymorphism; `min`/`max`'s string ordering); every scalar parameter **propagates null** — arithmetic is the Type area's archetype for the default, and since [v3-cases-for-review.md](v3-cases-for-review.md) #3 (the null-operand row) is now a headline review item, every `propagate` cell below is provisional in the aggregate; array parameters follow batch 2's template — element-wise policy, whole-`null` array is a runtime type error (ledger #8), and **failure beats propagation** (everything evaluates concurrently; any operand failure fails the node; nulls decide only when nothing failed). No operator polices `NaN` / `±Infinity`: the **engine finite-number guard** (Type area) converts any non-finite result into an ordinary runtime failure at the node boundary — newly ledgered as #10, since batch 2 already leaned on it and every division-by-zero and overflow case below is a structural consequence of it. Why twelve operators rather than modes: the invariant-signature rule (Operators) — `round(value, decimals)`, `pow(base, exponent)` and `min(values…)` can't share a signature, so they don't share a name; `plus` alone keeps a mode selector because its signature never varies. All twelve are pure: `useCache: false` throughout; nothing here is lazy, sequential or structural.

### `plus`

| Parameter | Type                                                  | Required | Default                        | Null policy                                                              | Evaluation      |
| --------- | ----------------------------------------------------- | -------- | ------------------------------ | ------------------------------------------------------------------------ | --------------- |
| `values`  | array — elements homogeneous, one of the four modes   | ✓        | —                              | `propagate`, element-wise — a null operand resolves the node to `null` (#3) | eager, parallel |
| `expect`  | `'number'` \| `'string'` \| `'array'` \| `'object'`   | optional | — (mode inferred from operands) | null-means-unset                                                         | eager           |

- **Four modes, dispatched on operand type**: all numbers → sum; all strings → concatenation; all arrays → concatenation (`{ $plus: [[1, 2], [3, 4]] }` → `[1, 2, 3, 4]` — each element is an operand, per the rest-only mapping); all objects → **shallow** merge, later operands' keys winning ([PLUS/operator.ts:31-32](../src/operators/PLUS/operator.ts#L31-L32) continuity — deep merge is unearned; drill and rebuild). The result is always the mode's own type. **Mixed operands are a runtime type error** — the headline death of the coercion table (`{ $plus: [1, "2"] }` → `"12"` dies; Type area), taking v2's cross-mode accidents with it: string/array concat mixing, where the *first* operand's type decided the outcome ([PLUS/operator.ts:27-28](../src/operators/PLUS/operator.ts#L27-L28)), and boolean "addition" via the JS `+` fallthrough ([PLUS/operator.ts:35](../src/operators/PLUS/operator.ts#L35)). Dispatch is semantics, not validation — it survives `runtimeTypeCheck: false` (Type area, agreed).
- **`expect` is the renamed mode selector — and it asserts, never converts.** v2's `type: 'string'` *coerced* operands before reducing; v3's `expect: 'number'` pins the mode and fails loud when any operand isn't that type — `convert` is the only cast in v3. The name is chosen against exactly that misreading: `type` is dead (the three-meanings overload, Node grammar), `to` is `convert`'s cast selector and would promise conversion, `as` reads as a cast in every language that has one and is already the iterators' binding name (batch 5), `mode` says nothing about which way the enforcement points. `expect` states the semantics: a declared expectation the operands must meet. The everyday case is dynamic data — `{ $plus: { values: '$data.scores', expect: 'number' } }` is guaranteed arithmetic; strings arriving in `scores` are a loud error, never a silent concatenation. A literal `expect` over literal operands checks at `validate()`; dynamic values land on the runtime check as usual.
- **Empty input: error unmoded, identity when the mode is pinned.** Unmoded `plus: []` has no unique identity — `0` / `""` / `[]` / `{}` is v2's roulette — so the batch-1 generating rule keeps it an error ([v3-cases-for-review.md](v3-cases-for-review.md) #2): literal `[]` fails `validate()`, a dynamically-empty array is a runtime failure, and the blanket host remedy stays `operatorDefaults: { plus: { fallback: 0 } }` (Options). But **with `expect`, the identity is unique and type-stable — exactly the generating rule's grant condition** — so `{ $plus: { values: '$data.lineItemPrices', expect: 'number' } }` over an empty cart returns `0` (`""` / `[]` / `{}` for the other modes): the pinned mode answers the question the unmoded operator can't, and more sharply than the fallback remedy, which also swallows genuine type errors. A literal `[]` *with* `expect` downgrades to a `validate()` dead-expression warning, matching `and`/`or`. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #11.*
- **Propagation precedes the assertion**: `{ $plus: { values: [null, 5], expect: 'number' } }` is `null`, not a type error — null policies apply engine-side before the operator body runs (implementation notes), and `expect` is a mode pin, not a strictness flag. If the group review flips row #3, this follows automatically.
- Single operand → returned verbatim (the mode/`expect` checks still apply; the fold is trivial).
- Positional: `['...values']` — `{ $plus: [1, 2, 3] }` → `6`; dynamic whole-array supply per the rest-position rule. `expect` is named-face only (the rest must be last — the `equal.caseInsensitive` shape).
- `useCache`: `false`.
- **v2 disposition**: `values` kept; `type` → `expect`, coerce → assert (the `'bool'` / `'boolean'` mode values die — there is no boolean addition); the empty-input return-the-input quirk ([PLUS/operator.ts:19](../src/operators/PLUS/operator.ts#L19) — empty in, `[]` out regardless of mode) dies per the stances above; aliases `add` / `concat` / `join` / `merge` died in Operators (`join` is reborn as the batch-4 string operator).

### `subtract` / `divide` / `modulo`

One pass for the three non-commutative binaries: identical shape, one operation apiece — and one naming rule.

| Parameter                                | Type     | Required | Default | Null policy | Evaluation |
| ---------------------------------------- | -------- | -------- | ------- | ----------- | ---------- |
| `value`                                  | `number` | ✓        | —       | `propagate` | eager      |
| `minus` / `by` / `mod` *(per operator)*  | `number` | ✓        | —       | `propagate` | eager      |

- **The shape is `(value, «infix»)` — the infix rule**: the main operand first (the naming convention's main-input rule), then a second parameter named as the word spoken between the operands, so the named face reads as the expression — `subtract(value, minus)` ("10 minus 3"), `divide(value, by)` ("10 divided by 3"), `modulo(value, mod)` ("10 mod 3"). One generating rule, not three bespoke words: the author's natural unprompted guess *is* the name, which is the plain-English rule doing real work at the parameter level (conventions §). Deliberately *not* batch 2's `values` pair: comparison operands are peers — the node asks a question *about* the pair — while these operators transform a primary input, a real role asymmetry that named parameters make visible in the editor's fields and the error messages ("`minus` is required" beats "expected 2 values, got 1"). `divide.by` is v2's own property alias for the divisor ([DIVIDE/data.ts:27](../src/operators/DIVIDE/data.ts#L27)), promoted to the one name. The rule is recorded here for later passes to reuse or decline deliberately; it applies only where a natural infix word exists — `pow` keeps its role names.
- **Considered and rejected**: a uniform `by` across the triple (an earlier draft of this pass — it reads literally only for `divide`, and "subtract 10 by 3" isn't English; uniformity of the generating rule beats uniformity of the literal string); `values: [a, b]` (uniform with the comparisons, but then the named face says nothing the positional face doesn't); `dividend` / `divisor` (rule-0 jargon); v2's `from` / `subtract` (a parameter echoing its own operator name, and the dual-spelling problem — below). One wrinkle recorded honestly: `mod` half-echoes `modulo` — it survives where `subtract.subtract` didn't because the echo is semantic rather than literal (no error message or conversation can confuse the two strings), and "value mod 3" is the universal spoken form, where the technically-correct `modulus` is jargon *and* still echoes.
- Positional: `['value', 'minus']` / `['value', 'by']` / `['value', 'mod']` — `{ $subtract: [10, 3] }` → `7`, `{ $divide: [10, 4] }` → `2.5`, `{ $modulo: [10, 3] }` → `1`: positional order is math-notation order. Surplus elements are a parse error (v2 silently ignored everything past the second value — dies).
- **`divide` is true division, always.** v2's `output` mode selector dies: `'quotient'` is the composition `{ $floor: { $divide: [a, b] } }`, `'remainder'` is `modulo` — both readings now have honest names. Note v2's *runtime* default was already true division (the switch's default case, [DIVIDE/operator.ts:28-29](../src/operators/DIVIDE/operator.ts#L28-L29)); the `'quotient'` in its metadata was an editor seed that never fired. Division by zero is a runtime failure — structural under the finite guard (ledger #10), not a bespoke check; v2's hand-rolled throw ([DIVIDE/operator.ts:21](../src/operators/DIVIDE/operator.ts#L21)) becomes redundant.
- **`modulo` is floored — the result takes the sign of `mod`**: `modulo(-7, 3)` → `2`, `modulo(7, -3)` → `-2`, identical to JS `%` whenever both operands are positive (the overwhelming case). The name commits to it: the operator is called `modulo`, not `remainder`, and its headline uses — cycling indices, bucketing, parity — always want the non-negative answer for a positive `mod`; JS's truncated `%` (sign of the dividend) is precisely the version that breaks `index mod length` on negative input. No continuity cost: v2's only remainder was DIVIDE's `output: 'remainder'` mode, which dies with `output`. A zero `mod` fails via the guard, same as division.
- `useCache`: `false` (all three). `modulo` is new in v3.
- **v2 disposition**: `subtract`'s dual spelling dies — v2 accepted *both* `values: [a, b]` *and* named `from` / `subtract` (plus alias `subtractFrom`), with `values` silently winning when both were present ([SUBTRACT/operator.ts:15](../src/operators/SUBTRACT/operator.ts#L15)); `divide`'s `dividend` / `divisor` and their aliases (`divide`, `by`, `divideBy`) collapse into `value` / `by`; converter lines: `output: 'quotient'` → `floor(divide)`, `output: 'remainder'` → `modulo`.

### `multiply`

| Parameter | Type                      | Required | Default | Null policy                | Evaluation      |
| --------- | ------------------------- | -------- | ------- | -------------------------- | --------------- |
| `values`  | array (elements `number`) | ✓        | —       | `propagate`, element-wise  | eager, parallel |

- Numbers only — there is no concat/merge cousin to be polymorphic with, so no `expect`.
- **Empty input → `1`, the vacuous product** — the one arithmetic aggregate the generating rule actually reaches. Batch 1's sweep ("the arithmetic aggregates stay errors") never had a premise covering `multiply`: its stated grounds were `plus`'s mode-ambiguity and `min`/`max`'s missing identity, and `multiply` has neither problem — numbers-only means the identity is `1`, unique and type-stable, full stop. The honest options were to amend the rule with a plausibility clause or to follow it; the rule wins: a compounding-factors list (`{ $multiply: '$data.discountFactors' }`) over zero applicable discounts is exactly `1`, mathematical convention agrees (the empty product), and v2's empty-`multiply` → `0` ([MULTIPLY/operator.ts:12](../src/operators/MULTIPLY/operator.ts#L12)) was the roulette's worst cell — not even the identity, the annihilator. Literal `[]` gets the `validate()` dead-expression warning, matching `and`/`or`. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #12, where the symmetric alternative (error, keeping "arithmetic aggregates error on empty" as one teachable sentence) is recorded.*
- Single operand → itself. Overflow fails via the guard.
- Positional: `['...values']`. `useCache`: `false`.
- **v2 disposition**: `values` kept; empty → `0` dies as above; aliases `x` / `times` died in Operators.

### `pow`

| Parameter  | Type     | Required | Default | Null policy | Evaluation |
| ---------- | -------- | -------- | ------- | ----------- | ---------- |
| `base`     | `number` | ✓        | —       | `propagate` | eager      |
| `exponent` | `number` | ✓        | —       | `propagate` | eager      |

- The names are the precise, searchable terms (rule 0's survival clause — the same grounds as the operator name itself), and the spec already uses this signature as the mode-rule's worked example (Operators § The canonical list).
- JS `**` semantics with the guard closing the holes: overflow (`pow(10, 400)` — the Type area's own example) and complex results (`pow(-8, 0.5)` → NaN) fail as ordinary runtime failures. `pow(0, 0)` → `1` (JS/IEEE parity), recorded.
- Positional: `['base', 'exponent']`. `useCache`: `false`. New in v3.

### `round`

| Parameter  | Type      | Required | Default | Null policy                | Evaluation |
| ---------- | --------- | -------- | ------- | -------------------------- | ---------- |
| `value`    | `number`  | ✓        | —       | `propagate`                | eager      |
| `decimals` | `integer` | optional | `0`     | **null-means-unset** → `0` | eager      |

- **The first `integer` parameter** — the refinement the Type area added to the vocabulary with this exact case as its example; a fractional or non-numeric `decimals` is a runtime type error.
- **And the motivating case for null-means-unset, landed**: `{ operator: 'round', value: '$data.price', decimals: '$data.settings.precision' }` with the setting unset rounds to `0` decimals via the layered default chain — the Type area's worked example, now normative. The resulting one-node asymmetry (null `decimals` defaults, null `value` propagates) is logged for the group view — [v3-cases-for-review.md](v3-cases-for-review.md) #14.
- **Negative `decimals` round to powers of ten**: `round(1234, -2)` → `1200` — Excel's ROUND precedent, free once the type is `integer`, and the natural spelling of "to the nearest hundred".
- **Ties round half away from zero**: `round(2.5)` → `3`, `round(-2.5)` → `-3` — the school rule, sign-symmetric, what a config author means by "round". Deliberately *not* JS `Math.round`'s half-toward-+∞ (`Math.round(-2.5)` → `-2`), a parity nobody expects — and no v2 continuity exists to defend it, since v2 had no rounding operator. Implementation constraint, recorded in the implementation notes: rounding must operate on the decimal representation, not naive scale-and-round float arithmetic, so `round(1.005, 2)` is `1.01`, not `1`.
- Positional: `['value', 'decimals']` — `{ $round: [3.14159, 2] }` → `3.14`; `{ $round: 3.7 }` → `4` (single payload → `value`). `useCache`: `false`. New in v3.

### `floor` / `ceil` / `abs`

| Parameter | Type     | Required | Default | Null policy | Evaluation |
| --------- | -------- | -------- | ------- | ----------- | ---------- |
| `value`   | `number` | ✓        | —       | `propagate` | eager      |

- `floor` toward −∞, `ceil` toward +∞ — by definition, recorded so nobody expects truncation: `floor(-3.5)` → `-4`, `ceil(-3.5)` → `-3`. Truncation-toward-zero deliberately gets no operator until a use case earns one.
- No `decimals` on the pair — scaled flooring is rare, `round` covers the everyday scaled case, and an optional parameter is non-breaking to add later.
- `abs`: absolute value; nothing to decide.
- Positional: `['value']` each. `useCache`: `false`. All three new in v3.

### `min` / `max`

| Parameter | Type                                                | Required | Default | Null policy                       | Evaluation      |
| --------- | --------------------------------------------------- | -------- | ------- | --------------------------------- | --------------- |
| `values`  | array — homogeneous: all `number` or all `string`   | ✓        | —       | `propagate`, element-wise (#3, #13) | eager, parallel |

- **The aggregate cousins of the ordering comparisons, and defined by them**: the winning element under batch 2's relation — numbers numerically, strings in plain codepoint order, one shared comparator (implementation notes), homogeneity declared per ledger #9 (whose Needed-by gains this pair). Everything the ordering pass settled transfers: booleans, arrays, objects and opaques are out; heterogeneous input is a runtime type error; the codepoint gotchas (`"Z" < "a"`, `"10" < "9"`) carry over verbatim.
- **Strings earn their place via dates**: ISO-8601 timestamps order correctly by codepoint, so `{ $max: ['$data.createdAt', '$data.updatedAt'] }` is the everyday "most recent" — the same reasoning that kept strings in the ordering comparisons.
- Returns the winning **element itself**, never a normalization of it.
- **Null element → `null`** (element-wise propagate): "the least of these" is unknowable while one of them is absent. SQL aggregates skip NULLs instead — that is row #3's rejected alternative (b), same silent-arity objection; opt out at the site: `{ $min: [{ $firstOf: ['$data.a', 0] }, '$data.b'] }`. A whole-`null` `values` is a runtime type error (ledger #8). *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #13, interlocked with row 3.*
- **Empty input → error, no exception**: the domain has no identity — `±Infinity` aren't values (batch 1's grounds, restated as this batch's ruling) — and unlike `plus` no mode pin could mint one, so an `expect`-style parameter was considered and rejected: it buys no empty-input identity, and the element type doesn't fork the semantics the way add-vs-concat does, making it a pure assertion — validating against sample data is `validate()`'s job. Host remedy as ever: `operatorDefaults: { min: { fallback: … } }`.
- Positional: `['...values']`; dynamic whole-array supply per the rest rule. `useCache`: `false`. Both new in v3.

## Batch 4 — String — **Draft**

*`buildString` was drafted first, alone: it is the batch's headline operator — the consumer the agreed rendering rules were written for (null → `""`, the stringification table), and the pass whose rulings (token grammar, unmatched-token posture, the no-rescan rule) the rest of the batch inherits or deliberately contrasts with. `join`, `split`, `lower`/`upper`/`trim` and `regex` follow it below; `substring` was cut in review (its record closes the batch).*

Batch-wide ground rules for the remainder, stated once: everything is **pure** (`useCache: false` throughout) and eager — nothing here is lazy, sequential or structural. The batch has a deliberate split personality on input types, mirroring the Type area's cast-is-not-a-render line: the two **renderers** (`buildString`, `join`) accept `any` wherever text is being produced and render by the stringification table, while the **value operators** (`split`, `lower`, `upper`, `trim`, `regex`) take strict strings — `{ $lower: 42 }` is a runtime type error, never `"42"`; `convert` first. Required value inputs propagate null; optional settings are null-means-unset (the `round.decimals` showcase, applied throughout). And one convention is recorded here for every later pass — its first *chosen* instances are in this batch, where batches 2–3 were merely forced by rest-tails: **options never travel positionally**. Positional slots carry operands and data, whose meaning position can imply; flags and mode selectors (`split.trim`, `regex.flags`, `regex.mode`) are named-face only, because a bare trailing `false` or `'i'` in a positional list tells the reader nothing.

### `buildString`

| Parameter       | Type                                          | Required | Default | Null policy                                                                                                  | Evaluation      |
| --------------- | --------------------------------------------- | -------- | ------- | ------------------------------------------------------------------------------------------------------------ | --------------- |
| `template`      | `string`                                      | ✓        | —       | `propagate`                                                                                                  | eager           |
| `substitutions` | array (elements `any`) \| object (values `any`) | optional | `[]`    | parameter: null-means-unset → `[]`; elements/values: **`value` — null renders `""`** (References, agreed) | eager, parallel |
| `trim`          | `boolean`                                     | optional | `false` | null-means-unset → `false`                                                                                   | eager           |

- **Two token styles, one per `substitutions` shape.** Positional `%N` tokens pair with an **array**; named `{{name}}` tokens pair with an **object**. Mode dispatches on the resolved type of `substitutions` — the `plus` pattern exactly (dispatch is semantics, not validation, so it survives `runtimeTypeCheck: false`; any other resolved type is a runtime type error), and the signature never varies, so one operator. **Cross-style tokens are inert text in the other mode** — deliberate, and load-bearing: the template alone tells the reader which shape `substitutions` must be, emitting the other style's syntax verbatim is free (generating a Mustache/Handlebars template is positional mode's job; `%`-heavy text like percent-encoded URLs is safest in named mode), and inert-in-the-other-mode is half of the no-escape design (below).
- **Positional tokens: `%N` is the Nth substitution, 1-indexed, strictly** (*agreed in discussion — [v3-cases-for-review.md](v3-cases-for-review.md) #17*). Digits are greedy (`%12` is token twelve, not token one then `"2"`), repeats are legal (each occurrence renders the same once-evaluated value), and a `%` not followed by digits is plain text — `'20% off'` needs no ceremony. **Considered and rejected: rank-compaction** — v2's actual behaviour, the template's distinct tokens sorted and zipped against the array ([STRING_SUBSTITUTION/operator.ts:49-56](../src/operators/STRING_SUBSTITUTION/operator.ts#L49-L56)) — which makes the quick edit "just work" (`'My %1 is %3'` + `['name', 'Smith']` → `My name is Smith`) but conflicts with legal surplus (the i18n ground, below): with the full value set `['name', 'Carl', 'Smith']`, rank reads `%3` as *rank two* → `My name is Carl` — wrong data inside plausible output, silently, the gradient's worst failure mode. Strict index degrades the trimmed-set case to a *visible* literal `%3` (unbound-token rule, below) plus a `validate()` renumber hint at authoring, which is where the quick-edit slip belongs.
- **Named tokens: `{{name}}`, exact-match against the object's keys.** The token grammar is a plain identifier (the shared name-legality rule — vars/params names), matched verbatim; v2's *bare* path drilling inside tokens (`{{user.name}}`, `{{friends[0]}}` — [STRING_SUBSTITUTION/operator.ts:69](../src/operators/STRING_SUBSTITUTION/operator.ts#L69)) dies along with the substitutions-then-`data` precedence murk it rode on — a bare token is a substitutions key, full stop; drilling is written in the value expression (`substitutions: { friend: '$data.friends[0]' }`) or as a reference token (next bullet). A `{{` that doesn't open a well-formed token renders **literally** (the unbound-token rule, below; `validate()` flags it on literal templates) — the outcome worth engineering against was never the visible near-miss but v2's invisible one: a malformed `{{ name }}` routed the *entire surrounding text fragment* through key-extraction and rendered it as `""` ([STRING_SUBSTITUTION/operator.ts:73-79](../src/operators/STRING_SUBSTITUTION/operator.ts#L73-L79) + [:108-112](../src/operators/STRING_SUBSTITUTION/operator.ts#L108-L112)), swallowing everything to the nearest valid-token boundary.
- **Reference tokens: `{{$data.user.firstName}}` — a token whose body is a reference *is* that reference.** Proposed by Carl (July 2026), adopted with three guardrails. (1) **Literal templates only, recognized at parse** — the References static-recognition rule (rule 4) applied verbatim: a literal template is authored tree, a dynamic template is runtime data, so a data-sourced template's `{{$d.…}}` renders literally and can never mint itself a data read. (2) **Exactly the reference grammar, nothing else** — any reference References recognizes (`$data` / `$vars` / `$params` / `$element` / `$index`, short aliases, `as`-names, path drilling); an unrecognized namespace (`{{$typo.x}}`) is not a reference and renders literally, matching the inert-namespace rule. (3) **Pure sugar over a substitution** — semantically identical to binding the same reference through a named substitution (an implementation may desugar exactly that way at parse): rendered by the table, null → `""`, and mode-independent — reference tokens never consult `substitutions`, so they work in positional templates and in templates with no substitutions at all. What it buys: the everyday one-liner `{ $buildString: 'Hello, {{$d.user.name}}!' }` (single payload → `template`, nothing else), and iterator bodies that read as sentences — `{ $buildString: 'Order {{$order.id}}: {{$e.name}} × {{$e.qty}}' }`. It also means a `vars` block plus a template can carry a whole named-mode use case with no `substitutions` object at all. What it deliberately does not revive: v2's bare-identifier drilling and its precedence murk, and the injection path — a data-sourced template still can't read anything the author didn't hand it. Amendment pointer recorded at References rule 2, whose "embedding is `buildString`'s job" clause this makes literal.
- **The injection path dies with nothing replacing it.** v2's named mode fell back to resolving `{{name}}` directly from `data` when absent from `substitutions`, then **re-evaluated** the extracted value as an expression ([STRING_SUBSTITUTION/operator.ts:111-114](../src/operators/STRING_SUBSTITUTION/operator.ts#L111-L114)) — the exact violation References rule 4 was written against. In v3, data access is spelled with a reference — in a substitution value or a reference token; the hidden channel and the re-evaluation both die. To be precise about what *is* scanned: the template is scanned for tokens wherever it came from — a dynamic template is the operator's whole job, and tokens are `buildString`'s own grammar, not references or nodes — but token recognition is the only thing that ever happens to it.
- **Replacement is one pass; substituted values are never re-scanned.** A value containing `%2` or `{{x}}` renders as literal text — References rule 4's sibling rule at the token level, and the reason user-supplied data can never expand into other substitutions: `{ $buildString: ['Bio: %1', '$data.user.bio'] }` with a bio reading `"call me {{$d.users[0].phone}}"` outputs that text verbatim — the token scan ran over the authored template once, before any value existed, and never runs again.
- **No escape sequences — deliberately.** v2's backslash machinery dies without a successor: cross-mode inertness plus the no-rescan rule already make every literal expressible — a literal `%1` in a positional template is delivered *through* a substitution (`{ $buildString: ['Enter code %1 at checkout', '%1'] }` — the token renders the value `'%1'`, which arrives unscanned, so the output is the literal text `Enter code %1 at checkout`), a literal `{{` in a named template likewise, and each style is plain text in the other's mode. What the machinery cost v2: the strip only ever removed the *first* escape in the string ([STRING_SUBSTITUTION/operator.ts:64](../src/operators/STRING_SUBSTITUTION/operator.ts#L64)), and backslash escapes silently vanish in JS source before the operator ever sees them (`'\$5'` === `'$5'` — the README's own escaping example ships the unescaped string). The residual `%`-collision all but dissolves under the unbound-token rule: a percent-encoding in a positional template (`'…?q=%20foo'`) parses as token twenty, is unbound in any sane expression, and renders itself — the URL comes out intact (v2 silently rendered it as `""`). Only a template that *both* contains a literal `%N` *and* genuinely supplies an Nth substitution needs the pass-it-as-a-value out.
- **Unbound tokens render themselves.** The full rendering ledger: a bound substitution renders its value; a bound substitution that is *null* renders `""` (deliberate absence renders as nothing — References, agreed); a token with **no binding at all** — `%N` beyond the array, `{{name}}` with no such key, a near-miss `{{ name }}`, the other mode's token style — renders **its own text, verbatim**: `'My name is {{first}} {{last}}'` with `{ first: 'Carl' }` → `My name is Carl {{last}}`. The flagship application of the disruption gradient (conventions §): the end user sees slightly-odd output and keeps working; the admin they report it to sees exactly what's wrong. Considered and rejected on both flanks: **runtime failure** (this pass's own first draft — loudness at the wrong layer, blocking end users on a config slip) and **v2's silent `""`** (positional: unmatched tokens zip to `undefined` and `Array.join` renders them empty, [STRING_SUBSTITUTION/operator.ts:53-63](../src/operators/STRING_SUBSTITUTION/operator.ts#L53-L63); named: the `extractProperty` chain bottoms out at `''`, [:112](../src/operators/STRING_SUBSTITUTION/operator.ts#L112)) — the seam erased, a `{{nmae}}` typo just vanishing, which the gradient forbids as firmly as the error. The loudness lives where it belongs: `validate()` errors on every literal-face mismatch (unbound token, `%0`, near-miss braces, unused literal substitution — with a renumber hint for the gap case), and `trace` records every literally-rendered token. *Agreed in discussion — [v3-cases-for-review.md](v3-cases-for-review.md) #15; the positional gap-binding sub-question (#17) likewise resolved — strict index.*
- **Surplus substitutions are legal — and they still evaluate.** The asymmetry with unmatched tokens is the i18n reality: a dynamically-selected template legitimately uses a subset of the supplied values (locale strings vary in which slots they mention), so erroring on surplus would force per-template curation of the value set. `validate()` warns when a *literal* template ignores a *literal* substitution (an authoring slip, not a data condition). Eager honesty, matching `and`/`or`'s side-effect note: every substitution evaluates, concurrently with the template, used or not — and per the batch-2/3 posture (nothing discarded, failure beats propagation) **a failing unused substitution fails the node**. The shield recipe is the usual demotion: a per-value `fallback: null` turns a risky candidate's failure into `""` at the render. Considered and recorded as live alternatives rather than adopted: Kleene/used-set discard (batch 1's machinery — the render provably doesn't depend on the unused value) and lazy by-need evaluation (serializes dynamic templates behind template resolution, and forks literal/dynamic behaviour). *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #16.*
- **Rendering is the stringification table, and the result is always a string.** Strings verbatim, numbers in decimal form, booleans `"true"`/`"false"`, null `""` — and **composite values render a placeholder, not an error**: an array renders `<array>`, an object `<object>` (the Type-area table amended accordingly). The gradient again: `"Total price: <array>"` tells the admin precisely what arrived where text was expected, without blocking the render; v2's `"[object Object]"` via bare `String(sub)` ([STRING_SUBSTITUTION/operator.ts:55](../src/operators/STRING_SUBSTITUTION/operator.ts#L55)) dies to a legible placeholder. `validate()` errors where a literal substitution is statically composite; `trace` records placeholder renders. `convert` `to: 'string'` keeps **failing** on composites — a cast is not a render — alongside its propagate-null distinction (Type area). A single-token template returns the *rendering*, not the value — `{ $buildString: ['%1', 42] }` is `"42"`: `buildString` is a renderer, never a value pipe.
- **`trim` (default `false`) trims whitespace from both ends of every rendered substitution and reference-token value** — after stringification, before insertion; the template text itself is never touched, and null's `""` and the composite placeholders pass through unaffected. Reborn at the batch review (Carl, July 2026 — this pass's first draft dropped v2's `trimWhiteSpace` outright): the flag lives on both sides of the parse/render boundary as `split.trim`'s twin (valid use cases in both contexts — the wound v2 dressed is real: user input with stray whitespace already stored in the database), and only the *default* dies, since default-on was silent mutation of every rendered value. Default-off makes it a per-node opt-in, or a host-wide one via `operatorDefaults: { buildString: { trim: true } }` ("set and forget"). One shared whitespace definition with the `trim` operator and `split.trim` (implementation notes); named-face only, like every option in the batch (the rest tail forces it here anyway).
- **Names.** `template` — the plain-English word; the Conforma collision that killed `template` as the *operator* name doesn't reach a parameter scoped inside a `buildString` node. v2's `string` dies on sight: a type word as a name, the `type` overload's little sibling. `substitutions` — a distinct semantic role gets the role's name (conventions §); `values` would additionally misread in object mode. Long, but the positional face never types it.
- **The literal-object face and the classification edges.** A plain-object `substitutions` is an ordinary evaluable object parameter — keys are data, values are expressions (References rule 5; no new machinery) — while a node or reference (`substitutions: '$data.templateValues'`) supplies the whole map dynamically, its values arriving as runtime data, rendered via the table, never evaluated. The `match.branches` classification edges apply verbatim: a literal map with a key named `operator` / `fragment` classifies as a malformed node → loud parse error (rename the token or supply the map dynamically); the single-`$name`-key edge can't produce a *working* named template at all, since `$` is not a legal token character.
- Positional: `['template', '...substitutions']` — the rest-tail motivator (conventions §, ledger #5). The rest face always produces **array mode**; named mode is written with the named face. A single non-array payload binds to `template` (`{ $buildString: '$data.tpl' }` — legal, zero substitutions; any token then fails unmatched).
- `useCache`: `false`.

```js
{ $buildString: 'Hello, {{$d.user.name}}!' }                         // the one-liner: single payload → template; reference token, no substitutions
{ $buildString: ['%1 of %2 items', '$data.count', '$data.total'] }   // positional face → array mode
{ $buildString: { template: 'Dear {{first}} {{last}}',              // named mode — named face only
    substitutions: { first: '$data.user.firstName', last: '$data.user.lastName' } } }
{ $firstOf: [                                                        // dynamic template: propagate composes
  { $buildString: ['$data.i18n.greeting', '$data.user.name'] },     //   missing locale key → null template
  { $buildString: ['Hello, %1!', '$data.user.name'] },              //   → node null → firstOf falls through
] }
```

- **v2 disposition**: renamed from STRING_SUBSTITUTION (Operators; aliases `substitute` / `stringSub` / `replace` die there); `string` → `template`; property aliases `replacements` / `values`, `trim` / `trimWhitespace`, `subCharacter` / `subChar`, `numMap` / `numberMap` / `pluralisation` / `pluralization` / `plurals` all die (no parameter aliases). Three parameters change — one renamed with its default flipped, two dying: **`trimWhiteSpace`** → **`trim`, default `false`** — *revised at the batch review* (Carl, July 2026; this pass's first draft dropped it outright): the parameter survives as `split.trim`'s twin (valid use cases on both sides of the parse/render boundary), while the *default* dies as default-on silent mutation of every rendered value ([STRING_SUBSTITUTION/operator.ts:55](../src/operators/STRING_SUBSTITUTION/operator.ts#L55), [:79](../src/operators/STRING_SUBSTITUTION/operator.ts#L79)). Conversion is thereby faithful rather than a behaviour change: the converter writes `trim: true` on every converted node that relied on v2's default, and omits the parameter where v2 set `trimWhiteSpace: false` explicitly. **`substitutionCharacter`** — one token grammar; `$N` templates convert by mechanical rewrite (`$1` → `%1`); parenthetically, its metadata default said `'$'` while the runtime default was `'%'` ([data.ts:39](../src/operators/STRING_SUBSTITUTION/data.ts#L39) vs [operator.ts:18](../src/operators/STRING_SUBSTITUTION/operator.ts#L18)) — the editor-seed pattern again (cf. `greaterThan.strict`). **`numberMapping`** — dies (the composition head-to-head conceded — Carl, July 2026): an embedded pluralization DSL — stringly `">5"` keys, magic `other`, a `{}` self-slot, and precedence the author can't see (the code checked exact → `>` → `<` → `other` regardless of authored order, [operator.ts:120-138](../src/operators/STRING_SUBSTITUTION/operator.ts#L120-L138)) — whose recorded composition is an ordinary branch in the substitution (or `vars`) expression: `{ $match: { value: '$vars.count', branches: { 0: 'no friends 😢', 1: 'only one friend' }, default: { $if: [{ $greaterThan: ['$vars.count', 4] }, 'loads of friends', { $buildString: '{{$v.count}} friends' }] } } }` — every rule a real operator, precedence exactly as written; real plural rules (ICU) exceeded the DSL anyway — custom-operator territory. Behaviour kills, each argued above: rank-compaction (case #17 — strict index confirmed), the named-token data fallback and its re-evaluation injection, bare-token path drilling, silently-`""` unmatched tokens (replaced by the visible literal render), the escape machinery, and the malformed-token text swallow.

### `join`

| Parameter   | Type                   | Required | Default                     | Null policy                                                                                                                | Evaluation      |
| ----------- | ---------------------- | -------- | --------------------------- | --------------------------------------------------------------------------------------------------------------------------- | --------------- |
| `values`    | array (elements `any`) | ✓        | —                           | parameter: whole-`null` is a runtime type error (ledger #8); elements: **`value` — null renders `""`, delimiter kept** (agreed — #18) | eager           |
| `delimiter` | `string`               | optional | `' '` — shared with `split` | null-means-unset → `' '`                                                                                                     | eager           |

- **`join` is a renderer — `buildString`'s list-shaped sibling.** Each element renders by the stringification table (strings verbatim, numbers in decimal form, booleans `"true"`/`"false"`, null `""`, composites the `<array>` / `<object>` placeholder — the Type area's "`join` to confirm in its pass" flag, hereby confirmed), and the renderings concatenate with `delimiter` between each adjacent pair. The result is **always a string** — `{ $join: [42] }` is `"42"`, the single-token `buildString` rule's twin: a renderer, never a value pipe. Every element evaluates, concurrently, and any failure fails the node — nothing here is surplus, so `buildString`'s #16 question doesn't arise. `validate()` errors on statically-composite literal elements (ledger #11); `trace` records placeholder and null renders — both matching `buildString`.
- **Null elements render `""` and keep their delimiter**: `['123 Main St', null, 'Springfield']` with `', '` → `"123 Main St, , Springfield"`. The two candidate readings each win a real use case — the reason the Type area flagged this one: keep-the-slot wins on positional records (a CSV/TSV row built with `{ $join: { values: […], delimiter: ',' } }` **must** keep a null field as an empty slot, because skipping shifts every later column — wrong data inside plausible output, silently, the #17 failure family), while skip-element-and-delimiter wins on prose lists (the address above, where `"123 Main St, Springfield"` is what the author wanted). Keep-`""` is drafted on three grounds: it *is* the stringification table (null → `""` is the agreed row; skipping would be a bespoke third policy — row #3's rejected skip-null in rendering clothes); its failure mode is the visible one (a stray doubled delimiter is a seam the admin can see — the gradient); and the skip reading stays one node away — filter first, `{ $join: { values: { $filter: { input: '$data.addressLines', each: { $notEqual: ['$element', null] } } }, delimiter: ', ' } }` (batch 5) — while under skip the keep reading would be *inexpressible* (nothing can resurrect a dropped slot). SQL's `CONCAT_WS`, which skips NULLs, was the live alternative's precedent. *Agreed in discussion — [v3-cases-for-review.md](v3-cases-for-review.md) #18 (Carl, July 2026): a null element still occupies its position in the array, so it takes up its (empty) space in the output.*
- **Empty input → `""`** — the vacuous render, granted by the batch-1 generating rule: `join` is monomorphic (the result is a string whatever the elements), so the identity is unique and type-stable — the pin `plus` had to buy with `expect` (#11), `join`'s contract carries for free. The everyday dynamic case wants it: an empty tag list renders as nothing, not as a failure. A *literal* empty `values` gets the `validate()` dead-expression warning, matching `and`/`or` — and under the rest-only face (below), `{ $join: [] }` binds `values: []` directly, exactly the `and`/`or` shape. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #19.*
- **`delimiter` — one contract shared with `split`**: same name, same type, same default in both directions — "make text of this list" / "make a list of this text" are symmetric silences. A single space over the JS-ish `','`: the everyday silent case is prose (`"Carl Smith"` from name parts), and every structured format (CSV, paths, query strings) specifies its delimiter anyway. And over `''`: a default that runs words together helps nobody, while an explicit `delimiter: ''` *is* the honest spelling of pure concatenation — which is also the recorded contrast with `plus`: `plus` string mode concatenates string **operands**, strictly, no rendering (`{ $plus: ['a', 1] }` is a type error); `join` **renders, then concatenates** (`{ $join: { values: ['a', 1], delimiter: '' } }` is `"a1"`). **No delimiter escaping, deliberately — so no reversibility promise either** (flagged and accepted — Carl, July 2026): an element containing the delimiter shatters on the way back — `join` of `['one', 'two three']` on `' '` then `split` yields three elements — because nothing is escaped on the way out, matching `buildString`'s no-escape design; round-tripping with integrity guarantees is a serialization format's job, not a delimiter's. The infix rule (batch 3) was considered and declined deliberately, as its own text invites: `with` reads beautifully (`join` these `with` `', '`) but the shared-contract rule outranks the nicety — the pair must use one word, `with` can't serve `split`, and `delimiter` is the precise, searchable term v2's `split` already made canonical.
- **Positional: `['...values']` — rest-only; `delimiter` never travels positionally.** The contested face, decided the other way from this pass's first draft (`['values', 'delimiter']` — the `split` mirror) at the batch review (Carl, July 2026), and logged for the group view as case #22 — deliberately, since it's the first time an *(input, second-parameter)* shape tilts named-only when a positional slot was available. The winning grounds: the flat literal list is the commoner payload and now binds directly — `{ $join: ['$data.first', '$data.last'] }` — and the delimiter, though everyday, tends to be *consistent within a given environment*, so its natural home is `operatorDefaults: { join: { delimiter: ', ' } }` ("set and forget"), with the named face for per-node overrides. `{ $join: '$data.tags' }` keeps its dynamic whole-array reading (the `and` rule), and `{ $join: [] }` binds `values: []` like `and`/`or`. The recorded cost — the first draft's grounds: *(input, delimiter)* no longer travels positionally, so `{ $join: ['$data.tags', ', '] }` is a **two-element join** (the tags array renders as `<array>`, then `', '` renders as itself) — visible in output and `trace`, but invisible to `validate()` when the first element is dynamic — and the positional face no longer mirrors `split`'s `['value', 'delimiter']`. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #22.*
- `useCache`: `false`. **New in v3** — no v2 disposition beyond the recycled-name hazard already tabled: v2 `join` was a PLUS alias meaning concatenation; the converter maps it to `plus`, and the migration doc carries the loud callout.

```js
{ $join: '$data.tags' }                          // one-liner: dynamic whole-array supply, space-joined
{ $join: ['$data.first', '$data.last'] }         // flat literal list — the everyday positional face
{ $join: { values: '$data.tags', delimiter: ', ' } }
                                                 // per-node delimiter: named face (or set once via operatorDefaults)
{ $join: { values: { $map: { input: '$data.items', each: '$element.name' } }, delimiter: ', ' } }
                                                 // the idiomatic pipeline: extract, then render (batch 5)
```

### `split`

| Parameter   | Type      | Required | Default                    | Null policy               | Evaluation |
| ----------- | --------- | -------- | -------------------------- | ------------------------- | ---------- |
| `value`     | `string`  | ✓        | —                          | `propagate`               | eager      |
| `delimiter` | `string`  | optional | `' '` — shared with `join` | null-means-unset → `' '`  | eager      |
| `trim`      | `boolean` | optional | `true`                     | null-means-unset → `true` | eager      |

- **Divides `value` on each occurrence of `delimiter`; the pieces, in order, as an array of strings.** JS `String.split` core semantics: every occurrence splits, adjacent delimiters produce empty elements, and a leading or trailing delimiter produces a leading or trailing `""`. Those empties are **kept** — visible seams, per the gradient — where v2 quietly popped exactly one: `excludeTrailing` removed a *single* trailing empty element and nothing else ([SPLIT/operator.ts:30](../src/operators/SPLIT/operator.ts#L30)) — asymmetric (a leading delimiter kept its empty), bespoke, and seam-erasing for positional data (a trailing empty CSV field is a field). It dies without a successor parameter; cleanup is written where it's meant — `filter` (batch 5), the same recipe as `join`'s skip reading. Behaviour change, migration-doc line; the converter does not wrap (the `trimWhiteSpace` precedent from `buildString`'s disposition). Consequence recorded: `split('', ',')` is `[""]` — one empty field, the CSV reading — where v2's pop made it `[]`.
- **`trim` (default `true`) trims whitespace from each piece.** The flag now lives on both sides of the string boundary — revised at the batch review (Carl, July 2026): the first draft kept it here and killed it on `buildString`, an asymmetry the parse/render argument didn't earn, since there are valid use cases in both contexts. `split.trim` and `buildString.trim` are the same flag, one shared whitespace definition (implementation notes); what survives of the parse/render line is the **defaults**: `split` defaults *on* — it parses a serialized list, and whitespace around delimiters is the format's grease, not payload (`'a, b, c'` split on `','` means `['a', 'b', 'c']` in any human reading, and the untrimmed `['a', ' b', ' c']` is the answer nobody has ever wanted; v2 agreed — same default, [SPLIT/operator.ts:29](../src/operators/SPLIT/operator.ts#L29)) — while `buildString` defaults *off*, a renderer emitting values untouched unless asked. The opt-out is one flag (`trim: false`, for whitespace-significant input), the name is v2's own property alias promoted ([SPLIT/data.ts:26](../src/operators/SPLIT/data.ts#L26)) — `trimWhiteSpace` dies to the short word — and sharing the *operator* name `trim` (below) is deliberate, not a collision: same word, same meaning, parameter-scoped here.
- **`delimiter` — the shared contract (argued at `join`), and v2's third seed contradiction.** The default `' '` is v2's *runtime* default ([SPLIT/operator.ts:6](../src/operators/SPLIT/operator.ts#L6)) — while its metadata said `','` ([SPLIT/data.ts:21](../src/operators/SPLIT/data.ts#L21)): the editor-seed pattern's third sighting (cf. `greaterThan.strict`, `substitutionCharacter`), resolved as ever — the seed moves to the hints module, the runtime value becomes the one default. An **empty delimiter splits into code points** — `'abc'` → `['a', 'b', 'c']`, and a surrogate pair is never torn (JS's `split('')` shears emoji into UTF-16 halves — defined away). With `substring` cut from core, this bullet is the **batch unit ruling's home**: strings segment by Unicode code point, one shared helper (implementation notes), and the constraint carries forward — batch 5's `length` counts code points on strings, and a future custom `substring` should consume the same helper. Grapheme clusters stay out of scope, honestly: a family emoji or flag is *several* code points and may still split visually — Intl-segmentation territory.
- Positional: `['value', 'delimiter']` — v2's own `parseChildren` order ([SPLIT/operator.ts:36](../src/operators/SPLIT/operator.ts#L36)); `trim` is named-face only (the batch convention — this is its first chosen instance). `{ $split: '$data.fullName' }` — single payload → `value`, split on spaces; `{ $split: ['$data.csv', ','] }` — the everyday face. Note the deliberate asymmetry with `join` (#22): `split`'s input is a single string, so `delimiter` rides second with no rest-tail to compete against; `join`'s rest-bound elements claimed the whole payload and sent its delimiter to the named face.
- `useCache`: `false`.
- **v2 disposition**: alias `arraySplit` died in Operators. Property aliases die throughout (no parameter aliases): `value`'s alias `string` dies on sight — the type-word name again, cf. `buildString` ([SPLIT/data.ts:10](../src/operators/SPLIT/data.ts#L10)); `delimiter` kept as the one name, alias `separator` dies; `trimWhiteSpace` → `trim` (its aliases were `trim` / `trimWhitespace` — the good one is promoted, the rest die); `excludeTrailing` (aliases `removeTrailing` / `excludeTrailingDelimiter`) dies as above — behaviour change, flagged loud. The `','`-vs-`' '` default contradiction resolved above; editor seeds → the hints module.

### `lower` / `upper` / `trim`

One pass for the three unary normalizers — the `floor`/`ceil`/`abs` template: identical shape, one transform apiece.

| Parameter | Type     | Required | Default | Null policy | Evaluation |
| --------- | -------- | -------- | ------- | ----------- | ---------- |
| `value`   | `string` | ✓        | —       | `propagate` | eager      |

- **Strict strings in, strings out** — the batch ground rule with teeth: `{ $lower: 42 }` is a runtime type error, not `"42"` (a cast is not a render; `convert` or `buildString` first). Null propagates: `{ $lower: '$data.nickname' }` with the path missing is `null`, composing with `firstOf` downstream.
- **`lower` / `upper` apply the Unicode default case mappings — locale-independent, deterministic, no locale parameter.** The batch-2 stance (plain codepoint order, no locale semantics) applied to casing: identical results in every runtime and every user locale. Gotchas recorded for docs rather than decided away: the Turkish/Azerbaijani dotted-i (`lower('I')` is `'i'`, never `'ı'`), and full case mapping can change length (`upper('ß')` → `'SS'`). Locale-aware casing is plugin/custom-operator territory — the dates precedent.
- **The motivating idiom is key canonicalization**: `match` branch keys are case-sensitive data, so `{ $match: { value: { $lower: '$data.status' }, branches: { active: …, closed: … } } }` is the sanctioned spelling of case-blind dispatch — `equal.caseInsensitive` covers *equality* only, and batch 2's ordering pass already pointed here ("fold explicitly with `lower`") for case-folded comparison.
- **`trim` strips Unicode whitespace (the JS `trim` set — WhiteSpace + line terminators) from both ends** — the standalone site-by-site tool; `split.trim` and `buildString.trim` apply exactly this normalization per piece / per rendered value — one shared definition (implementation notes). `trimStart` / `trimEnd` deliberately get no operator until a use case earns one (the truncation-toward-zero precedent).
- Positional: `['value']` each — `{ $lower: '$data.email' }`, the single-payload face throughout. `useCache`: `false`. All three **new in v3**; the `lower` recycled-name hazard (v2 LESS_THAN alias) is already tabled, with the converter line v2 `lower` → `lessThan`.

### `regex`

| Parameter | Type                                 | Required | Default  | Null policy                 | Evaluation |
| --------- | ------------------------------------ | -------- | -------- | --------------------------- | ---------- |
| `value`   | `string`                             | ✓        | —        | `propagate`                 | eager      |
| `pattern` | `string`                             | ✓        | —        | `propagate`                 | eager      |
| `flags`   | `string`                             | optional | `''`     | null-means-unset → `''`     | eager      |
| `mode`    | `'test'` \| `'extract'` \| `'match'` | optional | `'test'` | null-means-unset → `'test'` | eager      |

- **One operator, three result modes — sanctioned by the invariant-signature rule**: the signature never varies (`value`, `pattern`, `flags`, `mode`); only the return shape follows the declared mode — the `plus` dispensation exactly. **`test`** (the default — v2 parity, [REGEX/operator.ts:14-15](../src/operators/REGEX/operator.ts#L14-L15)) returns an actual boolean: does the pattern match anywhere. **`extract`** returns the first matching substring, or **`null` when nothing matches** — no-match is absence, not failure ("no number in this string" is an answer), composing down the gradient: `firstOf` falls through it, `convert` propagates it. **`match`** returns *every* matching substring, in order, as an array — `[]` when nothing matches (a list of findings is naturally empty where a single finding is naturally absent). Results are `boolean` / `string | null` / `array` — or `null` by propagation from a missing `value`, the batch-2 `boolean | null` shape: `{ $regex: ['$data.email', …] }` with the path absent is `null`, falsy in condition position. *Provisional (the no-match cells) — [v3-cases-for-review.md](v3-cases-for-review.md) #21; interlock: batch 5's `find` should agree with `extract`.*
- **The numeric-mining constraint, discharged** (the Operators-table line this pass owed): v2's `outputType` number-mining respells as `extract` + `convert` — `{ $convert: { value: { $regex: { value: '$data.weight', pattern: '\\d+(\\.\\d+)?', mode: 'extract' } }, to: 'number' } }` turns `"15 grams"` into `15`, and a weight with no digits propagates `null` through `convert` instead of inventing one.
- **Capture groups never change the return shape** — `extract` and `match` always return whole-match text; a group in the pattern affects matching, not output. The alternative (groups reshape the result — JS `String.match` style) makes the return shape depend on the *content of a string parameter*, which a dynamic `pattern` turns into a shape-shifting node. Group extraction — numbered, or named-groups-to-object — is deferred until a use case earns it: a compatible future `mode` value, non-breaking to add (and made slightly likelier by `substring`'s cut, below — position-flavoured extraction now has no other core home than patterns).
- **`flags`: the JS flag letters, admitted subset `i` / `m` / `s` / `u`.** Anything else is an error — loudest for `g` and `y`, with the hint in the message ("all matches is `mode: 'match'`"). The `g` ban is load-bearing, not squeamish: in JS, `g` is the flag that makes a RegExp *stateful* (`lastIndex` — the alternating-`test()` footgun) and that flips `String.match`'s **return shape** — and in v3 the result shape is declared by `mode`, a literal-union parameter the engine and editor can read, never smuggled through the content of a string parameter (the capture-group rule's sibling: admitting `g` would force a meaning for `mode: 'extract'` + `flags: 'g'`, where either the flag is silently inert or a *flag* changes the node's type). Position-stateful scanning is the operator's own business — `match` applies the global scan internally; zero-length matches advance one code point, the JS `matchAll` rule. Duplicate letters error (JS parity).
- **The pattern grammar is the JS dialect, and literal faces validate at parse**: a literal `pattern` — the overwhelming case — is compiled at `validate()`, so a malformed regex is an authoring error, caught where the gradient wants it (ledger #11); literal `flags` and `mode` likewise (`mode` is a literal union — dynamic supply is legal and lands on the runtime check, the `convert.to` treatment). A dynamic pattern that fails to compile is an ordinary runtime failure, fallback-catchable. One caution recorded honestly: a *data-sourced* pattern is a resource hazard as well as a correctness one — catastrophic backtracking stalls synchronously, where even the evaluation kill switch can't interrupt mid-regex — so patterns should normally be authored tree, and a host that accepts patterns from data owns that exposure.
- **Names**: `testString` → `value` — the main-input convention; "test" presumed the one mode v2 had. `pattern` kept — the precise term, v2's own canonical. The selector is **`mode`** (Carl, July 2026 — this pass's first draft said `output`, "what does the operator output"; demoted because `mode: 'extract'` reads as spoken — *run in extract mode* — where `output: 'test'` half-reads as naming the output value, and `output` had already died once as v2 DIVIDE's selector). One objection recorded honestly, then defused: in regex vocabulary the *flags* are traditionally the "modes" (multiline mode, dotAll mode), so a reader might reach for `mode: 'multiline'` — but the literal union rejects it loudly at parse with the three real values in the message, and the docs say *result mode*. The values `test` / `extract` / `match` are the Operators-table's own sketch words, assigned so the mining word does the mining — `extract` pulls one value out — and the plural-shaped word returns the collection. **Considered and rejected: `match` / `matchAll`** (the JS-parity pair): they borrow names from exactly the corner of the JS API whose return shape is least predictable — `String.match` flips shape on the `g` flag and on capture groups, and `matchAll` yields an *iterator of exec-arrays* — while v3's modes return plain strings and arrays of strings under declared metadata, so the familiar names would promise shapes the semantics deliberately don't deliver (a half-kept familiarity misleads worse than a fresh word); rule 0's audience isn't presumed to know them anyway; and `match` already names a core operator, so the word's double duty stops at one mode value rather than two.
- Positional: `['value', 'pattern']` — v2's `parseChildren` order ([REGEX/operator.ts:19](../src/operators/REGEX/operator.ts#L19)); `flags` and `mode` are named-face only (the batch convention) — `{ $regex: ['$data.email', '^\\S+@\\S+\\.\\S+$'] }` is the everyday condition-position face, boolean by default. `useCache`: `false`.
- **v2 disposition**: aliases `patternMatch` / `regexp` / `matchPattern` died in Operators; `testString` → `value`, its aliases `string` / `value` die ([REGEX/data.ts:10](../src/operators/REGEX/data.ts#L10)); `pattern`'s aliases `regex` / `regexp` / `regExp` / `re` die; v2 compiled with no flags and returned only the test ([REGEX/operator.ts:14](../src/operators/REGEX/operator.ts#L14)) — `mode: 'test'` + `flags: ''` defaults are exact v2 parity, so conversion is a pure rename job; editor seeds → the hints module.

### `substring` — cut from core

**Cut at the batch review (Carl, July 2026), before it ever shipped**: targeting string innards by *character position* is not something this kind of config needs — `regex` is the obvious tool for reaching inside strings, and `extract` + `convert` already owns the flagship mining case — so `substring` moves to the **possible-custom-operators list** (v3-api.md, Operators § Deferred), to be built as a first-class custom operator once the main package is done, if a use case ever demands it. No v2 predecessor, so the cut costs no converter or migration line. What the drafted pass leaves behind: the **code-point unit ruling survives, re-homed to `split`** (empty-delimiter mode), still constraining batch 5's `length`; the negative-index question ([v3-cases-for-review.md](v3-cases-for-review.md) #20) is **mooted for core** and travels with the custom-operator entry, which carries the drafted design's one-line summary — the full pass survives in this file's git history. The honest residue, recorded: fixed-width field extraction ("characters 3–5 encode the region") now leans on `regex` lookarounds or that custom operator — acceptable, because fixed-width formats are rare in config data, and a tool this situational is exactly what first-class custom operators are for.
