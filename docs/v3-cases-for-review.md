# FigTree v3 ‚Äî the absence / failure / default gradient ‚Äî settled rulings

*Register of the gradient rulings collected during the operator parameter passes: for each edge case, does it yield `null`, fire `fallback`, or error? The end-of-passes **group review has taken place** (Carl, July 2026, concluding from the null-policy review ‚Äî [v3-null-policy-review.md](v3-null-policy-review.md)): every row below is **agreed**, and this register is now the consolidated behaviour reference. Its rows override any stale *Provisional ‚Äî #N* markers left in the passes (kept there as pointers); row numbers are stable, since the passes cite them. Per-row rationale and the considered alternatives live in the passes and in this file's git history. Revisit deliberately, not casually.*

## The two former headline questions ‚Äî resolved (Carl, July 2026)

- **`fallback` means failure, only** ‚Äî fixed point 1 confirmed as drafted throughout. The **`‚Ä¶Default` parameter family** is the softer catch for absence: `if.else`, `match.default`, `find.noMatchDefault`, `regex.noMatchDefault`, `get.missingPathDefault`, `sql.noRowDefault`, and the null-replacement `nullValueDefault` / `nullInputDefault` (ledger #18). Parameters answer *absence*, `fallback` catches *failure*, and the two compose on one node. The acknowledged cost ‚Äî authors face a "fallback or `‚Ä¶Default`?" choice ‚Äî is accepted, mitigated by editor and `validate()` guidance rather than by collapsing the distinction.
- **Null operands `propagate`** (row 3 and its cells) ‚Äî confirmed as the unadorned default, with `nullValueDefault` as the declared per-site / host-wide opt-out. **Recorded reservation (Carl)**: a future flip to reject (null operand = error) remains on the table ‚Äî see Open items, below.

## Fixed points ‚Äî confirmed

1. **Absence is not failure**: a missing `$data` path yields `null` (never throws, absent `strictDataPaths`); `fallback` triggers on node *failure* only. (References ¬ß Absence semantics; Node grammar ¬ß `fallback` semantics)
2. **Null policies are per-parameter metadata** ‚Äî `propagate` / `value`, declared exactly where the parameter's type names `null`; admission is type-driven (null at a required/element position is legal only where `null` is a named type member), so `reject` is derived from the type, never declared. `propagate` is the default for value inputs. (Type ¬ß Null policy)
3. **Null at an optional parameter means unset** ‚Äî the type-driven rule governs required/element positions only; optionals keep null-means-unset with clean types, unless the type names `null` deliberately. (Type ¬ß Optional parameters)
4. **Empty aggregate input is an error**, excepted only where the identity element is unique and type-stable. (Type ¬ß Aggregates; the generating rule from the `and`/`or` pass, batch 1)

Rows 2 vs 3 remain the sharpest deliberate contrast in the set: an *empty* sum is a failure a fallback can catch, while a sum with a *null operand* succeeds as `null` and the same fallback does nothing. The tools differ accordingly ‚Äî `expect` or a default `fallback` for emptiness, `nullValueDefault` for null operands ‚Äî and both rulings follow from the confirmed fixed points.

## Open items

1. **Row 3's reservation (with row 13 attached)**: Carl may yet flip null operands from `propagate` to reject (a null operand = error, caught by `fallback`). Parked deliberately as a single future decision ‚Äî it re-touches every arithmetic operator at once (including the propagation-precedes-`expect` ordering in the `plus` pass), and `nullValueDefault` would become the way to *avoid* the error rather than the null.
2. **The `length` seam, recorded at the #24 flip**: `length` still *propagates* null (`string | array | null` ‚Äî Type area), where the iterators now reject. The batch-5 draft called `length` this rule's twin; they have parted ways ‚Äî either a deliberate difference (an iterator *does*, `length` *measures*) or an invitation for the same reject treatment. Decide when next touching `length`.
