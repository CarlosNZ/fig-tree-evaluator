# FigTree v3 — cases for group review

*Register of ambiguous or controversial rulings encountered during the design passes — above all the **absence / failure / default gradient**: for each edge case, does it yield `null`, fire `fallback`, or error? Each case is recorded with its current provisional ruling so drafting can continue without re-litigating per pass; the whole set is adjudicated **together, at the end of the per-operator parameter passes** — interlocking calls like these are easier to judge as a group than one at a time.*

**Protocol**: a case listed here is **provisional even where its host section is marked Agreed** — listing here overrides the status marker for that specific ruling. At the group review each row is confirmed or flipped, outcomes flow back into the spec docs, and the row is marked resolved. A flip that contradicts one of the fixed points below must restate that rule, not silently except it.

*The null-policy review sheet ([v3-null-policy-review.md](v3-null-policy-review.md), July 2026) consolidates every parameter's null cell and the adopted `…Default` / null-replacement family (ledger #18) — read it alongside this register at the group review.*

## Fixed points the rulings derive from

The currently-agreed foundations. The group review may reopen them deliberately — but flipping one reshapes every row derived from it, so they're listed for visibility:

1. **Absence is not failure**: a missing `$data` path yields `null` (never throws, absent `strictDataPaths`); `fallback` triggers on node *failure* only. (References § Absence semantics; Node grammar § `fallback` semantics)
2. **Null policies are per-parameter metadata** from a fixed vocabulary — `propagate` / `value` / `reject` — with `propagate` the default for value inputs. (Type § Null policy) *Amended at the null-policy review (Carl, July 2026): admission is type-driven — null is legal at a required/element position only where `null` is a named type member, so `reject` is derived from the type, never declared; `propagate` / `value` remain the declared policies where null is admitted.*
3. **Null at an optional parameter means unset**, unless the declared type includes `null`. (Type § Optional parameters) *Scope confirmed at the null-policy review (option (a)): the type-driven rule above governs required/element positions only — optionals keep null-means-unset with no `| null` in their types.*
4. **Empty aggregate input is an error**, excepted only where the identity element is unique and type-stable (`and` / `or` / `firstOf`). (Type § Aggregates; the generating rule from the `and`/`or` pass, batch 1)

Rows 2 vs 3 are the sharpest contrast in the set and the reason this register exists: an *empty* sum is a failure a fallback can catch, while a sum with a *null operand* succeeds as `null` and the same fallback does nothing. Both follow from the fixed points; whether the composed behaviour is the desirable one is exactly the group-review question.

**The `fallback` question — second headline item (crystallized at batch 5; Carl, July 2026).** What does `fallback` *mean*: the failure-catch only (fixed point 1, drafted throughout), or v2's broader "the value when nothing comes back", firing on no-match and no-result outcomes as well as errors? Rows 1, 2/3, 5, 7, 21, 23 and 29 are all cells of this one question, and Carl's recorded dissents — #3's ceremony complaint, #23's conflation case — push in the same direction, which is why it's now named as its own item rather than adjudicated row by row. Restoring the v2 reading flips fixed point 1 and must restate it, per the protocol above. The costs to weigh, recorded honestly: under the broader reading a no-match result depends on whether a fallback happens to be present (success-with-null vs caught absence — a safety net that rewrites success), "guard failures but keep no-match as `null`" becomes inexpressible on one node, and every mechanism that leans on fallback-means-failure — the nearest-enclosing-catch rule, timeout shielding, `mode: 'report'` error collection, `operatorDefaults` default fallbacks — acquires an absence reading too. The drafted system's counter-offer — adopted into the draft at the batch-5 review, pending this call — is the absence-default parameter line recorded at #23: per-operator parameters named for their firing condition (`find.noMatchDefault`, suffix `…Default`; Part 1 § Naming conventions), so that **parameters answer absence, `fallback` catches failure**. An overall call on the key's meaning, to be made once, at the group review. Recorded at the null-policy review (Carl, July 2026): the parameter family is adopted across the set (the lazy absence defaults plus ledger #18's null-replacement defaults), with the acknowledged cost that authors face a "fallback or `…Default`?" choice — accepted, to be mitigated by editor and `validate()` guidance rather than by collapsing the failure/absence distinction. The overall call on `fallback`'s meaning remains the group review's.

## Notes & live alternatives, by row

1. **`if` with no `else`** — current rule: an unmet condition is success-with-null (fixed point 1); "value when unmet" is what `else` is for. Alternatives: (a) missing `else` + falsy condition = *failure*, so `fallback` fires — blurs fixed point 1; (b) `else` required (v2 behaviour) — kills the null-gradient convenience.
2. **`plus` empty** — alternatives: an identity value — but which? `plus` is mode-polymorphic (`0` / `""` / `{}`), the ambiguity behind v2's empty-aggregate roulette; this is why the vacuous-identity carve-out (fixed point 4) excludes it. Host remedy under the current rule: `operatorDefaults: { plus: { fallback: 0 } }` (Options § reserved-modifier extension). Second remedy since batch 3: pin the mode — `expect: 'number'` makes the identity unique, so empty input returns `0` (row 11) without conceding the unmoded error.
3. **`plus` with a null operand** — current: `propagate` (fixed point 2's default for value inputs). Alternatives: (a) `reject` — a null operand is a *failure*, so a fallback **can** catch it (strict sums, one gradient tool instead of two); (b) skip-null — sum the present operands (→ `5`), SQL-aggregate style, but silently changes arity; (c) an explicit parameter (`nullValue: 0`-style) — pushes the choice to authors, grows every signature. Author remedy under the current rule: wrap operands — `{ $firstOf: ['$data.a', 0] }`. **Carl (July 2026, batch-2 discussion): still unconvinced that reaching for `firstOf` should be the price of defaulting a null/missing value — the remedy feels like ceremony for the most common gradient case.** Global question, not per-operator (any resolution touches rows 1, 3, 5, 9, 10, 13 and possibly fixed point 1); flagged as a headline item for the group review; judgement explicitly reserved again at the batch-3 `plus` review. Batch 3 drafted to the current rule throughout — including that propagation precedes `plus.expect`'s assertion and `min`/`max`'s comparison — so a flip here re-touches every arithmetic operator at once. **Adopted since (null-policy review, July 2026): alternative (c) in per-operator, presence-sensitive form** — `nullValueDefault` on `plus` / `multiply` / `min` / `max` and the ordering comparisons (ledger #18): unset → propagate as drafted; supplied → each null operand replaced, host-wide via `operatorDefaults`. This answers the ceremony complaint inside fixed points 1–2; the row's global question (whether *propagate* remains the right unadorned default) stays open for the group review.
4. **`and` / `or` empty** — vacuous identity per the quantifier reading ("all of these are true" holds over zero elements), agreeing with `every` / `some` (batch 5 constraint). Alternative: error — rejected in discussion; dynamic condition lists legitimately have zero elements.
5. **`firstOf` exhausted** — `null` is the contract: it's the absence tool, and "nothing was there" is a total answer. Alternative: error when exhausted — would collapse absence into failure and make `firstOf` + `fallback` the defaulting idiom.
6. **`firstOf` failing candidate** — current: skips nulls, *not* errors (fixed point 1's boundary); the demotion recipe is a per-candidate fallback: `{ $firstOf: [{ $http: …, fallback: null }, 'default'] }`. Alternative: failures count as absence and are skipped — the resilient-backup-chain reading, attractive for exactly the `firstOf` use case, but it makes `firstOf` a silent try/catch and swallows real errors. Worked example for the review: `{ $firstOf: [{ $http: '…/primary' }, { $http: '…/backup' }] }` with the primary *down* (error, not null) — current rule: the node fails and the backup never fires, unless the primary carries `fallback: null`; skip-errors alternative: the backup fires automatically. Carl leaning confirm on the current rule (July 2026); kept for the group pass.
7. **`match` no-match** — current: error, keeping typo-loudness (v2 parity — a missed branch is likelier an authoring error than a data condition); `default` is the sanctioned soft path. Alternative: `null` on no-match (soft by default, consistent with `find`'s expected no-match-→-null). The disruption gradient recorded at batch 4 ([v3-operator-parameters.md](v3-operator-parameters.md) § conventions) puts fresh weight on the alternative — apply that lens deliberately at the group review.
8. **`match` null value** — current: null matches *no* branch (deliberately not the `""` branch key, even though the stringification table renders null as `""` — that near-miss is worth a loud note somewhere). Alternatives: (a) runtime type error on a null match value; (b) let it match a literal `"null"` branch key (rejected out of hand — stringly spooky).
9. **`not` over a propagated null — absence becomes affirmative.** The composed behaviour of two agreed rules: ordering comparisons propagate a null operand (fixed point 2), and FigTree truthiness reads `null` as falsy (Type § Truthiness), so `not(null)` → `true`. The trap: `{ $not: { $greaterThan: ['$data.age', 18] } }` with `age` missing yields `true` — "age unknown" silently becomes "definitely not greater" — while the direct complement `{ $lessThanOrEqual: ['$data.age', 18] }` yields `null`, which stays falsy downstream. Current mitigation is doc guidance recorded in batch 2: *negate a comparison with its complement operator, never `$not`*. Live alternative, flagged by Carl (July 2026): flip `not.value` from truthiness (`value` policy) to `propagate`, making `not` absence-transparent — `not(null)` → `null` — which defuses the trap and makes `$not`-wrapping safe. Costs: `not` no longer always returns an actual boolean (the Type-area "actual booleans, never operands" rule gains an exception), and the blessed is-unset idiom `{ $not: '$data.x' }` breaks precisely on the missing-path case it exists for (null would propagate through as falsy instead of reading `true`, while `0`/`""`/`false` still read `true` — a half-working test, the worst kind) — presence testing would fall back to `{ $equal: ['$data.x', null] }`. Interlock: a flip also relaxes the negate-via-complement rule, removing the strongest structural argument for keeping `notEqual` (which would then rest on frequency and the `!=` idiom alone). **Resolved (Carl, July 2026, null-policy review): `value` kept.** The deciding cases: the optional-flag idiom (`{ $not: '$data.disabled' }` — an absent flag must read *not disabled*; under `propagate`, `if(b)` and `if(not(b))` both take the else branch on missing data, a condition and its negation agreeing), complement coherence within the truthiness algebra (`or(b, not(b))` must not read `false` on a missing flag), and the is-unset idiom (which would half-break: `0`/`""`/`false` → `true` but missing-path null → falsy). The trap this row records is answered by guidance instead of a flip: `validate()` lints `$not` directly over a propagate-family node, and the lint message and docs explicitly recommend defaulting the absence at its source — `missingPathDefault` on the getter (`{ $greaterThan: [{ $get: ['age', 0] }, 18] }`) or `nullValueDefault` on the comparison. `notEqual`'s structural rationale stands.
10. **Inclusive ordering on two nulls** — users will expect `>=` to concede equality when both sides are "the same null"; it doesn't: propagation resolves the node before any comparison runs, and null-equals-null belongs to `equal`'s `value` policy exclusively (batch 2). The both-unset question is written with equality — `{ $equal: ['$data.a', '$data.b'] }` is `true` when both are null. Alternative, not seriously entertained: a both-null → `true` carve-out for the inclusive pair — it would make `>=` disagree with `>` about whether null is a value and reintroduce SQL's `IS NULL` special-casing in reverse. Listed mainly so the group review sees equality's null-as-comparable and ordering's propagate side by side.
11. **`plus` empty with `expect`** — the generating rule's grant condition (fixed point 4) is met once the mode is pinned: per mode the identity is unique and type-stable (`0` / `""` / `[]` / `{}`), and the everyday dynamic case — `{ $plus: { values: '$data.lineItemPrices', expect: 'number' } }` over an empty cart — wants `0`, not an error. Sharper than the `operatorDefaults` fallback remedy, which also swallows genuine type errors. Alternatives: (a) error regardless — `expect` stays a pure assertion and `plus` keeps one uniform empty stance, but the fallback ceremony is re-imposed on exactly the authors who pinned their mode; (b) grant the identity for `'number'` only — arbitrary; the rule's condition doesn't distinguish the modes; (c) *unmoded* empty input defaults to the number reading — `{ $plus: [] }` → `0` with no `expect` at all, minimizing author overhead for the dominant numeric case. **Carl tempted by (c) (July 2026, batch-3 review), conceded for now**: a concatenation author would receive a nonsensical `0` — the mode-ambiguity fixed point 4's carve-out exists to avoid — but the minimize-cognitive-overhead bias is recorded for the group review to re-weigh.
12. **`multiply` empty** — the one aggregate where fixed point 4's exception clause is satisfied without help: numbers-only means the identity is `1`, unique and type-stable. Batch 1's "arithmetic aggregates stay errors" sweep had no premise covering it — its grounds were `plus`'s mode-ambiguity and `min`/`max`'s missing identity, and `multiply` has neither problem. Motivation: compounding-factor lists (`{ $multiply: '$data.discountFactors' }` — zero applicable discounts is factor `1`), with the empty-product convention agreeing; v2 returned `0`, the annihilator rather than the identity ([MULTIPLY/operator.ts:12](../src/operators/MULTIPLY/operator.ts#L12)). Alternative: error — keeping "arithmetic aggregates error on empty" as one teachable sentence and reserving vacuous identity for the boolean quantifiers; the group review should weigh that teaching cost against rule-consistency. Carl leaning confirm on the vacuous product (July 2026, batch-3 review); kept for the group pass.
13. **`min`/`max` with a null element** — the SQL-trained expectation is skip-the-NULL (`MIN(3, NULL, 7)` → `3`); FigTree's element-wise `propagate` says "the least of these" is unknowable while one candidate is absent → `null`. The drafting rationale: silently shrinking the candidate pool misreports extremes — a missing sensor reading shouldn't quietly disappear from "max temperature" — and skip-null is row 3's rejected alternative (b) wearing aggregate clothes. Site opt-out is the usual wrap: `{ $min: [{ $firstOf: ['$data.a', 0] }, '$data.b'] }`. Alternative: skip-null, SQL parity — attractive precisely because SQL is where min/max intuitions come from, and this is arguably the operator pair where the skip reading is *most* defensible ("the least of what's there"). Interlock: this is the same policy cell as row 3 — whatever the group review decides there should decide this row the same way, or the asymmetry needs its own defense. Since the null-policy review, `nullValueDefault` (note 3's adoption) gives this pair the declared site opt-out; a skip reading would still be a flag, not a default.
14. **`round` with null `decimals`** — the deliberate asymmetry showcase: null at the *optional* `decimals` means unset (fixed point 3), so the metadata default `0` applies — `{ $round: [2.5, null] }` → `3` — while null at the *required* `value` propagates — `{ $round: [null, 2] }` → `null`. Same operator, same null, two behaviours, decided by declared parameter metadata — which is the point (the Type area's motivating case: a missing `settings.precision` shouldn't nullify a present price). The contentious edge: an author expecting null to poison the node gets a confidently-rounded number instead. Alternative: `propagate` on `decimals` too — restores "null poisons everything" uniformity, but then a missing *setting* nullifies present *data*, re-imposing exactly the `firstOf` armour fixed point 3 exists to remove. Listed so the group review sees fixed points 2 and 3 operating side by side on one node.

15. **`buildString` unbound token — renders itself.** The rendering ledger: bound → the value; bound-but-null → `""` (deliberate absence renders as nothing); unbound (`%N` beyond the array, `{{name}}` with no such key, near-miss braces, the other mode's token style) → the token text verbatim. First application of the **disruption gradient** ([v3-operator-parameters.md](v3-operator-parameters.md) § conventions, recorded July 2026): the pass's first draft ruled *error* and was reversed in discussion — a config slip must not block an end user, and the literal token is self-explanatory to the admin it gets reported to. v2's silent `""` in both modes ([STRING_SUBSTITUTION/operator.ts:53-63](../src/operators/STRING_SUBSTITUTION/operator.ts#L53-L63), [:112](../src/operators/STRING_SUBSTITUTION/operator.ts#L112)) is rejected just as firmly — the seam erased, a `{{nmae}}` typo invisibly vanishing. Compensations: `validate()` errors on literal-face mismatches; `trace` records literal renders. Bonus: the positional `%`-collision gotcha dissolves — `'…?q=%20foo'` renders intact unless twenty substitutions are genuinely supplied. Since the null-policy review (July 2026): `nullValueDefault` (ledger #18) is the opt-in replacement for the bound-but-null `""` render (`Name: <unknown>`); unbound tokens and composites are outside its reach.
16. **`buildString` failing unused substitution** — current: the batch-2/3 posture (everything supplied evaluates, any failure fails the node, nothing discarded) applied even though surplus values are *legal* (dynamically-selected i18n templates legitimately use a subset of the supplied values). Shield recipe: per-value `fallback: null` renders the failure as `""`. Alternatives: (a) **Kleene/used-set discard** — batch 1's `and`/`or` machinery: once the template resolves, the used token set is known, the render provably doesn't depend on the unused value, so its failure is discarded (and its in-flight work could even cancel via ledger #7); principled and consistent with batch 1, but it forks from batch 3's "failure beats propagation" simplicity and lets a broken expression lurk unnoticed until some locale's template finally uses the slot; (b) lazy by-need — only referenced substitutions evaluate; serializes dynamic templates behind template resolution and makes literal vs dynamic templates behave differently. The disruption gradient recorded at batch 4 tilts this row toward alternative (a): a failing unused value blocking the whole render is exactly the runtime disruption the stance disfavours — re-weigh at the group review. Interlock: if the group review generalizes the Kleene rule beyond `and`/`or`, this row should follow it.

17. **`buildString` positional gap binding — rank vs index.** Carl's reading (July 2026): rank-compaction, v2's actual behaviour ([STRING_SUBSTITUTION/operator.ts:49-56](../src/operators/STRING_SUBSTITUTION/operator.ts#L49-L56)) — the template's distinct tokens sorted and zipped against the array — so the quick edit "delete `%2` and its value" just works: `'My %1 is %3'` + `['name', 'Smith']` → `My name is Smith`, and the intent is unambiguous *when the counts pair up rank-wise*. The counter-example that keeps strict index drafted: surplus values are legal (row 16's i18n ground — supply the full value set, let each template use a subset), and the two readings disagree precisely there: `'My %1 is %3'` with the full set `['name', 'Carl', 'Smith']` reads `%3` as *rank two* → `My name is Carl` — wrong data inside plausible output, silently, the failure mode the disruption gradient rates worst of all. Strict index on the same expressions: full set → `My name is Smith` (correct, order-stable); trimmed set → `My name is %3` (a visible seam, row 15's remedy, plus a `validate()` renumber hint at authoring). The readings are mutually exclusive on identical inputs, so one had to lose. **Resolved (Carl, July 2026): strict index confirmed** — `My name is %3` is the correct output for the trimmed set; the silent-rebinding counter-example decided it, and it keeps row 15's index-flavoured "extra tokens render literally" rule coherent (rank-compaction contradicts it whenever numbering is non-dense).

18. **`join` null element — render `""` (delimiter kept) vs skip element-and-delimiter.** The two readings each win a real use case, which is why the Type area flagged it: keep-the-slot wins on positional records — a CSV/TSV row built with `join` must keep a null field as an empty slot, because skipping **shifts every later column** — wrong data inside plausible output, silently, the #17 failure family; skip wins on prose lists — address lines with a null `address2`, where `"123 Main St, Springfield"` is exactly what the author wanted and the drafted `"123 Main St, , Springfield"` is the visible seam. Keep-`""` drafted: it *is* the stringification table's null row (skip would be a bespoke third policy — row 3's rejected alternative (b) in rendering clothes); its failure mode is the visible one (gradient); and it keeps both behaviours expressible — the skip reading is one `filter` away (`{ $filter: { input: …, each: { $notEqual: ['$element', null] } } }`, batch 5), while under skip the keep reading is inexpressible (nothing can resurrect a dropped slot). Live alternative: skip — SQL `CONCAT_WS` precedent, and arguably the more everyday `join`. Interlock note: this is a *rendering* position, not arithmetic — null here already renders rather than poisons — so this row can flip independently of rows 3/13. **Resolved (Carl, July 2026): render `""` confirmed** — a null element still occupies its position in the array, so it takes up its (empty) space in the output string; the position-preserving reading carried it. Since the null-policy review: `join.nullValueDefault` (ledger #18) optionally replaces the `""` render — slot and delimiter still kept.
19. **`join` empty input** — `""` granted per fixed point 4: `join` is monomorphic (the result is a string whatever the elements arrive as), so the identity is unique and type-stable — the pin `plus` had to buy with `expect` (row 11), `join`'s contract carries for free. Everyday ground: an empty tag list renders as nothing, not as a failure. Alternative: error, for the one-sentence teachability of "aggregates error on empty" — the same trade as row 12 (`multiply`), and the two rows should be judged together: both are the generating rule honestly applied against the simpler headline.
20. **`substring` negative indices — from-end offsets.** Drafted to slice semantics: `-N` counts from the end (`end: -1` drops the last character). Grounds: the last-N idiom is real and its composition is ceremony (`subtract` + `length`, three nodes for four characters); `round`'s negative `decimals` is the precedent — a meaningful reading over an error, free once the type is `integer`. The hazard, recorded rather than hidden: a *computed* index that unexpectedly goes negative silently switches to tail-reading — plausible output, wrong data, the family the gradient rates worst; `trace` records resolved indices, but no runtime mechanism can distinguish the intended negative from the accidental one. Alternatives: (a) runtime error on negatives — loud, but kills the idiom and blocks end users on data-driven slips; (b) clamp negatives to 0 (JS `substring`) — the seam fully erased, strictly worse. If the group review rates the silent-flip hazard above the idiom, (a) is the fallback ruling. **Mooted (Carl, July 2026): `substring` cut from core** — position-targeted extraction is rare in config, and `regex` owns string innards; the operator moves to the possible-custom-operators list (v3-api.md, Operators § Deferred), taking this question with it unresolved. The row stays as the design record for whoever builds it.
21. **`regex` no-match — absence, not failure.** `extract` (first matching substring) returns `null` when nothing matches — "no match" is an answer, fixed point 1's spirit applied at the operator level; `match` (all matching substrings) returns `[]` — a list of findings is naturally empty where a single finding is naturally absent; `test` is total (`false`). Composition ground: the numeric-mining pipeline (`extract` + `convert to: 'number'`) degrades to `null` on digit-less input instead of failing the node — the gradient's preferred shape, and `convert` propagates it onward. Alternative: no-match fails the node (fallback-catchable) — makes the pipeline loud and forces per-site `fallback: null` on the most common regex outcome. Interlock: batch 5's `find` no-match (known future entry, below) should land the same way as `extract`; the rows exist to be judged together. **Adopted since (null-policy review, July 2026): `regex.noMatchDefault`** — `extract`'s declared no-match default, runtime default `null`, so this row's ruling is the unadorned behaviour; the `find` interlock is now mechanical (same name, same contract).
22. **`join` positional face — rest-only vs `['values', 'delimiter']`.** Not a gradient case; listed at Carl's request so the tilt can be checked for consistency against its cousins at the group review. Tilted (Carl, July 2026) to rest-only with a named-face-only `delimiter`, reversing the pass's first draft: the flat literal list is the commoner payload (`{ $join: ['$data.first', '$data.last'] }` should just work), and delimiters tend to be consistent within an environment — `operatorDefaults: { join: { delimiter: ', ' } }` is their natural home, demoting the per-node delimiter to the named face. The cousins to compare: `equal.caseInsensitive` and `plus.expect` went named-only *forced* (rest-tail — the same resulting shape, but no choice was involved); `split.delimiter` stays positional (`['value', 'delimiter']` — a single-string input leaves the second slot free, so the same conceptual parameter rides positionally on one sibling and named-only on the other); `regex.pattern` stays positional as an operand (as did the since-cut `substring`'s `start`/`end`) — and note the options-never-positional convention was never in play here, since a delimiter is data too, which is exactly why this was a live choice. Batch 6 added the family's newest pair: `get.missingPathDefault` rides positionally (`['path', 'missingPathDefault']` — defaulting is `get`'s everyday second input, and v2's own positional shape) while its sibling `find.noMatchDefault` stays named-only — the `split`/`join` asymmetry again. The first draft's grounds, recorded: the delimiter is `join`'s everyday second input (specified in most real uses), and the winning face's one trap — `{ $join: ['$data.tags', ', '] }` reads as *(input, delimiter)* but is a two-element join rendering `"<array> , "` — is invisible to `validate()` when the first element is dynamic, where the losing face's traps were all parse-loud. If the environment-consistent-delimiter premise proves wrong in the field, this flips back cleanly: the named face is legal under both readings.

23. **`find` no-match — absence, not failure — and the row where the `fallback` question came to a head.** The drafted ruling is the batch-5 landing of the interlock #21 anticipated: `find` answers "which element?", and *none* is an answer — `null`, composing with the defaulting idioms (`{ $firstOf: [{ $find: […] }, fallbackValue] }`) exactly as `regex` `extract` does. Empty `input` is the degenerate no-match — `null` on `firstOf`'s grounds (#5): an absence tool's contract is already total. The cost is the conflation: a *found* null element also returns `null` (`each: { $equal: ['$element', null] }` succeeds and returns the null it found), and no downstream composition can split the two apart — `firstOf` sees one null; only a re-query (`some` with the same predicate) distinguishes them. **Carl (July 2026, batch-5 review): a strong case for restoring the v2 reading of `fallback`** — fallback fires on "nothing found", while a genuinely found `null` returns `null`: the conflation dissolves with zero new parameters, and it matches what v2 muscle memory always expected the key to do. That reading contradicts fixed point 1, so per the protocol it must be flipped deliberately and globally, not excepted per-operator — logged as the headline `fallback` question (below the matrix) rather than decided here. Counter-proposal from the drafting side: an absence-default **parameter** on `get`'s missing-default contract (fires on *no-match only*; a found null passes through unchanged — the parameter since named `missingPathDefault` in batch 6) — it resolves the conflation *inside* fixed point 1 and keeps `fallback` orthogonal for genuine predicate failures (the two compose on one node: "none matched" vs "the search broke"). **Adopted at the review (Carl, July 2026), with the name sharpened per operator: `noMatchDefault`**, not a uniform `default` — the firing condition is load-bearing precision, so it lives in the name; no special global property, just an ordinary parameter whose name varies with its purpose (now a naming convention — Part 1 § Naming conventions — with the `…Default` suffix as the family marker; knock-on: `get`'s drafted `default` re-weighs in batch 6; `match.default` keeps the bare word as the established dispatch term). A bonus the parameter spelling buys over the fallback spelling: `operatorDefaults: { find: { noMatchDefault: … } }` gives a host-wide no-match default without touching failure semantics. Also considered: no-match fails the node outright — the `match` #7 posture; rejected in drafting because `find` is a *search over data* where `match` is a *dispatch over branches*: a missed dispatch is likelier an authoring error, an empty search result is ordinary data. Judge together with #21, #5 and the headline question — if the group review restores the v2 `fallback` reading, `noMatchDefault` is re-examined as redundant machinery. The family has since grown per the null-policy review (July 2026): `regex.noMatchDefault`, `sql.noRowDefault`, the null-replacement `nullValueDefault` / `nullInputDefault` (ledger #18), and `get`'s parameter sharpened to `missingPathDefault`.

24. **Iterator `input` null — unknown, not empty.** Drafted to `propagate` (fixed point 2's default; `length`'s twin — the Type area lists `length` under propagate): a missing collection makes every iterator resolve `null`. The live alternative is **null-as-empty** (iterate zero times: `map`/`filter` → `[]`, `some` → `false`, `every` → `true`, `find` → `null`), and each reading owns a bad cell — which is why the row exists. **Worked comparison for the review — judge the pair together.** The quiet-is-a-trap cell: `{ $every: { input: '$data.checks', each: '$element.passed' } }` with `checks` missing — propagate (drafted): `null`, falsy downstream; null-as-empty: `true` — vacuous truth over a list that isn't there, silent affirmation from absence, the #9 failure family with no doc-guidance escape (and row #3's rejected skip-null wearing collection clothes). The loud-is-a-cost cell: `{ $join: { values: { $map: { input: '$data.tags', each: '$element.label' } }, delimiter: ', ' } }` with `tags` missing — propagate (drafted): `map` yields `null`, and the node **fails** (whole-null at `join.values` is ledger #8's type error), needing a `fallback` or `firstOf` at the seam — #3's ceremony complaint resurfacing in pipeline form; null-as-empty: `map` yields `[]` and `join` renders `""`, the quiet degradation the gradient usually prefers. So propagate is safest exactly where null-as-empty is most dangerous (the quantifiers) and costliest exactly where null-as-empty is most convenient (the render pipelines). Third option, `reject` (runtime type error at the iterator itself) — loudest, anti-gradient for the most ordinary absence case; not seriously entertained. **Judgement explicitly reserved (Carl, July 2026, batch-5 review)** — drafted to propagate per the fixed points; to be reviewed in total, with this comparison in view. Interlock: another "absence at a value input" cell — whatever the group review does with #3's headline question should be checked against this row and #13 together. **Adopted since (null-policy review, July 2026): `nullInputDefault`** (ledger #18) — presence-sensitive: unset → propagate as drafted; supplied (typically `[]`) → the iterator runs over it instead, so hosts buy null-as-empty per operator via `operatorDefaults` without flipping this row. Both bad cells get an answer — the quantifier trap stays opt-in, the pipeline seam costs one host-level default — but the drafted-default judgement itself stays reserved.

25. **`buildObject` null-valued entries — keep the key.** `{ key: 'phone', value: null }` builds `{ phone: null }`: null is a domain value and the entry occupies its slot — #18's position-preserving ruling in object clothes — and **v2 agrees** (`'value' in element` passes a null value through — [BUILD_OBJECT/operator.ts:33](../src/operators/BUILD_OBJECT/operator.ts#L33)); JSON's own convention concurs (`JSON.stringify` keeps null, drops only `undefined` — and `undefined` isn't a v3 value). The expressibility asymmetry that decided #18 decides here too: under keep, the drop reading is one node away — filter the entries array first (`{ $filter: [entries, { $notEqual: ['$element.value', null] }] }`, batch 5) — while under drop, keep would be inexpressible (nothing resurrects a dropped key). The live alternative: **drop** — the optional-fields idiom (building query/filter objects where an absent field should mean *no key at all*), genuinely the more common intent for dynamically-assembled request objects. Interlock: batch 8's `http.query` null question (omit the parameter vs `?x=null`) is this same idiom one layer down — judge the two rows together, and note a drop ruling there does *not* require one here (query serialization is a rendering boundary; `buildObject` is a value constructor).

26. **`get` — missing vs stored null, the distinction preserved per site.** Largely settled in References (the four-tool absence table); listed so the group review sees the **absence-default family** side by side: `get.missingPathDefault` fires on *missing only*, `find.noMatchDefault` on *no-match only* — the family rule (Part 1 § Naming conventions): named for the firing condition, never firing on a found/stored `null`. One subtlety recorded from the pass: the parameter is deliberately **presence-sensitive** (no metadata default) — a default of `null` would neutralize `strictDataPaths` on every `get` node, so "missing → `null`" is the unadorned behaviour rather than a default value, and `missingPathDefault: null` is the per-site strictness opt-out (the Type area's known case, discharged in the pass). The near-lossless converter mapping rides this row: v2 `getData` + `fallback` → `get` + `missingPathDefault` (v2 threw on missing only — [OBJECT_PROPERTIES/operator.ts:16](../src/operators/OBJECT_PROPERTIES/operator.ts#L16) — but its `fallback` also caught genuine errors, hence *near*-lossless). (The parameter was renamed from the drafted `missingDefault` at the null-policy review, July 2026 — the firing condition sharpened into the name.)

27. **`http.body` whole-null — no body, not a null document.** Fixed point 3 applied at an I/O boundary: `body` is optional and its type doesn't include `null`, so a null arriving there means unset — the request goes out with no body. The wire rule (batch 8) still holds *inside* a present body: `body: { note: '$data.note' }` with `note` missing posts `{"note":null}` — JSON has a null to carry. The recorded cost: a whole-body JSON `null` document (`POST` with literal body `null` — legal JSON, vanishingly rare) is inexpressible, since even `{ $literal: null }` arrives as the same null value. The live alternative: declare the type as including `null` (fixed point 3's opt-out), making `body: null` send the null document — but then every missing `$data.payload` posts a literal `null` to a server (outward-facing garbage manufactured from absence — the #9 family at the wire, and I/O actions can't be un-fired), and *no-body* becomes the inexpressible intent instead. The asymmetry decides it: absence-means-no-body fails visibly server-side (a 400 on a missing body beats a 200 on a null document), and the rare null-document author has a custom-client escape.
28. **`http.query` / `http.headers` null values — omit the pair.** The wire rule's omit side: a URL query string and a header block have no null representation, so a null value drops its pair from the serialization. The candidate send-readings are both worse: `?assignee=null` — v2's actual output, `URLSearchParams`'s `String(null)` coercion ([httpClients.ts:74](../src/httpClients.ts#L74)) — is the stringly-null bug class (a server reading the literal string `"null"`); `?assignee=` asserts *empty string*, a different claim than absence, silently plausible (the #17 failure family). Omission matches what query parameters mean to most servers (presence-based filters) and wins the expressibility asymmetry as #18/#25 did: under omit, `?x=` stays spellable (`x: ''`) and even a literal `?x=null` stays spellable (`x: 'null'`); under either send-reading, omission requires rebuilding the object upstream. The deliberate contrast with #25 (`buildObject` **keeps** null entries) is the distinction that row's note reserved — value constructor vs serialization boundary — and the two rulings compose: a `buildObject`-assembled `query` keeps nulls as entry values (#25) which the wire then omits as pairs (#28), the optional-filter idiom with zero ceremony. For `headers`, omission doubles as the per-node removal tool (null out an instance-default header). Honesty cell, recorded rather than hidden: *whole-null* `query` is unset per fixed point 3, so a missing `$data.filters` fires the request **unfiltered** — absence silently widening a fetch; the remedies are the standard ones (`strictDataPaths`, an explicit presence guard), and the row exists so the group review weighs this cell alongside #24's iterator-input call.
29. **`sql` empty result under `shape: 'row'` / `'value'` — `null`, absence not failure.** "No matching row" is an answer, not a breakdown — the #21 (`regex extract`) and #23 (`find`) family, landed the same way as promised by their interlock notes: the defaulting idiom composes (`{ $firstOf: [{ $sql: { …, shape: 'value' } }, 0] }`), and `'rows'` / `'column'` return `[]` (a list of findings is naturally empty — #21's `match`-mode twin). The alternative — no-row fails the node, fallback-catchable — is the strict read, rejected in drafting for the family reasons; genuine failures (connection down, SQL error, timeout) remain failures. Interlock: if the group review restores the v2 `fallback` reading at #23, this row follows it; if a `noMatchDefault`-style parameter family survives instead, a `noRowDefault` here is the natural (deferred, non-breaking) sibling. **Adopted since (null-policy review, July 2026): `noRowDefault`**, no longer deferred — runtime default `null`, so this row's ruling is the unadorned behaviour.
30. **`graphQL` errors — failure, even beside partial data.** GraphQL transports errors in-band (HTTP 200, `errors` array), and can return partial `data` alongside them. Drafted: any non-empty `errors` fails the node, with the array in `errorData` — Apollo's default `errorPolicy: 'none'`, the ecosystem's own conservative reading; v2 ignored the `errors` field entirely (silent nulls where fields failed to resolve). The live alternative: return the partial `data` and demote errors to `trace` — the degraded-render reading of the gradient. Rejected in drafting because a partially-resolved field is indistinguishable downstream from a genuinely-null field — the #23 conflation at scale, with no per-site default tool to split them — and because partial results from a *failing* backend are wrong-data-inside-plausible-output, the family the gradient rates worst. The future escape is an `errorPolicy`-shaped parameter (non-breaking union addition) if field experience demands it.
