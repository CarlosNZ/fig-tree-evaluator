## Batch 5 — Arrays & iteration — **Draft**

### `length`

| Parameter | Type                | Required | Default | Null policy | Evaluation |
| --------- | ------------------- | -------- | ------- | ----------- | ---------- |
| `value`   | `string` \| `array` | ✓        | —       | `propagate` | eager      |

- **One measure, two carriers**: an array's element count, or a string's **Unicode code-point count** — the batch-4 unit ruling, homed at `split` and consumed here through the one shared segmentation helper (implementation notes § One code-point segmentation), so no operator tears a surrogate pair while another counts it as two: `{ $length: '😀' }` is `1` where JS `.length` says `2`. Grapheme clusters stay out of scope, as recorded at `split` — a flag emoji is two code points and counts as two.
- **Not an aggregate — no empty-input question**: `{ $length: [] }` → `0` and `{ $length: '' }` → `0`; zero is the answer, not an anomaly. Null propagates (the Type area already names `length` among the propagate holders): `{ $length: '$data.tags' }` with the path missing is `null`, and the composed guard `{ $greaterThan: [{ $length: '$data.tags' }, 0] }` degrades to null → falsy — the standard gradient, no ceremony.
- An **object** input is a runtime type error. Counting keys is a different question with no core spelling yet (`keys` / `values` sit on the maybe-later list, v3-api.md Operators § Deferred) — honest gap, recorded.
- Positional: `['...value']` — the conventions' own worked example: rest-only is how array-*input* operators bind, so `{ $length: [1, 2, 3] }` is `value: [1, 2, 3]`, never three surplus arguments; the single non-array payload binds verbatim, giving `{ $length: '$data.tags' }` (dynamic, runtime-checked `string | array`) and `{ $length: 'hello' }` → `5` from the standard rule with nothing bespoke.
- `useCache`: `false`.
- **v2 disposition**: COUNT → `length` — the canonical name is v2's own alias promoted, and the other alias `count` dies with the one-alias rule (both map mechanically in the converter; no recycled-name hazard — the meaning never changed). `values` → `value` per the naming convention: this is one input that happens to be an array, not an operand list — which is also why ledger #8's element-wise machinery doesn't apply here. Gains string support (v2 was arrays-only — [COUNT/operator.ts:12](../src/operators/COUNT/operator.ts#L12)); the `[1, 2, 3, 4, 5]` metadata default on a *required* parameter ([COUNT/data.ts:13](../src/operators/COUNT/data.ts#L13)) is the editor-seed pattern's fourth sighting, resolved as ever — seed → hints module, no runtime default.

### The iterators — one contract, five operators

`map`, `filter`, `find`, `some`, `every` share a single shape; the per-operator sections below record only what differs — result shape, decision rule, empty-input identity.

| Parameter | Type                            | Required | Default                          | Null policy                                                                                                                                              | Evaluation                                                                    |
| --------- | ------------------------------- | -------- | -------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `input`   | `array`                         | ✓        | —                                | `propagate` — a missing collection resolves the node to `null` (*provisional — #24*)                                                                      | eager                                                                         |
| `each`    | `any` (expression)              | ✓        | —                                | `value` — per-element results judged by the operator's semantics: `map` takes any value; the predicates are truthiness positions, null falsy (ledger #4) | **lazy, once per element, in the binding scope** — concurrency per operator |
| `as`      | `string` — literal identifier   | optional | — (bind `$element` / `$index`)   | *(structural — a parse-time name, never evaluated)*                                                                                                       | structural                                                                    |

- **Names confirmed as locked in References** (`input` / `each` / `as` — used throughout that area's examples, formally settled here as its follow-up list asks). `input` over the convention's `value`: the role-name rule applies — the collection under iteration is a role, and `value` would misdescribe the relationship, since the elements, not the array, are what `each` consumes. `each` reads as spoken — *for each element, this expression* — and is deliberately uniform across transform and predicate operators (one word to learn, one shape for the editor); the jargon candidates (`callback`, `predicate`, `fn`) fail rule 0. `as` is the aliasing word every query language already taught.
- **The binding scope is exactly the `each` subtree.** Within it, `$element` (drillable) and `$index` (bare) — or the `as`-renamed pair `$name` / `$nameIndex` — resolve to the current element and position; References § `$element` / `$index` and `as` fixes the grammar, the innermost-wins rule and the collision checks. Everything outside that subtree is outside the scope, mechanically: the same iterator's `input` (nothing is bound yet — though a *nested* iterator's `input` sees the outer bindings, per the References example), and `vars` or `fallback` on the iterator node itself (they belong to the node, which evaluates once, not per element) — references to the node's own bindings from any of these are validation errors, while nodes *inside* `each`, including their `vars` and `fallback`s, are in scope. **Fragments stay sealed**: a fragment called inside `each` never sees the caller's bindings implicitly — pass them as parameters (`parameters: { user: '$element' }`), the same rule as caller `vars`.
- **`$index` is zero-based**, agreeing with the path grammar's bracket indices (`$data.users[0]`); `buildString`'s 1-based `%N` is token numbering, not indexing — a different convention, deliberately unreconciled.
- **Per-element evaluation is a fresh scope instance**: a `vars` block on a node inside `each` memoizes per element per evaluation, never once for the whole iteration (the References lazy-var mechanism applied per instance). Ledgered as the batch's one new contract capability (#12): the lazy machinery (#1) extended from at-most-once to once-per-element-with-bindings.
- **Concurrency: elements evaluate in parallel, always.** The `and`/`or` argument transfers whole: elements are homogeneous *data*, not a priority chain — `map` and `filter` need every element on every path, and the deciders (`find` / `some` / `every`) reuse the early-resolution-and-cancellation machinery (ledger #7) rather than serializing. The deliberate contrast is `firstOf` once more: its candidates are heterogeneous *alternatives* whose sequencing is semantics (a backup must not fire); an iterator's `each` is one expression stamped over a list. The side-effect note transfers verbatim: every element's evaluation *starts* — order-dependent or skip-dependent mutations don't belong in `each`.
- **Error semantics**: for `map` and `filter` every element's result is used, so any per-element failure fails the node — nothing is surplus, the batch-3 posture, no Kleene question. For `find` / `some` / `every` the Kleene rule (batch 1) applies via decision-dependence, per operator below; `trace` records failed and cancelled elements, as for `and`/`or`.
- **Empty input never errors in this batch** — every contract here is total or carries a unique, type-stable identity, so the batch-1 generating rule grants each case: `map: []` → `[]` and `filter: []` → `[]` (monomorphic — the `join` argument), `find: []` → `null` (the degenerate no-match — the `firstOf` argument: an absence tool's contract is already total), `some: []` → `false` and `every: []` → `true` (the quantifier identities — the constraint recorded at `and`/`or`, hereby discharged). Dynamically-empty input behaves identically; a *literal* `[]` draws the `validate()` dead-expression warning, matching `and`/`or`/`firstOf`/`join`.
- **`input` null → propagate**, drafted as fixed point 2's default and `length`'s twin — a missing collection is *unknown*, not empty: `{ $every: { input: '$data.checks', each: '$element.passed' } }` with `checks` missing yields `null` (falsy downstream), where a null-as-empty carve-out would answer `true` — vacuous truth over a list that isn't there, silent affirmation from absence, the #9 failure family with no doc-guidance escape. The cost, recorded honestly beside it: propagate composes *loudly* into whole-null positions — `join`-over-`map` with a missing collection is a runtime type error (`map` propagates `null`, and whole-null at `join.values` is ledger #8's type failure), where null-as-empty would render `""`. Both readings own a bad cell; the row exists so the group review weighs them side by side, and it carries the worked pipeline comparison. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #24; judgement explicitly reserved at the batch review (Carl, July 2026).* A non-null, non-array `input` — strings included: `split` first — is an ordinary runtime type error.
- **`as` is grammar, not data**: a literal identifier, resolved at parse (References fixes the validation: name legality, no reserved-namespace collision in long or short form, no enclosing-`as` collision — derived `…Index` forms included). A dynamic `as` is a parse error — the `vars`-names rule, for the same reason: static resolvability. Legal on any iterator, nested or not — naming the bindings purely for readability is fine.
- **`validate()` checks, all engine-generic** (bindings are grammar, so nothing rides the ledger-#11 hook): binding references outside any iterator, or outside their iterator's `each` subtree, are errors (References); an `each` that references none of its own iterator's bindings draws a **warning** — the dead-binding lint, the unreferenced-`vars` pattern's sibling: a constant transform is conceivable, but it's almost always a mistyped reference or a mis-nested payload.
- Positional: `['input', 'each']`, all five; `as` is named-face only (the batch convention). `{ $map: ['$data.users', '$element.name'] }` — the pluck idiom — is the everyday face; surplus elements are the standard parse error; `{ $map: '$data.orders' }` (single payload → `input`) fails parse on the missing required `each`.
- `useCache`: `false`, all five.
- **v2 disposition: all five new** ([#92](https://github.com/CarlosNZ/fig-tree-evaluator/issues/92) — the `$element` / `$index` operators). No predecessor, no converter work, no recycled names (none was ever a v2 alias). The gap v2 filled host-side — array transforms via `customFunctions` — now has a first-class, sandboxed spelling, and the migration doc can point those rewrites here. `reduce` is deliberately absent (maybe-later list, v3-api.md Operators § Deferred): an accumulator is genuinely new binding grammar, and the everyday folds are already operators (`plus`, `min`/`max`, `join`, `some`/`every`); `findIndex` / `flatMap` likewise wait for a use case (the `trimStart` precedent).

```js
{ $map: ['$data.users', '$element.name'] }                    // the pluck idiom
{ $filter: ['$data.lineItems', { $greaterThan: ['$element.qty', 0] }] }
{ $find: ['$data.users', { $equal: ['$element.id', '$data.targetId'] }] }
{ $some: ['$data.roles', { $equal: ['$element', 'admin'] }] }
{ $filter: { input: '$data.addressLines', each: { $notEqual: ['$element', null] } } }
                                                              // strip nulls — the join/split cleanup recipe
{ $filter: { input: '$data.values', each: '$element' } }      // strip all falsy — filter-truthy, spelled explicitly
```

(Nested iteration and `as` renaming: the worked example lives in References § `$element` / `$index` and `as` — not restated here.)

### `map`

- Result: an array — every element's `each` result, input order, same length, nulls included (`each` is `any`-typed; a null result is a value, handled downstream per each consumer's own table — the `join` pipeline renders it `""`).
- All elements evaluate, in parallel; any failure fails the node. Nothing more to settle — `map` is the common contract with no decision rule.

### `filter`

- Result: the **original elements** (never the predicate results), input order preserved, kept where the predicate is truthy (shared FigTree truthiness, ledger #4). An all-falsy outcome is `[]` — an answer, not an anomaly.
- All predicates evaluate, in parallel; any failure fails the node — *membership of every element is part of the result*, so no Kleene discard arises, unlike the deciders below.
- The recipes earlier passes promised now exist (examples above): strip-nulls — `join`'s skip reading, `split`'s trailing-empty cleanup — and strip-falsy.

### `find`

One parameter beyond the common contract:

| Parameter        | Type  | Required | Default | Null policy                                                    | Evaluation                                          |
| ---------------- | ----- | -------- | ------- | -------------------------------------------------------------- | --------------------------------------------------- |
| `noMatchDefault` | `any` | optional | `null`  | `value` (`any` includes `null` — the `if.then` note applies) | **lazy** — evaluates only on no-match (ledger #1) |

- Result: the **first element, in input order, whose predicate is truthy** — the element itself, never its index (`findIndex` waits on the maybe-later list). **No match → the value of `noMatchDefault`**, whose runtime default is `null` — so an unadorned `find` keeps the absence reading carried from `regex` `extract` (#21): "nothing matched" is an answer, composing down the gradient, never a failure. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #23.*
- **`noMatchDefault` — the null conflation resolved inside fixed point 1** (the batch review's one signature change — Carl, July 2026). The objection: a *found* null element is indistinguishable from a no-match `null` (`each: { $equal: ['$element', null] }` succeeds and returns the null it found), and no downstream composition can split the two apart — `firstOf` sees one null; `some` is the re-query workaround. Carl read this as a strong case for restoring **v2's `fallback`-on-nothing-found**; that reading contradicts fixed point 1 and stays live at #23, feeding the register's **headline `fallback` question** (what the key *means* — one call, at the group review). The adopted counter: an ordinary operator-specific parameter on `get`'s missing-default contract (the parameter since named `missingDefault`, batch 6) — fires on **no-match only**, a found null passes through unchanged — kept orthogonal to `fallback` (`noMatchDefault` answers "none matched", `fallback` catches "the search *broke*"; they compose on one node). The name is **deliberately sharpened per operator** rather than a uniform `default` (Carl's call, recorded as a naming convention in Part 1): the firing condition is load-bearing precision, so it lives in the name — this is not a special global property, just a parameter, and its name varies with its purpose. Bonus the parameter spelling buys over the fallback spelling: it rides the layered defaults chain — `operatorDefaults: { find: { noMatchDefault: … } }` gives a host-wide no-match default without touching failure semantics. Named-face only, like every option-shaped parameter in these batches.
- **Parallel, order-aware early resolution**: all predicates start concurrently; the node resolves at the first index whose predicate is truthy *once every earlier predicate is known falsy*, cancelling the rest (ledger #7, applied order-aware). Kleene by decision-dependence: the result depends on every index up to and including the decider — a failure at or before it fails the node; failures beyond it are discarded with the cancelled work. Completion timing changes only speed and how much work gets cancelled, never the outcome — the batch-1 sentence, unchanged. A failing `noMatchDefault` fails the node (ordinary lazy-parameter behaviour), which a `fallback` can catch — the orthogonality above, demonstrated.

### `some` / `every`

- **The quantifiers — `or`/`and` with the condition list factored into data plus one predicate**: `{ $some: { input: xs, each: p } }` is `or` over *p*-of-each-element, and the machinery is literally batch 1's — parallel start, early resolution (`some`: first truthy → `true`; `every`: first falsy → `false`), cancellation, Kleene discard of failures the decision doesn't depend on (`some(error, truthy)` → `true`). Results are **actual booleans**, never elements (the Type-area rule; the element-selecting reading is `find`'s job).
- Empty identities as above — `some: []` → `false`, `every: []` → `true` — closing the loop the `and`/`or` pass opened: JS parity, the quantifier reading, and the cousins agree.
- **Composition note — the #9 family surfacing per element**: a predicate that *propagates* null (`{ $greaterThan: ['$element.age', 18] }` over an element with `age` missing) reads falsy at the quantifier, so one absent field makes `every` confidently `false` — and can never make it `true`. That is the agreed `and`/`or` cell applied element-wise, not a new rule; the mitigation is the same doc guidance — test presence explicitly where absence matters (`{ $notEqual: ['$element.age', null] }`).
- Membership testing is the everyday `some` (`{ $some: ['$data.roles', { $equal: ['$element', 'admin'] }] }`) — which is why `includes` can stay on the maybe-later list until the shorthand earns its keep. And when the conditions are heterogeneous expressions rather than one predicate over data, `and`/`or` remain the right spelling — two tools, one distinction, recorded so the docs don't present them as rivals.

## Batch 6 — Data & objects — **Draft**

### `get`

| Parameter        | Type                | Required | Default                                          | Null policy                                                                                             | Evaluation |
| ---------------- | ------------------- | -------- | ------------------------------------------------ | -------------------------------------------------------------------------------------------------------- | ---------- |
| `path`           | `string` \| `array` | ✓        | —                                                | `propagate`                                                                                               | eager      |
| `from`           | `any`               | optional | **the merged evaluation data** (ledger #13)      | `value` — a null source is a source where every path is missing (drill-through; see below)                | eager      |
| `missingDefault` | `any`               | optional | — *(presence-sensitive — deliberately no default)* | `value` — `any` includes `null`, so `missingDefault: null` is a supplied value: the strictness opt-out | **lazy** — evaluates only on a missing path (ledger #1) |

- **Role and the sugar contract**: `get` is the reference layer's dynamic face — for paths that arrive as data, for drilling into an inline node's result, and for keys a reference can't spell. The equivalence is exact and two-way: `{ $get: 'user.name' }` ≡ `"$data.user.name"` — same missing-→-`null` behaviour, same `strictDataPaths` response — which is what "literal references are sugar for it" (Operators) means in practice. One honest asymmetry recorded for docs: `{ $get: '$data.a.b' }` is *not* that sugar — the payload is an ordinary expression, so the reference resolves first and its **result** becomes the path (double indirection: "read the path stored at `a.b`, then follow it"). Legitimate — paths stored in data are the operator's whole purpose — but a likely slip for authors who meant `{ $get: 'a.b' }`; `validate()` can't distinguish intent, so this is a documentation callout, not a check.
- **Path grammar — the References grammar plus the deferred exotic-key extension, settled here.** Base: dot-separated keys and bracket indices (`users[0].name` — v2's `object-property-extractor` syntax). Extension one, **quoted bracket segments**: `settings["a.b"]`, `["weird [key]"]` — single or double quotes, backslash escapes for the quote character and backslash itself; this grammar is `get`-only — references stay clean, per the References ruling. Extension two, **the segments array**: `path: ['users', 0, 'first.name']` — strings are keys *verbatim* (never parsed — no escaping question can arise), numbers are indices; the fully-general form and the natural target for computed paths. **Numeric segments read JS-style** — an index against an array, a key against an object, however spelled (`[0]`, `.0`, `'0'`, `0`) — so a path built by `split` (all-string segments) still indexes arrays instead of silently missing. **Resolution reads own enumerable properties only** (JSON semantics): `__proto__`, `constructor` and friends are *missing*, never JS internals — the References rule-4 spirit (no injection surface) applied to path lookup. An empty path (`''` / `[]`) is the bare-reference rule: the whole source. A literal `path` parses at `validate()` (ledger #11 — the `regex.pattern` twin); a dynamic path that fails to parse is an ordinary runtime failure.
- **Missing semantics — the layered story, now with its per-site tool.** A missing path yields `null`; a supplied `missingDefault` fires on **missing only** — a genuine stored `null` passes through unchanged (the References contract, `find.noMatchDefault`'s sibling under the Part 1 naming convention). Under `strictDataPaths: true` a missing path is a runtime failure — unless `missingDefault` is supplied, which is precisely the per-site opt-out: `missingDefault: null` reads "give me null instead of throwing" (the Type area's known case, discharged). This is why the parameter is deliberately **presence-sensitive** — a metadata default of `null` would neutralize `strictDataPaths` on every `get` node. `strictDataPaths` governs `get` uniformly, supplied `from` included — it is *the* missing-path strictness switch, and the operator is the reference layer wearing parameters. *Largely settled in References; listed for the group view — [v3-cases-for-review.md](v3-cases-for-review.md) #26.*
- **`from` — replace, never merge, and absence drills through.** When supplied, `path` resolves against `from` *instead of* the evaluation data (v2 merged `additionalData` over the data object — [OBJECT_PROPERTIES/operator.ts:15](../src/operators/OBJECT_PROPERTIES/operator.ts#L15) — a silent-collision surface; merging stays expressible as `plus` object mode feeding `from`). A **null `from` is a source where every path is missing** — `{ $get: { from: '$data.user', path: 'name', missingDefault: 'Anonymous' } }` with `user` absent returns `'Anonymous'`, matching what the same author gets from `"$data.user.name"`'s drill-through plus a default. Recorded loudly because the plausible misreadings are both worse: null-means-unset would silently widen scope to the whole data context (reading top-level `name` — wrong data, silently), and `propagate` would skip `missingDefault` exactly where its author most expects it to fire. Drilling into a non-container behaves the same way — a scalar, `null`, or opaque constant has no paths, so every segment against it is missing (matching the reference layer's drill-through; strings are scalars here — no JS `.length` leakage).
- Positional: `['path', 'missingDefault']` — revised at the batch review (Carl, July 2026; the first draft said `['path', 'from']`): defaulting is `get`'s everyday second input — the v2 `getData` + `fallback` pair in one payload — where drilling a supplied object is the rarer, named-face-friendly case, so **`from` is named-face only**. `{ $get: '$data.selectedField' }` — single payload → `path`, the everyday dynamic-path face; `{ $get: ['user.phone', 'N/A'] }` — path plus default. The deliberate contrast with `find.noMatchDefault` (named-face only) is the `split`/`join` delimiter asymmetry again: the same conceptual parameter rides positionally where it's the operator's everyday second input and named-only where it isn't (#22's cousin table gains the pair). `useCache`: `false`.
- **v2 disposition**: OBJECT_PROPERTIES → `get` (operator aliases died in Operators). `property` → `path` — v2's own parameter alias promoted ([OBJECT_PROPERTIES/data.ts:18](../src/operators/OBJECT_PROPERTIES/data.ts#L18)), `propertyName` dies. `additionalData` (aliases `additional` / `objects` / `data` / `additionalObjects`) → **replaced** by `from`, semantics changed from merge to replace — behaviour change, migration-doc line. v2 threw on missing ([OBJECT_PROPERTIES/operator.ts:16](../src/operators/OBJECT_PROPERTIES/operator.ts#L16)), so v2 expressions pair it with `fallback`; the converter maps that pair to `missingDefault` — near-lossless (References' note: v2's fallback also caught genuine errors; `missingDefault` covers missing only). One v2 quirk is made honest rather than killed: v2's positional form smuggled its second child into the node's `fallback` ([OBJECT_PROPERTIES/operator.ts:20-23](../src/operators/OBJECT_PROPERTIES/operator.ts#L20-L23)) — a parameter position writing a reserved key; v3's second position is the declared `missingDefault`, doing openly what that fallback was almost always there to do — so v2's two-child positional nodes convert to the *same positional shape*. Editor seeds (`'path.to[0].my.data'`, `{}`) → the hints module — the pattern's fifth sighting.

```js
{ $get: '$data.selectedField' }                       // dynamic path against the evaluation data
{ $get: 'user.name' }                                 // ≡ "$data.user.name" — the sugar contract
{ $get: ['user.phone', 'N/A'] }                       // path + missingDefault — v2's positional pair, made honest
{ $get: { path: 'name', from: { $find: ['$data.users', { $equal: ['$element.id', '$data.id'] }] } } }
                                                      // drill into a node result — from: named face only
{ $get: { path: ['users', 0, 'first.name'], missingDefault: 'Unknown' } }
                                                      // segments array: exotic key, no escaping; default on missing
```

### `buildObject`

| Parameter | Type                                         | Required | Default | Null policy                                                                                                                                  | Evaluation      |
| --------- | -------------------------------------------- | -------- | ------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | --------------- |
| `entries` | array — elements are `{ key, value }` objects | ✓        | —       | parameter: whole-null is a runtime type error (ledger #8); per entry: `key` **rejects** null, `value` holds it as a value — **kept** (*#25*) | eager |

- **Role, narrowed to what literals can't do**: under deep evaluation a static-keyed object is just a plain literal (`{ title: '$data.name' }` evaluates in place), so `buildObject` exists for **dynamic keys** — and doubles as the sanctioned escape for output that must *contain* reserved words as data (`vars`, `operator`, a `$`-shaped key): its output keys are runtime data, never re-parsed (Node grammar), which is why the `match` pass and the `vars` rules both point here.
- **Entry semantics**: each entry is a plain `{ key, value }` object; both sides are ordinary expressions (deep evaluation — no special two-mode machinery: a dynamic whole-`entries` array simply arrives as resolved data). Keys take the canonical string form for `string` / `number` / `boolean` (`key: 1` → `"1"` — `match`'s branch-key rule, and v2's own admitted key types, [BUILD_OBJECT/operator.ts:27](../src/operators/BUILD_OBJECT/operator.ts#L27)); a **null key is a runtime type error** (`reject` — deliberately *not* the stringification table's `""`, the same near-miss `match` refused for branch matching), as is a composite key. A **malformed entry — not an object, or missing `key` or `value` — is a runtime type error**: v2 silently filtered these out ([BUILD_OBJECT/operator.ts:33](../src/operators/BUILD_OBJECT/operator.ts#L33)), the seam erased; the gradient forbids that as firmly as it forbids blocking on data-driven nulls. Element-shape checking rides ledger #9's constraint vocabulary (amended there).
- **Null-valued entries are kept**: `{ key: 'phone', value: null }` → `{ phone: null }` — null is a value, the entry occupies its slot (`join` #18's ruling in object clothes), and **v2 agrees** (`'value' in element` passes with a null value — same line as above). The drop reading — the optional-fields idiom, building query/filter objects where absent means *omit the key* — stays one node away under keep: filter the entries first, `{ $buildObject: { entries: { $filter: [entriesArray, { $notEqual: ['$element.value', null] }] } } }` (batch 5's recipe, again), while under drop the keep reading would be inexpressible — the same asymmetry that decided #18. Interlocks with batch 8's `http.query` null question, which is the same idiom one layer down. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #25.*
- **Duplicate keys: last wins**, in declaration order — JS/JSON parity (`Object.fromEntries`, spread), and collisions under dynamic keys are data-dependent, so erroring would block end users on config they can't see. The seam is kept visible where the gradient wants it: `validate()` warns on statically-detectable duplicate literal keys (ledger #11), and `trace` records runtime overwrites.
- **Empty `entries` → `{}`** — monomorphic, identity unique and type-stable: granted by the batch-1 generating rule (`join`/`map`'s argument); the everyday dynamically-empty case (an optional-fields object where nothing is set) wants an empty object, not an error. Literal `[]` draws the dead-expression warning.
- Positional: `['...entries']` — rest-only: `{ $buildObject: [{ key: 'a', value: 1 }, { key: 'b', value: 2 }] }`; single non-array payload is the dynamic whole-array supply, per the standard rule. **v2's alternating flat form dies** (`['a', 1, 'b', 2]` — key, value, key, value): v2 accepted both shapes in one parameter and *sniffed at runtime* which it had ([BUILD_OBJECT/operator.ts:21-23](../src/operators/BUILD_OBJECT/operator.ts#L21-L23)) — shape-dispatch on runtime data, with an odd-length throw as its failure mode. The converter maps literal alternating payloads mechanically; dynamically-supplied ones are statically undetectable — loud migration-doc callout.
- `useCache`: `false`.
- **v2 disposition**: `properties` → `entries` (the `Object.fromEntries` term, precise and plain; aliases `values` / `keyValPairs` / `keyValuePairs` die); operator aliases `build` / `object` died in Operators; the alternating-pairs form and the silent malformed-entry filter die as above (both behaviour changes, migration-doc lines); the editor seed (`['firstKey', 'firstValue', …]` — itself in the dead alternating shape) → the hints module.

## Batch 7 — Special — **Draft**

### `literal`

| Parameter | Type                    | Required | Default | Null policy                                          | Evaluation                                              |
| --------- | ----------------------- | -------- | ------- | ---------------------------------------------------- | -------------------------------------------------------- |
| `value`   | `any` — never inspected | ✓        | —       | *(structural — a null is content like anything else)* | **structural** — never parsed, validated or evaluated |

- **The node-grammar treatment covers it — confirmed** (the constraint this pass owed): node kind 5 already defines `literal` as grammatically an operator node / shorthand face that is a **parse boundary**; Comments already exempts its contents from `//` stripping by fiat; References §5 already stops reference recognition at its edge. What the pass adds is the parameter name — `value`, the single-input convention — and one precision: **the boundary begins at the parameter's value.** Canonical form `{ operator: 'literal', value: X }`; shorthand `{ $literal: X }`, where the payload is **never disambiguated by JSON type** — no positional-list reading, no named-parameter reading: the payload *is* the content. Consequence recorded: `{ $literal: { value: 1 } }` quotes the object `{ value: 1 }` — shorthand `literal` deliberately has no named face (the canonical form expresses anything the shorthand can't).
- **The boundary is total inward, ordinary outward.** Inward: contents are never walked — references stay inert, `vars` keys are data (not consumed), `//` keys are data (not stripped), `$`-shaped keys draw no warnings, reserved words are just strings. Outward: the node itself obeys the normal grammar — sibling modifiers on the shorthand face are legal per the sibling-key rule, and a `literal` node sits inside any expression like any constant.
- **Once quoted, quoted forever** (confirmed in discussion — the natural first question): `{ $literal: { operator: 'plus', values: [1, 2, 3] } }` evaluates to that object *as data*, and it stays data all the way to the output — References §4 guarantees it, since recognition runs only at parse and runtime values are never re-interpreted, no enclosing operator, `match` lookup or fragment can ever re-capture and evaluate the quoted expression. Enclosing operators still treat it as the *value* it is under their normal rules (a rendering position shows `<object>`; a typed position type-errors) — but nothing evaluates it. This is the mechanism behind the sanctioned use case: expressions whose intended *output* is itself a FigTree expression.
- **Every modifier on it is dead, and `validate()` says so**: `literal` cannot fail, so `fallback` never fires; nothing evaluates, so `vars` scopes nothing and `useCache` is moot. All three stay grammatically *legal* — the reserved-key set is uniform by design — but each draws a dead-modifier warning, the unreferenced-`vars` family.
- **Zero runtime cost**: a `literal` subtree is part of the compiled constant skeleton (Options § Deep evaluation) — spliced into the result, never walked. `useCache`: `false` (moot, per above).
- **v2 disposition: new** — v2 had no quote mechanism; the collision case (data whose keys look like operators) was simply a standing hazard. Deliberately **not** PASSTHRU's successor: v2 PASSTHRU was identity over its *evaluated* child ([PASSTHRU/operator.ts:9](../src/operators/PASSTHRU/operator.ts#L9)) — `literal` quotes instead of evaluating; the converter maps v2 `passThru` nodes to their child unwrapped, never to `literal`. `literal` was never a v2 alias — no recycled-name hazard.

### `convert`

| Parameter | Type                                                          | Required | Default | Null policy                                                                                                                                 | Evaluation |
| --------- | ------------------------------------------------------------- | -------- | ------- | --------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| `value`   | `any` — per-mode domains in the Type-area table               | ✓        | —       | **mode-conditional** (ledger #14): `propagate` for `to: 'number'` / `'string'` / `'array'`; `value` for `to: 'boolean'` — null consumed, → `false` | eager      |
| `to`      | `'number'` \| `'string'` \| `'boolean'` \| `'array'`          | ✓        | —       | `reject` — falls out of the literal-union runtime check (null is not a member)                                                                | eager      |

- **The semantics are the Type area's, verbatim** (§ `convert` semantics — the strictness table, the per-target failure rules, the boolean carve-out and its null exception): this pass records shape only, as the constraint promised, and restates nothing. Every conversion failure is an ordinary runtime failure, fallback-catchable. One honesty line the table implies but doesn't state: a strict numeric parse can still produce a non-finite (`"1e999"`) — the ledger #10 boundary guard fails it like any other operator result.
- **The one contract wrinkle, ledgered (#14)**: `value`'s null policy is not fixed — it is keyed by `to` (`propagate` three ways, consumed for `'boolean'`, per the agreed table). Declaring that conditionally keeps the policy engine-side (implementation notes: null policies never live in operator bodies) and introspection-honest, instead of smuggling the boolean exception into the operator body.
- **`to` is the literal-union selector, one operator sanctioned by the mode rule** (signature invariant across targets — the Operators-table note, confirmed). A literal `to` validates at parse; a dynamic `to` is legal and lands on the runtime union check — the `regex.mode` / `plus.expect` treatment exactly. Future targets are non-breaking additions to the union; the Type table already names JSON-stringification as a possible later `to` value (a cast and a serialization being different intents is why `to: 'string'` fails on composites today). Deliberately no `'integer'` target: a cast is not arithmetic — rounding belongs to `round` / `floor`.
- Positional: `['value', 'to']` — `{ $convert: ['$data.qty', 'number'] }`, the everyday face; a single payload is a parse error (`to` is required). `useCache`: `false`.
- **v2 disposition: new operator, replacing the `outputType` / `type` node modifier** — the reserved-key set killed both spellings, and with them v2's one dynamically-evaluated structural key ([evaluate.ts:213](../src/evaluate.ts#L213)); Node grammar already records the contrast that `convert.to`, as an ordinary *parameter*, may be dynamic. The converter wraps `outputType`-carrying v2 nodes in a `convert` node. v2's implicit number-mining is not carried — the `regex` `extract` + `convert` pipeline is the sanctioned respelling (discharged at batch 4). Recycled-name hazard already tabled at Operators: v2's `convert` was a PASSTHRU alias (identity) — the converter maps those to their evaluated child, never to v3 `convert`.
