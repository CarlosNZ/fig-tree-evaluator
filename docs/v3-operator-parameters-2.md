## Batch 5 — Arrays & iteration — **Draft**

### `length`

| Parameter | Type                | Required | Default | Null policy | Evaluation |
| --------- | ------------------- | -------- | ------- | ----------- | ---------- |
| `value`   | `string` \| `array` | ✓        | —       | `propagate` | eager      |

- **One measure, two carriers**: an array's element count, or a string's **Unicode code-point count** — the batch-4 unit ruling, homed at `split` and consumed here through the one shared segmentation helper (implementation notes § One code-point segmentation), so no operator tears a surrogate pair while another counts it as two: `{ $length: '😀' }` is `1` where JS `.length` says `2`. Grapheme clusters stay out of scope, as recorded at `split` — a flag emoji is two code points and counts as two.
- **Not an aggregate — no empty-input question**: `{ $length: [] }` → `0` and `{ $length: '' }` → `0`; zero is the answer, not an anomaly. Null propagates (the Type area already names `length` among the propagate holders): `{ $length: '$data.tags' }` with the path missing is `null`, and the composed guard `{ $greaterThan: [{ $length: '$data.tags' }, 0] }` degrades to null → falsy — the standard gradient, no ceremony.
- An **object** input is a runtime type error. Counting keys is a different question with no core spelling yet (`keys` / `values` sit on the maybe-later list, v3-api.md Operators § Deferred) — honest gap, recorded.
- Positional: `['...value']` — the conventions' own worked example: rest-only is how array-*input* operators bind, so `{ $length: [1, 2, 3] }` is `value: [1, 2, 3]`, never three surplus arguments; the single non-array payload binds verbatim, giving `{ $length: '$data.tags' }` (dynamic, runtime-checked `string | array`) and `{ $length: 'hello' }` → `5` from the standard rule with nothing bespoke.
- `useCache`: `false`.
- **v2 disposition**: COUNT → `length` — the canonical name is v2's own alias promoted, and the other alias `count` dies with the one-alias rule (both map mechanically in the converter; no recycled-name hazard — the meaning never changed). `values` → `value` per the naming convention: this is one input that happens to be an array, not an operand list — which is also why ledger #8's element-wise machinery doesn't apply here. Gains string support (v2 was arrays-only — [COUNT/operator.ts:12](../src/operators/COUNT/operator.ts#L12)); the `[1, 2, 3, 4, 5]` metadata default on a *required* parameter ([COUNT/data.ts:13](../src/operators/COUNT/data.ts#L13)) is the editor-seed pattern's fourth sighting, resolved as ever — seed → hints module, no runtime default.

### The iterators — one contract, five operators

`map`, `filter`, `find`, `some`, `every` share a single shape; the per-operator sections below record only what differs — result shape, decision rule, empty-input identity.

| Parameter | Type                            | Required | Default                          | Null policy                                                                                                                                              | Evaluation                                                                    |
| --------- | ------------------------------- | -------- | -------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `input`   | `array`                         | ✓        | —                                | reject — `array` only: a null collection is a **runtime type error** (#24, agreed at the group review); `nullInputDefault` is the declared softener                                                                      | eager                                                                         |
| `each`    | `any` (expression)              | ✓        | —                                | `value` — per-element results judged by the operator's semantics: `map` takes any value; the predicates are truthiness positions, null falsy (ledger #4) | **lazy, once per element, in the binding scope** — concurrency per operator |
| `as`      | `string` — literal identifier   | optional | — (bind `$element` / `$index`)   | *(structural — a parse-time name, never evaluated)*                                                                                                       | structural                                                                    |
| `nullInputDefault` | `array` | optional | — *(presence-sensitive)* | unset — null here reads as omitted | **lazy** — evaluates only when `input` is null (ledger #18) |

- **Names confirmed as locked in References** (`input` / `each` / `as` — used throughout that area's examples, formally settled here as its follow-up list asks). `input` over the convention's `value`: the role-name rule applies — the collection under iteration is a role, and `value` would misdescribe the relationship, since the elements, not the array, are what `each` consumes. `each` reads as spoken — *for each element, this expression* — and is deliberately uniform across transform and predicate operators (one word to learn, one shape for the editor); the jargon candidates (`callback`, `predicate`, `fn`) fail rule 0. `as` is the aliasing word every query language already taught.
- **The binding scope is exactly the `each` subtree.** Within it, `$element` (drillable) and `$index` (bare) — or the `as`-renamed pair `$name` / `$nameIndex` — resolve to the current element and position; References § `$element` / `$index` and `as` fixes the grammar, the innermost-wins rule and the collision checks. Everything outside that subtree is outside the scope, mechanically: the same iterator's `input` (nothing is bound yet — though a *nested* iterator's `input` sees the outer bindings, per the References example), and `vars` or `fallback` on the iterator node itself (they belong to the node, which evaluates once, not per element) — references to the node's own bindings from any of these are validation errors, while nodes *inside* `each`, including their `vars` and `fallback`s, are in scope. **Fragments stay sealed**: a fragment called inside `each` never sees the caller's bindings implicitly — pass them as parameters (`parameters: { user: '$element' }`), the same rule as caller `vars`.
- **`$index` is zero-based**, agreeing with the path grammar's bracket indices (`$data.users[0]`); `buildString`'s 1-based `%N` is token numbering, not indexing — a different convention, deliberately unreconciled.
- **Per-element evaluation is a fresh scope instance**: a `vars` block on a node inside `each` memoizes per element per evaluation, never once for the whole iteration (the References lazy-var mechanism applied per instance). Ledgered as the batch's one new contract capability (#12): the lazy machinery (#1) extended from at-most-once to once-per-element-with-bindings.
- **Concurrency: elements evaluate in parallel, always.** The `and`/`or` argument transfers whole: elements are homogeneous *data*, not a priority chain — `map` and `filter` need every element on every path, and the deciders (`find` / `some` / `every`) reuse the early-resolution-and-cancellation machinery (ledger #7) rather than serializing. The deliberate contrast is `firstOf` once more: its candidates are heterogeneous *alternatives* whose sequencing is semantics (a backup must not fire); an iterator's `each` is one expression stamped over a list. The side-effect note transfers verbatim: every element's evaluation *starts* — order-dependent or skip-dependent mutations don't belong in `each`.
- **Error semantics**: for `map` and `filter` every element's result is used, so any per-element failure fails the node — nothing is surplus, the batch-3 posture, no Kleene question. For `find` / `some` / `every` the Kleene rule (batch 1) applies via decision-dependence, per operator below; `trace` records failed and cancelled elements, as for `and`/`or`.
- **Empty input never errors in this batch** — every contract here is total or carries a unique, type-stable identity, so the batch-1 generating rule grants each case: `map: []` → `[]` and `filter: []` → `[]` (monomorphic — the `join` argument), `find: []` → `null` (the degenerate no-match — the `firstOf` argument: an absence tool's contract is already total), `some: []` → `false` and `every: []` → `true` (the quantifier identities — the constraint recorded at `and`/`or`, hereby discharged). Dynamically-empty input behaves identically; a *literal* `[]` draws the `validate()` dead-expression warning, matching `and`/`or`/`firstOf`/`join`.
- **`input` null → runtime type error** — resolved at the group review (Carl, July 2026 — [v3-cases-for-review.md](v3-cases-for-review.md) #24, flipping the drafted propagate): `input` admits `array` only, so a null collection fails the node like any other type miss — a genuine failure, caught by `fallback`, consistent with fallback-means-failure. This retires both drafted readings' bad cells at once: no vacuous quantifier truth manufactured from absence (null-as-empty's trap), and no null composing quietly onward or exploding two nodes downstream (propagate's pipeline cost) — the failure is at the iterator, named, and the remedies are declared: `nullInputDefault` where absence should read as empty, `fallback` where it should degrade, `firstOf` / `missingPathDefault` at the source. Knock-on seam, recorded: `length` still *propagates* null (the Type area lists it) — the earlier draft called it this rule's twin; they now part ways, either as a deliberate difference (an iterator *does*, `length` *measures*) or as an invitation for the same treatment — decide when next touching `length` (register, Open items). A non-null, non-array `input` — strings included: `split` first — is the same ordinary runtime type error.
- **`nullInputDefault`** (adopted at the null-policy review, July 2026 — ledger #18): when `input` evaluates to `null`, this value is used as the collection instead — typically `[]`, giving `map`/`filter` → `[]`, `some` → `false`, `every` → `true`, `find` → its `noMatchDefault`. Presence-sensitive, applied in ledger #18's layer *before* the type check: unset → #24's type error; supplied → no error at all. `operatorDefaults: { map: { nullInputDefault: [] } }` is the host-wide null-as-empty stance — opt-in and visible. Named-face only, all five.
- **`as` is grammar, not data**: a literal identifier, resolved at parse (References fixes the validation: name legality, no reserved-namespace collision in long or short form, no enclosing-`as` collision — derived `…Index` forms included). A dynamic `as` is a parse error — the `vars`-names rule, for the same reason: static resolvability. Legal on any iterator, nested or not — naming the bindings purely for readability is fine.
- **`validate()` checks, all engine-generic** (bindings are grammar, so nothing rides the ledger-#11 hook): binding references outside any iterator, or outside their iterator's `each` subtree, are errors (References); an `each` that references none of its own iterator's bindings draws a **warning** — the dead-binding lint, the unreferenced-`vars` pattern's sibling: a constant transform is conceivable, but it's almost always a mistyped reference or a mis-nested payload.
- Positional: `['input', 'each']`, all five; `as` is named-face only (the batch convention). `{ $map: ['$data.users', '$element.name'] }` — the pluck idiom — is the everyday face; surplus elements are the standard parse error; `{ $map: '$data.orders' }` (single payload → `input`) fails parse on the missing required `each`.
- `useCache`: `false`, all five.
- **v2 disposition: all five new** ([#92](https://github.com/CarlosNZ/fig-tree-evaluator/issues/92) — the `$element` / `$index` operators). No predecessor, no converter work, no recycled names (none was ever a v2 alias). The gap v2 filled host-side — array transforms via `customFunctions` — now has a first-class, sandboxed spelling, and the migration doc can point those rewrites here. `reduce` is deliberately absent (maybe-later list, v3-api.md Operators § Deferred): an accumulator is genuinely new binding grammar, and the everyday folds are already operators (`plus`, `min`/`max`, `join`, `some`/`every`); `findIndex` / `flatMap` likewise wait for a use case (the `trimStart` precedent).

```js
{ $map: ['$data.users', '$element.name'] }                    // the pluck idiom
{ $filter: ['$data.lineItems', { $greaterThan: ['$element.qty', 0] }] }
{ $find: ['$data.users', { $equal: ['$element.id', '$data.targetId'] }] }
{ $some: ['$data.roles', { $equal: ['$element', 'admin'] }] }
{ $filter: { input: '$data.addressLines', each: { $notEqual: ['$element', null] } } }
                                                              // strip nulls — the join/split cleanup recipe
{ $filter: { input: '$data.values', each: '$element' } }      // strip all falsy — filter-truthy, spelled explicitly
```

(Nested iteration and `as` renaming: the worked example lives in References § `$element` / `$index` and `as` — not restated here.)

### `map`

- Result: an array — every element's `each` result, input order, same length, nulls included (`each` is `any`-typed; a null result is a value, handled downstream per each consumer's own table — the `join` pipeline renders it `""`).
- All elements evaluate, in parallel; any failure fails the node. Nothing more to settle — `map` is the common contract with no decision rule.

### `filter`

- Result: the **original elements** (never the predicate results), input order preserved, kept where the predicate is truthy (shared FigTree truthiness, ledger #4). An all-falsy outcome is `[]` — an answer, not an anomaly.
- All predicates evaluate, in parallel; any failure fails the node — *membership of every element is part of the result*, so no Kleene discard arises, unlike the deciders below.
- The recipes earlier passes promised now exist (examples above): strip-nulls — `join`'s skip reading, `split`'s trailing-empty cleanup — and strip-falsy.

### `find`

One parameter beyond the common contract:

| Parameter        | Type  | Required | Default | Null policy                                                    | Evaluation                                          |
| ---------------- | ----- | -------- | ------- | -------------------------------------------------------------- | --------------------------------------------------- |
| `noMatchDefault` | `any` | optional | `null`  | `value` (`any` includes `null` — the `if.then` note applies) | **lazy** — evaluates only on no-match (ledger #1) |

- Result: the **first element, in input order, whose predicate is truthy** — the element itself, never its index (`findIndex` waits on the maybe-later list). **No match → the value of `noMatchDefault`**, whose runtime default is `null` — so an unadorned `find` keeps the absence reading carried from `regex` `extract` (#21): "nothing matched" is an answer, composing down the gradient, never a failure. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #23.*
- **`noMatchDefault` — the null conflation resolved inside fixed point 1** (the batch review's one signature change — Carl, July 2026). The objection: a *found* null element is indistinguishable from a no-match `null` (`each: { $equal: ['$element', null] }` succeeds and returns the null it found), and no downstream composition can split the two apart — `firstOf` sees one null; `some` is the re-query workaround. Carl read this as a strong case for restoring **v2's `fallback`-on-nothing-found**; that reading contradicted fixed point 1 and was settled the drafted way at the group review (July 2026): `fallback` catches failure only, and `noMatchDefault` stands (#23, agreed). The adopted counter: an ordinary operator-specific parameter on `get`'s missing-default contract (the parameter since named `missingPathDefault`, batch 6) — fires on **no-match only**, a found null passes through unchanged — kept orthogonal to `fallback` (`noMatchDefault` answers "none matched", `fallback` catches "the search *broke*"; they compose on one node). The name is **deliberately sharpened per operator** rather than a uniform `default` (Carl's call, recorded as a naming convention in Part 1): the firing condition is load-bearing precision, so it lives in the name — this is not a special global property, just a parameter, and its name varies with its purpose. Bonus the parameter spelling buys over the fallback spelling: it rides the layered defaults chain — `operatorDefaults: { find: { noMatchDefault: … } }` gives a host-wide no-match default without touching failure semantics. Named-face only, like every option-shaped parameter in these batches.
- **Parallel, order-aware early resolution**: all predicates start concurrently; the node resolves at the first index whose predicate is truthy *once every earlier predicate is known falsy*, cancelling the rest (ledger #7, applied order-aware). Kleene by decision-dependence: the result depends on every index up to and including the decider — a failure at or before it fails the node; failures beyond it are discarded with the cancelled work. Completion timing changes only speed and how much work gets cancelled, never the outcome — the batch-1 sentence, unchanged. A failing `noMatchDefault` fails the node (ordinary lazy-parameter behaviour), which a `fallback` can catch — the orthogonality above, demonstrated.

### `some` / `every`

- **The quantifiers — `or`/`and` with the condition list factored into data plus one predicate**: `{ $some: { input: xs, each: p } }` is `or` over *p*-of-each-element, and the machinery is literally batch 1's — parallel start, early resolution (`some`: first truthy → `true`; `every`: first falsy → `false`), cancellation, Kleene discard of failures the decision doesn't depend on (`some(error, truthy)` → `true`). Results are **actual booleans**, never elements (the Type-area rule; the element-selecting reading is `find`'s job).
- Empty identities as above — `some: []` → `false`, `every: []` → `true` — closing the loop the `and`/`or` pass opened: JS parity, the quantifier reading, and the cousins agree.
- **Composition note — the #9 family surfacing per element**: a predicate that *propagates* null (`{ $greaterThan: ['$element.age', 18] }` over an element with `age` missing) reads falsy at the quantifier, so one absent field makes `every` confidently `false` — and can never make it `true`. That is the agreed `and`/`or` cell applied element-wise, not a new rule; the mitigation is the same doc guidance — test presence explicitly where absence matters (`{ $notEqual: ['$element.age', null] }`).
- Membership testing is the everyday `some` (`{ $some: ['$data.roles', { $equal: ['$element', 'admin'] }] }`) — which is why `includes` can stay on the maybe-later list until the shorthand earns its keep. And when the conditions are heterogeneous expressions rather than one predicate over data, `and`/`or` remain the right spelling — two tools, one distinction, recorded so the docs don't present them as rivals.

## Batch 6 — Data & objects — **Draft**

### `get`

| Parameter        | Type                | Required | Default                                          | Null policy                                                                                             | Evaluation |
| ---------------- | ------------------- | -------- | ------------------------------------------------ | -------------------------------------------------------------------------------------------------------- | ---------- |
| `path`           | `string` \| `array` | ✓        | —                                                | `propagate`                                                                                               | eager      |
| `from`           | `any`               | optional | **the merged evaluation data** (ledger #13)      | `value` — a null source is a source where every path is missing (drill-through; see below)                | eager      |
| `missingPathDefault` | `any`               | optional | — *(presence-sensitive — deliberately no default)* | `value` — `any` includes `null`, so `missingPathDefault: null` is a supplied value: the strictness opt-out | **lazy** — evaluates only on a missing path (ledger #1) |

- **Role and the sugar contract**: `get` is the reference layer's dynamic face — for paths that arrive as data, for drilling into an inline node's result, and for keys a reference can't spell. The equivalence is exact and two-way: `{ $get: 'user.name' }` ≡ `"$data.user.name"` — same missing-→-`null` behaviour, same `strictDataPaths` response — which is what "literal references are sugar for it" (Operators) means in practice. One honest asymmetry recorded for docs: `{ $get: '$data.a.b' }` is *not* that sugar — the payload is an ordinary expression, so the reference resolves first and its **result** becomes the path (double indirection: "read the path stored at `a.b`, then follow it"). Legitimate — paths stored in data are the operator's whole purpose — but a likely slip for authors who meant `{ $get: 'a.b' }`; `validate()` can't distinguish intent, so this is a documentation callout, not a check.
- **Path grammar — the References grammar plus the deferred exotic-key extension, settled here.** Base: dot-separated keys and bracket indices (`users[0].name` — v2's `object-property-extractor` syntax). Extension one, **quoted bracket segments**: `settings["a.b"]`, `["weird [key]"]` — single or double quotes, backslash escapes for the quote character and backslash itself; this grammar is `get`-only — references stay clean, per the References ruling. Extension two, **the segments array**: `path: ['users', 0, 'first.name']` — strings are keys *verbatim* (never parsed — no escaping question can arise), numbers are indices; the fully-general form and the natural target for computed paths. Extension three (adopted at the batch-8 review — Carl, July 2026), **the projection segment `[*]`** — and *not* `get`-only: unlike the quoted-segment escape hatch, projection is everyday, so it joins the shared grammar (References rule 3) and reaches every path consumer, `returnPath` included. Semantics: the value at the `[*]` must be an array; the path remainder applies to each element in order, results collected into an array (`user.weapons[*].name` → `['Blaster', 'Seismic charge']`); a per-element miss yields `null` in that slot (the absence rule, element-wise); `[*]` against a non-array or missing value is a missing path for the whole result (drill-through, the standard rule); a trailing `[*]` is the degenerate identity projection (the array itself). **This replaces v2's implicit projection** — the extractor's documented "array handling" silently projected any key over an array of objects (`user.weapons.name` → `['Blaster', 'Seismic charge']`), which Carl had already flagged to keep-but-make-explicit; in v3 the spelling without `[*]` is an honest miss (`name` is not a key of an array). Behaviour change, **loud migration line**: whether a v2 path traverses an array is data-dependent, so the converter cannot add `[*]` mechanically — v2 paths leaning on implicit projection need hand-annotation (the `buildObject` alternating-form / `sql.flatten` callout family). Sub-questions flagged for design, deliberately not ruled here: chained projections (`a[*].b[*].c` — nested result arrays vs flattening); whether `missingPathDefault` fires per element or only on a whole-path miss; and the **segments-array representation** — strings there are keys verbatim by rule, so `'[*]'` cannot be the wildcard's spelling and a dedicated token is needed. **Numeric segments read JS-style** — an index against an array, a key against an object, however spelled (`[0]`, `.0`, `'0'`, `0`) — so a path built by `split` (all-string segments) still indexes arrays instead of silently missing. **Resolution reads own enumerable properties only** (JSON semantics): `__proto__`, `constructor` and friends are *missing*, never JS internals — the References rule-4 spirit (no injection surface) applied to path lookup. An empty path (`''` / `[]`) is the bare-reference rule: the whole source. A literal `path` parses at `validate()` (ledger #11 — the `regex.pattern` twin); a dynamic path that fails to parse is an ordinary runtime failure.
- **Missing semantics — the layered story, now with its per-site tool.** A missing path yields `null`; a supplied `missingPathDefault` fires on **missing only** — a genuine stored `null` passes through unchanged (the References contract, `find.noMatchDefault`'s sibling under the Part 1 naming convention). Under `strictDataPaths: true` a missing path is a runtime failure — unless `missingPathDefault` is supplied, which is precisely the per-site opt-out: `missingPathDefault: null` reads "give me null instead of throwing" (the Type area's known case, discharged). This is why the parameter is deliberately **presence-sensitive** — a metadata default of `null` would neutralize `strictDataPaths` on every `get` node. `strictDataPaths` governs `get` uniformly, supplied `from` included — it is *the* missing-path strictness switch, and the operator is the reference layer wearing parameters. *Largely settled in References; listed for the group view — [v3-cases-for-review.md](v3-cases-for-review.md) #26. The name was sharpened from the drafted `missingDefault` at the null-policy review (July 2026) — the firing condition lives in the name, per the Part 1 convention.*
- **`from` — replace, never merge, and absence drills through.** When supplied, `path` resolves against `from` *instead of* the evaluation data (v2 merged `additionalData` over the data object — [OBJECT_PROPERTIES/operator.ts:15](../src/operators/OBJECT_PROPERTIES/operator.ts#L15) — a silent-collision surface; merging stays expressible as `plus` object mode feeding `from`). A **null `from` is a source where every path is missing** — `{ $get: { from: '$data.user', path: 'name', missingPathDefault: 'Anonymous' } }` with `user` absent returns `'Anonymous'`, matching what the same author gets from `"$data.user.name"`'s drill-through plus a default. Recorded loudly because the plausible misreadings are both worse: null-means-unset would silently widen scope to the whole data context (reading top-level `name` — wrong data, silently), and `propagate` would skip `missingPathDefault` exactly where its author most expects it to fire. Drilling into a non-container behaves the same way — a scalar, `null`, or opaque constant has no paths, so every segment against it is missing (matching the reference layer's drill-through; strings are scalars here — no JS `.length` leakage).
- Positional: `['path', 'missingPathDefault']` — revised at the batch review (Carl, July 2026; the first draft said `['path', 'from']`): defaulting is `get`'s everyday second input — the v2 `getData` + `fallback` pair in one payload — where drilling a supplied object is the rarer, named-face-friendly case, so **`from` is named-face only**. `{ $get: '$data.selectedField' }` — single payload → `path`, the everyday dynamic-path face; `{ $get: ['user.phone', 'N/A'] }` — path plus default. The deliberate contrast with `find.noMatchDefault` (named-face only) is the `split`/`join` delimiter asymmetry again: the same conceptual parameter rides positionally where it's the operator's everyday second input and named-only where it isn't (#22's cousin table gains the pair). `useCache`: `false`.
- **v2 disposition**: OBJECT_PROPERTIES → `get` (operator aliases died in Operators). `property` → `path` — v2's own parameter alias promoted ([OBJECT_PROPERTIES/data.ts:18](../src/operators/OBJECT_PROPERTIES/data.ts#L18)), `propertyName` dies. `additionalData` (aliases `additional` / `objects` / `data` / `additionalObjects`) → **replaced** by `from`, semantics changed from merge to replace — behaviour change, migration-doc line. v2 threw on missing ([OBJECT_PROPERTIES/operator.ts:16](../src/operators/OBJECT_PROPERTIES/operator.ts#L16)), so v2 expressions pair it with `fallback`; the converter maps that pair to `missingPathDefault` — near-lossless (References' note: v2's fallback also caught genuine errors; `missingPathDefault` covers missing only). One v2 quirk is made honest rather than killed: v2's positional form smuggled its second child into the node's `fallback` ([OBJECT_PROPERTIES/operator.ts:20-23](../src/operators/OBJECT_PROPERTIES/operator.ts#L20-L23)) — a parameter position writing a reserved key; v3's second position is the declared `missingPathDefault`, doing openly what that fallback was almost always there to do — so v2's two-child positional nodes convert to the *same positional shape*. Editor seeds (`'path.to[0].my.data'`, `{}`) → the hints module — the pattern's fifth sighting.


### `buildObject`

| Parameter | Type                                         | Required | Default | Null policy                                                                                                                                  | Evaluation      |
| --------- | -------------------------------------------- | -------- | ------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | --------------- |
| `entries` | array — elements are `{ key, value }` objects | ✓        | —       | parameter: whole-null is a runtime type error (ledger #8); per entry: `key` **rejects** null, `value` holds it as a value — **kept** (*#25*) | eager |

- **Role, narrowed to what literals can't do**: under deep evaluation a static-keyed object is just a plain literal (`{ title: '$data.name' }` evaluates in place), so `buildObject` exists for **dynamic keys** — and doubles as the sanctioned escape for output that must *contain* reserved words as data (`vars`, `operator`, a `$`-shaped key): its output keys are runtime data, never re-parsed (Node grammar), which is why the `match` pass and the `vars` rules both point here.
- **Entry semantics**: each entry is a plain `{ key, value }` object; both sides are ordinary expressions (deep evaluation — no special two-mode machinery: a dynamic whole-`entries` array simply arrives as resolved data). Keys take the canonical string form for `string` / `number` / `boolean` (`key: 1` → `"1"` — `match`'s branch-key rule, and v2's own admitted key types, [BUILD_OBJECT/operator.ts:27](../src/operators/BUILD_OBJECT/operator.ts#L27)); a **null key is a runtime type error** (`reject` — deliberately *not* the stringification table's `""`, the same near-miss `match` refused for branch matching), as is a composite key. A **malformed entry — not an object, or missing `key` or `value` — is a runtime type error**: v2 silently filtered these out ([BUILD_OBJECT/operator.ts:33](../src/operators/BUILD_OBJECT/operator.ts#L33)), the seam erased; the gradient forbids that as firmly as it forbids blocking on data-driven nulls. Element-shape checking rides ledger #9's constraint vocabulary (amended there).
- **Null-valued entries are kept**: `{ key: 'phone', value: null }` → `{ phone: null }` — null is a value, the entry occupies its slot (`join` #18's ruling in object clothes), and **v2 agrees** (`'value' in element` passes with a null value — same line as above). The drop reading — the optional-fields idiom, building query/filter objects where absent means *omit the key* — stays one node away under keep: filter the entries first, `{ $buildObject: { entries: { $filter: [entriesArray, { $notEqual: ['$element.value', null] }] } } }` (batch 5's recipe, again), while under drop the keep reading would be inexpressible — the same asymmetry that decided #18. Interlocks with batch 8's `http.query` null question, which is the same idiom one layer down. *Provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #25.*
- **Duplicate keys: last wins**, in declaration order — JS/JSON parity (`Object.fromEntries`, spread), and collisions under dynamic keys are data-dependent, so erroring would block end users on config they can't see. The seam is kept visible where the gradient wants it: `validate()` warns on statically-detectable duplicate literal keys (ledger #11), and `trace` records runtime overwrites.
- **Empty `entries` → `{}`** — monomorphic, identity unique and type-stable: granted by the batch-1 generating rule (`join`/`map`'s argument); the everyday dynamically-empty case (an optional-fields object where nothing is set) wants an empty object, not an error. Literal `[]` draws the dead-expression warning.
- Positional: `['...entries']` — rest-only: `{ $buildObject: [{ key: 'a', value: 1 }, { key: 'b', value: 2 }] }`; single non-array payload is the dynamic whole-array supply, per the standard rule. **v2's alternating flat form dies** (`['a', 1, 'b', 2]` — key, value, key, value): v2 accepted both shapes in one parameter and *sniffed at runtime* which it had ([BUILD_OBJECT/operator.ts:21-23](../src/operators/BUILD_OBJECT/operator.ts#L21-L23)) — shape-dispatch on runtime data, with an odd-length throw as its failure mode. The converter maps literal alternating payloads mechanically; dynamically-supplied ones are statically undetectable — loud migration-doc callout.
- `useCache`: `false`.
- **v2 disposition**: `properties` → `entries` (the `Object.fromEntries` term, precise and plain; aliases `values` / `keyValPairs` / `keyValuePairs` die); operator aliases `build` / `object` died in Operators; the alternating-pairs form and the silent malformed-entry filter die as above (both behaviour changes, migration-doc lines); the editor seed (`['firstKey', 'firstValue', …]` — itself in the dead alternating shape) → the hints module.

## Batch 7 — Special — **Draft**

### `literal`

| Parameter | Type                    | Required | Default | Null policy                                          | Evaluation                                              |
| --------- | ----------------------- | -------- | ------- | ---------------------------------------------------- | -------------------------------------------------------- |
| `value`   | `any` — never inspected | ✓        | —       | *(structural — a null is content like anything else)* | **structural** — never parsed, validated or evaluated |

- **The node-grammar treatment covers it — confirmed** (the constraint this pass owed): node kind 5 already defines `literal` as grammatically an operator node / shorthand face that is a **parse boundary**; Comments already exempts its contents from `//` stripping by fiat; References §5 already stops reference recognition at its edge. What the pass adds is the parameter name — `value`, the single-input convention — and one precision: **the boundary begins at the parameter's value.** Canonical form `{ operator: 'literal', value: X }`; shorthand `{ $literal: X }`, where the payload is **never disambiguated by JSON type** — no positional-list reading, no named-parameter reading: the payload *is* the content. Consequence recorded: `{ $literal: { value: 1 } }` quotes the object `{ value: 1 }` — shorthand `literal` deliberately has no named face (the canonical form expresses anything the shorthand can't).
- **The boundary is total inward, ordinary outward.** Inward: contents are never walked — references stay inert, `vars` keys are data (not consumed), `//` keys are data (not stripped), `$`-shaped keys draw no warnings, reserved words are just strings. Outward: the node itself obeys the normal grammar — sibling modifiers on the shorthand face are legal per the sibling-key rule, and a `literal` node sits inside any expression like any constant.
- **Once quoted, quoted forever** (confirmed in discussion — the natural first question): `{ $literal: { operator: 'plus', values: [1, 2, 3] } }` evaluates to that object *as data*, and it stays data all the way to the output — References §4 guarantees it, since recognition runs only at parse and runtime values are never re-interpreted, no enclosing operator, `match` lookup or fragment can ever re-capture and evaluate the quoted expression. Enclosing operators still treat it as the *value* it is under their normal rules (a rendering position shows `<object>`; a typed position type-errors) — but nothing evaluates it. This is the mechanism behind the sanctioned use case: expressions whose intended *output* is itself a FigTree expression.
- **Every modifier on it is dead, and `validate()` says so**: `literal` cannot fail, so `fallback` never fires; nothing evaluates, so `vars` scopes nothing and `useCache` is moot. All three stay grammatically *legal* — the reserved-key set is uniform by design — but each draws a dead-modifier warning, the unreferenced-`vars` family.
- **Zero runtime cost**: a `literal` subtree is part of the compiled constant skeleton (Options § Deep evaluation) — spliced into the result, never walked. `useCache`: `false` (moot, per above).
- **v2 disposition: new** — v2 had no quote mechanism; the collision case (data whose keys look like operators) was simply a standing hazard. Deliberately **not** PASSTHRU's successor: v2 PASSTHRU was identity over its *evaluated* child ([PASSTHRU/operator.ts:9](../src/operators/PASSTHRU/operator.ts#L9)) — `literal` quotes instead of evaluating; the converter maps v2 `passThru` nodes to their child unwrapped, never to `literal`. `literal` was never a v2 alias — no recycled-name hazard.

## Batch 8 — I/O — **Draft**

### The I/O contract — shared by all three

- **Registration is structural, and so is absence.** The three operators enter only via the factories — `httpOperators(client)` → `http` + `graphQL`, `sqlOperators(connection)` → `sql` (Options § Operator registration; clients are wiring, never options). Consequence worth stating: on a clientless instance these operators *don't exist*, so `{ $http: … }` is an unknown-operator error at parse/`validate()` — v2's evaluation-time `'No HTTP client provided'` ([GET/operator.ts:21](../src/operators/GET/operator.ts#L21)) moves from the end user's runtime to the author's toolchain, the gradient's preferred direction. v2's silent browser fallback (auto-adopting `window.fetch` when no client was supplied — [httpClients.ts:25](../src/httpClients.ts#L25)) dies with it: visible wiring only. The convenience survives one level down, in the client wrapper (clarified at review — Carl, July 2026): `FetchClient()` with no argument adopts the environment's global `fetch` (browser, or Node ≥ 18), so the minimal wiring is `httpOperators(FetchClient())` — what dies is a network capability the host never wrote down, not the one-liner.
- **The read contract.** v3 treats every I/O node as an **idempotent read**: expressions denote values, and the machinery the rest of the spec grants — parallel children, early resolution with cancellation (ledger #7), memoization, re-evaluation — is only legitimate over requests that can be fired, cancelled, duplicated or skipped without changing the world. This is why `http.method` is `'get' | 'post'` only: POST is admitted because read APIs genuinely require it (search endpoints; GraphQL itself is POST-shaped), while the mutating verbs (`put` / `patch` / `delete`) are deliberately absent from v3.0 — a workflow that mutates belongs host-side or in a custom operator whose author owns the interaction with cancellation and caching. Widening the union later is non-breaking, but arrives only with that story. Inside the admitted verbs, the boundary is policed by the mutation lints (below) and docs, not the grammar.
- **Signals and timeouts (ledger #15).** Every request receives a derived `AbortSignal` composing the evaluation `signal`, the whole-evaluation `timeout`, every enclosing early-resolution scope (#7 — a `firstOf` or `find` resolving early really cancels the losing requests), and the node's own `timeout` parameter — the constraint Node grammar fallback rule 3 recorded, discharged. The split matters: a **per-request `timeout` expiry is an ordinary runtime failure**, fallback-catchable — the network-flakiness guard, applied per node — while the whole-evaluation timeout and the caller's signal still cut through unconditionally (only static shielding shapes a timed-out result, and nothing shapes a cancelled one). `timeout` is uniform across the three operators: optional `integer` (ms), **no metadata default** (unset = bounded only by the evaluation-level `timeout`), null means unset, host-wide via `operatorDefaults: { http: { timeout: 5000 } }`. Cancellation is best-effort where a driver can't abort (SQLite's synchronous API) — recorded, not hidden.
- **Caching (ledger #16).** `useCache` metadata default is **`true` for all three** — the conventions' I/O rule, and deliberately POST included (revised at review — Carl, July 2026; the first draft carried v2's get/post split, [POST/operator.ts:46](../src/operators/POST/operator.ts#L46)): with mutations pushed out of the operator entirely (the read contract, policed by the lints), a POST here is a read that happens to need a verb with a body — GraphQL, search endpoints, complex-filter APIs — and reads cache. The memo key is the **effective request** — method, fully-resolved URL, serialized query/body/variables/binds, merged headers — never the raw parameter spellings, so two nodes spelling one request share one entry; **`returnPath` sits outside the key** — drilling applies post-cache to the shared entry, so v2's duplicate fetches per drill path (it sat inside the memoized call — [GET/operator.ts:76](../src/operators/GET/operator.ts#L76)) die as an implementation detail no author needs to know. **Failures are never cached.** Store/TTL per the `cache` option; precedence per the settled chain; `trace` records hit/miss.
- **Mutation lints (riding ledger #11).** A literal `graphQL.query` document opening with `mutation`, or a literal `sql.query` opening with a write verb (`INSERT` / `UPDATE` / `DELETE` / `TRUNCATE` / DDL), draws a `validate()` warning **unconditionally** — revised at review (Carl, July 2026; the first draft let `useCache: false` silence it): the warning's message *is* the read contract — this library is not the tool to be running mutations through — not a configuration hint, so no configuration silences it. Dynamic texts pass (runtime never lints).
- **Options access (ledger #17).** `http` and `graphQL` read their options blocks (`baseEndpoint` / `endpoint` / `headers`) from the frozen per-evaluation snapshot — declared, engine-supplied, introspectable; #13's sibling (that one supplies *data*, this one *options*). Clients stay closed into definitions; for custom operators the factory-closure pattern remains the recommended wiring, though the capability is in the contract per the first-class principle.
- **Null at the wire — the batch's one new rule.** A serialization target that *has* a null representation carries null through: JSON bodies, `graphQL.variables`, `sql` bind values (→ SQL `NULL`, as agreed). A target with *no* null representation omits the pair: URL query parameters and headers (#28). The contrast with `buildObject`'s keep ruling (#25) is the distinction its note reserved — a value constructor keeps the slot; a wire boundary with nothing to put in the slot drops the pair — and the two compose: a `buildObject`-assembled `query` object keeps its null entries as *values* (#25) and the wire omits them as *pairs* (#28), which is the optional-filter idiom for free, no `filter` step.
- **Client contract constraints** (the TS shape lands with the Extensibility contract; the passes record what it must satisfy): one request entry point receiving the derived signal; JSON-first response handling — the parsed body is the value, a non-JSON body is a runtime failure, an empty success body (204) is `null`; failures carry structured `errorData` (status, URL, response payload) for `fallback` / `mode: 'report'` / `trace`; and **header values never echo** — trace and error output render the effective request with header *names only*, since headers are the secret-bearing channel.

### `http`

| Parameter | Type                              | Required | Default                          | Null policy                                                                     | Evaluation |
| --------- | --------------------------------- | -------- | -------------------------------- | -------------------------------------------------------------------------------- | ---------- |
| `url`     | `string`                          | ✓        | —                                | `reject` (the Type-area assignment, confirmed)                                   | eager      |
| `method`  | `'get'` \| `'post'`               | optional | `'get'`                          | unset (fixed point 3)                                                            | eager      |
| `query`   | `object` — scalar values          | optional | — (no query string)              | parameter: unset; per value: `value` — **the pair is omitted** (*#28*)           | eager      |
| `body`    | `any` — a JSON payload            | optional | — (no body)                      | unset — **no body** (*#27*); nulls *inside* a present body are JSON `null`s      | eager      |
| `headers` | `object` — scalar values          | optional | —                                | parameter: unset; per value: `value` — the pair is omitted (#28's rule)          | eager      |
| `returnPath` | `string` \| `array` — `get.path`'s grammar | optional | — (the whole response)  | unset                                                                            | eager      |
| `timeout` | `integer` (ms)                    | optional | — (evaluation bound only)        | unset                                                                            | eager      |

- **Role**: the one HTTP operator, GET + POST merged behind `method` (Operators). The everyday face is the bare URL: `{ $http: 'https://api.example.com/rates' }` — already this spec's stock example.
- **`url`**: `reject` on null — an address has no meaningful degradation, *even on a base-configured instance*: "the base itself" is a deliberate address, spelled `url: ''`, never manufactured from absence — `url: '$data.detailsUrl'` with the path missing must not silently fetch the base endpoint and return whatever lives there (wrong resource inside plausible output, the #9 family at an I/O boundary). Relative URLs resolve against `http.baseEndpoint` (v2's rule kept: a full `http(s)://` URL passes verbatim, anything else — `''` included — joins the base); a relative URL with no base, or a resolved value that isn't a valid URL, is a runtime failure. Query pairs from `query` append to any query string already literal in the URL; keeping the two from colliding is the author's job (one honest line, no merge magic). v2's undocumented `{ url, headers }` object payload ([GET/operator.ts:40](../src/operators/GET/operator.ts#L40)) dies — headers have a parameter.
- **`method`**: lowercase literal union, case-sensitive like every literal union (the `regex.mode` / `convert.to` treatment); `validate()` hints on the near-miss (`'GET'` → "did you mean `'get'`"); dynamic values land on the runtime union check. Union deliberately narrow per the read contract (above).
- **`query`**: an object of scalars — values render onto the query string via the stringification table (`true` → `"true"`, `5` → `"5"`); a **null value omits the pair** (*provisional — [v3-cases-for-review.md](v3-cases-for-review.md) #28*): the wire rule above, and the expressibility asymmetry decides it like #18/#25 — under omit, `?x=` stays spellable (`x: ''`) and even a literal `?x=null` stays spellable (`x: 'null'`), while under either send-reading the omission intent needs the object rebuilt. v2 actually sent `?x=null` — `URLSearchParams`'s `String(null)` coercion ([httpClients.ts:74](../src/httpClients.ts#L74)) — the stringly-null bug class, dead. **Composite values are a runtime type error**: repeated-key, comma-joined and bracket conventions all exist and none is canonical — render explicitly (`join`) or supply a fully-formed URL; repeated keys consequently have no object spelling (honest gap, same escape). Whole-null `query` is unset per fixed point 3 — with the honesty cell recorded at #28: a missing `$data.filters` fires the request *unfiltered*.
- **`body`**: any JSON value, serialized by the client (JSON is the operator's content type; non-JSON bodies are a custom client's business). **Whole-null means no body** (fixed point 3; *provisional — #27*) — which makes a whole-body JSON `null` *document* deliberately inexpressible, recorded honestly; nulls nested inside a present body serialize as JSON `null` per the wire rule. `body` with `method: 'get'` is a runtime type error (`validate()` catches the literal case): GETs don't carry bodies, and the likely intent is a forgotten `method: 'post'` — the message says so.
- **`headers`**: per-key merge over the options chain — `http.headers` option ← node `headers`, node wins; a **null value removes the pair**, which is the per-node tool for *unsetting* an instance default header (a public endpoint inside an authed environment); scalar values render via the table, composites reject. Merge is exact-key (HTTP treats names case-insensitively; FigTree doesn't normalize — one spelling per environment, recorded honestly).
- **Result**: the parsed JSON response body, as data (References §4 — never re-parsed). Non-2xx is a runtime failure carrying `errorData`.
- **`returnPath`** — kept at review (Carl, July 2026; the first draft dropped it for `get.from` composition): response drilling is the operator's everyday second act, and frequency earns the fused spelling — the `notEqual` argument exactly; the composition stays available for anything richer. Grammar and resolution are precisely `get.path`'s (batch 6): dot/bracket string or segments array, literal parses at `validate()` (the ledger-#11 hook), a missing path in the response → `null` (the reference layer's absence rule; `firstOf` or a wrapping `get.missingPathDefault` supplies defaults). Caching: `returnPath` sits outside the memo key (the intro's #16 note) — an implementation detail, invisible to authors. A **`missingPathDefault`** sibling (defaulting a missed `returnPath` — `get`'s contract, same name, same firing condition) is recorded as a deferred, non-breaking candidate at the null-policy review (July 2026), for `graphQL` too; a wrapping `get` covers it meanwhile.
- **The `extractAndSimplify` record — what the v2 magic was for** (kept against re-investigation — Carl's ask, July 2026). *What it did* ([operatorUtils.ts:86-104](../src/operators/operatorUtils.ts#L86-L104)): after drilling, any **single-key object collapsed to its value** — applied *per element* when the result was an array, and the array branch ran even when no `returnProperty` was supplied at all. *Why it existed*: **field projection in a map-less v2** — a GraphQL-shaped response (`data.countries` → `[{ name: 'Fiji' }, { name: 'NZ' }]`) carries one more layer than anyone wants, the path grammar can't spell "the `name` of *each* element", and v2 had no iterator to spell it either; the collapse was the compensation (`sql.flatten` was the same instinct in SQL clothes). *Why v3 doesn't carry it*: it dispatches on runtime key count — select two fields and the node returns objects, select one and it returns scalars, same expression — the misreporting family batch 8 killed at `sql.flatten`. *The v3 replacements*: the **projection segment `[*]`** in the shared path grammar — `returnPath: 'countries[*].name'`, no `map` required — with `map` and `sql.shape: 'column'` for anything richer. The wildcard began this record as a candidate future landing spot for Carl's recorded reservation (map-as-the-price-of-projection) and was **adopted on the spot at review (Carl, July 2026)** — into the shared grammar, semantics homed at `get`'s pass (batch 6, extension three; References rule 3 records the grammar) — because it also resolves the *other* piece of v2 extractor magic in one stroke: the implicit array projection Carl had previously flagged to keep-but-make-explicit. One spelling now serves both needs, explicit where the collapse was inferred, statically visible, no key-count sniffing.
- Positional: `['url']` — the single-payload face is the star; everything else is named-face only (all option-shaped, and no second slot has an everyday claim: whether it would mean `query` or `body` depends on `method`, exactly the guessing a positional face must never do).
- `useCache`: `true` — POST included (the intro's revised ruling).
- **v2 disposition**: GET + POST → `http` (aliases `GET` / `get` / `api` / `POST` / `post` died in Operators). **The flat reservation bites its one operator**: v2's main parameter was literally named `parameters` ([GET/data.ts:32](../src/operators/GET/data.ts#L32), [POST/data.ts:32](../src/operators/POST/data.ts#L32)) — query params on GET, JSON body on POST, *one name, two meanings across the pair* — now a reserved word, force-renamed to the honest split `query` / `body`; the converter's mapping is keyed by the v2 operator and mechanical. Parameter aliases die: `endpoint`, `queryParams` / `queryParameters` / `urlQueries`, `bodyJson` / `data` — that last a double death, `data` being a reserved namespace word. **`returnProperty` → `returnPath`** (alias `outputProperty` dies): kept at review on the `notEqual` argument, renamed to say what it holds (`get.path`'s grammar) — the converter renames mechanically; the `extractAndSimplify` collapse magic is *not* carried (see the parameter bullet — `map` is the projection tool now), a behaviour change and migration-doc line for responses that leaned on it. The `useCache` *parameter* becomes the reserved node key (same spelling at the same level in shorthand — the sibling-key rule; the converter hoists it in canonical form). Editor seeds (the restcountries URL et al.) → the hints module — the pattern's sixth sighting.

| `timeout`   | `integer` | optional | —                                | unset                                                              | eager      |

- **`query`** is the GraphQL document — the protocol's own word (rule 0's established-term clause). The near-miss with `http.query` is recorded deliberately: each name is its own protocol's precise term, and per-domain precision beats cross-operator uniformity here (`sql.query` completes the trio).
- **`variables`**: a JSON carrier, so **null values are carried**, not omitted — an explicit `null` variable is meaningful GraphQL (nullable filter arguments); the deliberate contrast with `http.query` inside one wire rule. Whole-null → unset.
- **`url`**: optional because GraphQL is a single-endpoint protocol — the endpoint is environment config, homed at the `graphQL.endpoint` option; the parameter is the per-node override for multi-server hosts. Same resolution rule as `http.url` (relative joins `http.baseEndpoint`); neither node `url` nor option endpoint supplied → runtime failure. v2's magic placeholder string (`url: 'graphQLEndpoint'` meaning "use the option" — [GRAPHQL/operator.ts:58](../src/operators/GRAPHQL/operator.ts#L58)) dies: omission means that now.
- **`headers`** merge chain: `http.headers` ← `graphQL.headers` ← node `headers`, per-key, later wins — v2's chain restated over the settled options split (shared auth flows down; the graphQL block specializes).
- **Result and errors**: the value is the response's `data` field, drilled by `returnPath` when supplied (as at `http` — same grammar, same missing-→-`null`, same outside-the-cache-key treatment). A response with a non-empty `errors` array is a **runtime failure** — partial `data` included — with the errors in `errorData` (*provisional — #30*): Apollo's default `errorPolicy: 'none'`, adopted; v2 silently ignored the errors field. Partial-data tolerance is deliberately not v3.0 — an `errorPolicy`-shaped parameter is the non-breaking future form if the field demands it.
- Positional: `['query', 'variables']` (revised at review — Carl, July 2026; the first draft was query-only): the standard call pair when the endpoint is environment config — the common case of a host hitting mainly its own GraphQL API — so `{ $graphQL: ['query ($id: ID!) { user(id: $id) { name } }', { id: '$data.userId' }] }` just works; a single payload is the bare document. Everything else named-face only. `useCache`: `true` (the mutation lint fires on mutation documents regardless — v2 cached mutations silently by default, [GRAPHQL/operator.ts:67](../src/operators/GRAPHQL/operator.ts#L67)).
- **v2 disposition**: aliases `graphql` / `graphQl` / `gql` died in Operators (casing matches the options block). `returnNode` / `outputNode` / `returnProperty` → `returnPath`, as at `http` (drilling within the `data` field, exactly v2's scope — [GRAPHQL/operator.ts:80](../src/operators/GRAPHQL/operator.ts#L80) — minus the collapse magic). `endpoint` alias dies. The `graphQLConnection` option split was settled in Options — including its private `httpClient` sub-key: **one client serves both operators** from the one factory call; a host genuinely needing a distinct GraphQL transport wraps the switch inside its client or registers a custom operator (accepted narrowing, recorded). v2's positional children form (`[query, url, fieldNames, ...values]`) dies with `children`. Editor seeds (the trevorblades countries endpoint) → the hints module.

```js
{ $graphQL: 'query { countries { name } }' }                  // endpoint from options
{ $graphQL: ['query ($code: ID!) { country(code: $code) { name } }', { code: '$data.countryCode' }] }
                                                              // the positional pair: document, variables
{ $graphQL: { query: 'query { countries { name } }', returnPath: 'countries[*].name' } }
                                                              // v2 returnNode, renamed — [*] projects the names out

```

### `sql`

| Parameter | Type                                        | Required | Default  | Null policy                                                          | Evaluation |
| --------- | ------------------------------------------- | -------- | -------- | -------------------------------------------------------------------- | ---------- |
| `query`   | `string`                                    | ✓        | —        | `reject` (the Type-area assignment, confirmed)                        | eager      |
| `values`  | `array` \| `object`                         | optional | — (no binds) | parameter: unset; per element/value: `value` — **SQL `NULL`** (agreed) | eager      |
| `shape`   | `'rows'` \| `'row'` \| `'column'` \| `'value'` | optional | `'rows'` | unset                                                                 | eager      |
| `noRowDefault` | `any` | optional | `null` | `value` (`any` includes `null`) | **lazy** — evaluates only on an empty `'row'` / `'value'` result (ledger #1) |
| `timeout` | `integer`                                   | optional | —        | unset                                                                 | eager      |

- **`query`**: the SQL text with dialect-owned placeholders (`$1` / `?` / named — the injected connection's business; FigTree never parses SQL). **The injection lint** (riding #11): a `query` whose value is statically a string-composition node (`buildString` / `plus` / `join` feeding SQL text) draws a `validate()` warning pointing at bind values — authoring-side only, honestly partial (dynamic text from `$data` can't be linted), but the seam it catches is the classic one.
- **`values`**: positional binds as an array, named binds as an object where the driver supports them (v2's SQLite path already did); a **null element or value goes to the wire as SQL `NULL`** — the carried batch constraint, confirmed; the wire rule's carry side, with `graphQL.variables`. Elements otherwise pass to the driver verbatim (a `jsonb` object, a driver-coerced value) — the connection owns serialization. Whole-null → unset. The name keeps v2's canonical `values` — also node-postgres's own vocabulary (`{ text, values }`) — with the tension recorded: elsewhere in v3 `values` means a variadic operand list (naming conventions), which this is not; rule 0's established-term clause carries it, and `substitutions` (`buildString`'s cousin — same fill-the-placeholders role) is the recorded alternative if the collision reads worse at review.
- **`shape` — the declarative reshape replacing v2's `single` / `flatten` booleans** and their runtime magic (result shape depending on the arriving column count — `vals.length <= 1 ? vals[0] : vals`, [SQL/operator.ts:56](../src/operators/SQL/operator.ts#L56) — the plausible-wrong family). Four everyday result shapes, one literal union: **`'rows'`** (default) — array of row objects, `[]` when empty; **`'row'`** — the first row object, **no row → `null`** (*provisional — #29*: absence, not failure — the `find` / `regex extract` family); **`'column'`** — the single column's values as an array, `[]` when empty, and a multi-column result is a **runtime type error** (name what you want in the SQL — silent column-dropping is the misreporting the gradient rates worst); **`'value'`** — first row's single column, no row → `null`, multi-column → error. Multi-*row* results under `'row'` / `'value'` take the first, no error — queryOne semantics, pg's own `rows[0]`; determinism is the query's job (`ORDER BY` / `LIMIT` — doc line). Literal `shape` validates at parse; dynamic lands on the runtime union check. (The breakdown is agreed; the four names are provisional — Carl, July 2026 — to be finalized at implementation.)
- **`noRowDefault`** (adopted at the null-policy review, July 2026 — the sibling #29's note anticipated, no longer deferred): under `shape: 'row'` or `'value'`, an empty result set returns this value instead of `null`; runtime default `null`, so the unadorned behaviour above is unchanged. Never fires under `'rows'` / `'column'` (`[]` is the answer) nor on genuine failures (connection down, SQL error, timeout — `fallback`'s). Named-face only.
- **Result values arrive as data** (References §4): driver-produced non-JSON values (pg `Date`s, buffers) are opaque constants — they flow through untyped positions and fail typed ones; cast in SQL (`::text`) or wait for the date plugin — honest line. The row-as-value-array shape (v2's `flatten` over multi-column results) has no v3 spelling — the `keys` / `values` maybe-later gap, recorded at `length` already, resurfacing here.
- Positional: `['query', '...values']` — v2's own children shape kept: `{ $sql: ['SELECT name FROM users WHERE id = $1', '$data.userId'] }`; a single payload is the bare query; `shape` and `timeout` are named-face only (option-shaped). `useCache`: `true` (v2 parity; the write-verb lint fires regardless — the intro's revised ruling).
- **v2 disposition**: operator aliases `pgSql` / `postgres` / `pg` / `sqLite` / `sqlite` / `mySql` died in Operators; parameter aliases die (`text`, `replacements`, `singleRecord`, `flat` / `array`). `single` / `flatten` → `shape` — converter mapping: neither → `'rows'`, `single` → `'row'`, `flatten` → `'column'`, both → `'value'`; **loud migration line**: v2's `flatten` over multi-column results produced arrays-of-values per row, which v3 `'column'` rejects at runtime, and the divergence is statically undetectable (it lives in the SQL text). The hidden `queryType` / `outputType` compat rider ([SQL/operator.ts:49-51](../src/operators/SQL/operator.ts#L49-L51)) dies with `outputType`. `sqlConnection` option → the factory (settled in Options); the `useCache` parameter → the reserved node key.

```js
{ $sql: ['SELECT name FROM users WHERE id = $1', '$data.userId'] }
                                                              // positional binds; default shape: rows
{ $sql: { query: 'SELECT total FROM orders WHERE id = $1', values: ['$data.orderId'], shape: 'value' } }
                                                              // one scalar out; no row → null (#29)
{ $sql: { query: 'SELECT code FROM regions ORDER BY code', shape: 'column' } }
{ $sql: { query: 'UPDATE jobs SET run = now() WHERE id = $1', values: ['$data.id'] }, useCache: false }
                                                              // write verb: draws the mutation warning regardless —
                                                              // this library is not the tool for mutations
```
